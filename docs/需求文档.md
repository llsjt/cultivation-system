## 1. 产品定位与展示形式

### 1.1 一句话定位

这是一个本地桌面工具，用来记录学习资料的当前进度、完成百分比和下一步任务，并在下次打开时快速恢复学习上下文。

修仙主题是展示层，用来增强学习成就感；核心产品不是阅读器、播放器、浏览器插件、编辑器插件，也不是完整学习管理系统。

### 1.2 产品展示形式

产品以“本地桌面端修仙风学习进度仪表盘”的形式呈现。

第一屏不是营销页、不是游戏大厅、不是大面积世界观封面，而是一个可以直接继续学习的工作台。用户打开后应立刻看到：

1. 当前最应该继续学习的资料。
2. 上次学到哪里。
3. 当前完成百分比。
4. 下次具体要做什么。
5. 是否有一次学习尚未记录。

### 1.3 页面形态

MVP 包含四类核心页面或界面：

| 展示形态 | 修仙主题名称 | 真实含义 |
| --- | --- | --- |
| 首页仪表盘 | 修炼总览 | 查看推荐资料、项目进度、待记录状态 |
| 项目详情页 | 修炼方向详情 | 查看一个学习主题下的全部资料 |
| 资料详情页 | 功法详情 | 查看单个资料的进度、下一步和历史记录 |
| 记录弹窗 | 出关记录 | 学习后快速更新进度和下一步 |

### 1.4 UI 方向

UI 应该像一个有氛围的学习控制台：

- 可以使用“修炼、闭关、出关、参悟、功法、经卷”等词汇。
- 必须同时显示真实学习含义，避免用户不知道按钮实际会做什么。
- 进度条、百分比、资料名、下一步任务必须清晰可扫读。
- 视觉上可以有卷轴、玉简、墨色、星图、阵纹等轻量氛围元素。
- 不做游戏化大厅、角色养成页、等级面板、战力榜、抽卡、排行榜。
- MVP 只提供一套深色修仙主题，不提供浅色模式或主题切换；若未来支持浅色模式，必须单独补充配色 token 和对比度验收。

### 1.5 双层文案原则

所有关键文案采用“双层可理解”：

| 不推荐 | 推荐 |
| --- | --- |
| 入定 | 继续闭关：打开资料继续学习 |
| 破境 | 保存出关记录 |
| 灵气 +35 | 参悟进度 35% |
| 心魔 | 待破难点：需要复习的内容 |

修仙文案可以营造感觉，但不能替代真实学习信息。

## 2. MVP 目标

MVP 只验证一件事：

> 用户能否把一个学习资料的“当前学到哪里、完成多少、下次做什么”低成本记录下来，并在下次打开工具时立刻恢复学习上下文。

MVP 的唯一核心对象是“资料级学习进度”。项目、首页、日志、修仙主题、成就反馈都只能服务这个对象。

第一版必须让用户在首页第一屏回答三个问题：

1. 我现在该继续学哪个资料？
2. 这个资料上次学到哪里、完成多少？
3. 下一步具体做什么？

项目总进度只作为资料进度的汇总展示，不作为独立管理目标。

## 3. MVP 核心原则

1. 真实学习进度永远第一。
2. 修仙主题只做命名、氛围和轻量反馈。
3. 不做境界、等级、经验值、战力、排行榜。
4. 不做完整学习管理系统。
5. 不做阅读器、播放器、浏览器或编辑器插件。
6. 所有 P0 功能都必须服务“记录进度”和“下次继续”。

一句话准绳：

> 让用户保存一次进度像“出关”一样有满足感，但整个过程仍然像记一条学习进度一样快。

### 3.1 MVP 范围闸门

任何功能进入 MVP，必须同时满足以下条件：

1. 能帮助用户记录资料进度。
2. 能帮助用户下次恢复学习上下文。
3. 不要求用户维护额外系统。
4. 不引入新的长期数据管理页面。
5. 不让“出关记录”超过 30 秒完成。

不满足以上条件的功能，即使有用，也进入 MVP 之后再做。

## 4. MVP 范围

### 4.1 必须做

| 模块 | MVP 功能 | 说明 |
| --- | --- | --- |
| 项目 | 创建学习项目 | 修仙主题下称为“修炼方向” |
| 项目 | 编辑项目名称和描述 | 不做复杂设置 |
| 项目 | 查看项目列表 | 显示项目总进度 |
| 项目 | 删除项目 | 简单二次确认 |
| 资料 | 添加学习资料 | 修仙主题下称为“功法 / 经卷” |
| 资料 | 编辑资料 | 标题、类型、打开方式、路径、状态；不直接编辑进度和下一步 |
| 资料 | 查看资料详情 | 重点展示进度和下一步 |
| 资料 | 删除资料 | 删除资料，历史记录保留资料名称快照 |
| 进度 | 记录资料进度描述 | 例如“第 4 章 Borrowing 小节” |
| 进度 | 记录资料进度百分比 | 0 到 100 |
| 进度 | 自动计算项目进度 | 项目下资料进度平均值 |
| 下一步 | 记录下一步任务 | 修仙主题下称为“下次闭关目标” |
| 首页 | 继续闭关卡片 | 推荐一个继续学习的资料 |
| 续学 | 继续闭关 | 打开文件、文件夹、网页，或进入仅记录流程 |
| 记录 | 出关记录 | 学习后快速更新进度和下一步 |
| 反馈 | 基础成就反馈 | 保存成功、进度提升、资料完成 |
| 数据 | 本地持久化 | 关闭重开不丢失 |

进度描述、进度百分比和下一步任务的唯一变更入口是“出关记录”。资料编辑表单只能改资料元信息和状态，不提供直接修改进度或下一步的输入，避免出现有日志和无日志两套进度来源。

### 4.2 明确不做

MVP 不做：

- 独立日志管理页
- 日志编辑和删除
- 搜索
- 筛选
- 归档
- 导入导出
- 设置页
- 统计图表
- 连续学习完整成就系统
- 境界、等级、经验值
- AI 总结或 AI 建议
- PDF 页数识别
- Markdown 标题解析
- 视频时长识别
- 云同步
- 账号登录
- 多端同步
- 内置阅读器
- 内置播放器
- 浏览器插件
- 编辑器插件

### 4.3 学习记录边界

MVP 中的“修炼日志 / 学习记录”只用于保留每次保存进度时的历史快照。它不是独立日志产品。

限制：

1. 只能由“保存出关记录”自动生成。
2. 只在首页和详情页作为最近记录展示。
3. 不提供独立日志管理页。
4. 不提供日志编辑、删除、搜索、筛选、归档。
5. 不要求用户为了写日志而额外填写长文本。

### 4.4 MVP 优先级分层

MVP 不等于所有内容同时开工。为避免范围膨胀，需求按 P0/P1/P2 分层执行。

#### P0：核心闭环验证

P0 只验证用户是否愿意低成本记录学习进度，并能下次恢复。

必须包含：

| 能力 | P0 要求 |
| --- | --- |
| 创建项目 | 能创建一个修炼方向，作为资料归属 |
| 添加资料 | 能添加一个学习资料，默认进度 0 |
| 记录进度 | 能记录“学到哪里”和 0 到 100 百分比 |
| 记录下一步 | 能记录下次继续做什么 |
| 首页推荐 | 首页能推荐一个可继续资料 |
| 出关记录 | 用最少字段保存一次学习进度 |
| 自动快照 | 保存时自动生成学习记录 |
| 本地持久化 | 关闭重开后项目、资料、进度、下一步仍存在 |
| 基础反馈 | 保存成功、进度提升、100% 完成有轻量提示 |

P0 出关记录默认只保留 3 个核心字段：

1. 出关进度：这次学到哪里。
2. 参悟进度：完成百分比。
3. 下次闭关目标：下次做什么。

`功法状态`、`本次参悟`、`闭关时长` 可以在 P0 的数据模型中预留，但不要求作为 P0 主流程必填或强曝光字段。

P0 阶段仅会自动产生 `not_started`、`learning`、`completed` 三种资料状态：进度 0 对应 `not_started`，1 到 99 对应 `learning`，100 对应 `completed`。`review` 和 `paused` 从 P1 起可在出关记录中手动选择；P0 下推荐和项目派生规则会自然退化为只处理 `not_started`、`learning`、`completed`。

#### P1：MVP 发布硬化

P1 用于让 P0 闭环具备发布质量。自用 MVP 可以先完成 P0 并持续使用；准备给他人使用或公开分发前，必须补齐对应 P1 验收。

包括：

- 待记录会话。
- 继续闭关打开文件、文件夹、URL。
- `record_only` 完整规则。
- 打开失败分类。
- 路径和 URL 安全规则。
- 删除确认与删除影响。
- 数据库强约束。
- 事务回滚。
- 迁移可靠性。
- 桌面壳 / Electron 最小权限。
- 项目详情页和资料详情页完整状态。
- 最近学习记录展示。

#### P2：体验增强与后续扩展

P2 只能在 P0/P1 稳定后进入。

包括：

- 更完整修仙视觉氛围。
- 进度提升动画。
- 更细状态，例如暂停、待破难点。
- 学习时长增强。
- 备注增强。
- 更多历史记录视图。
- 搜索、筛选、统计、导入导出、AI、复习提醒。

### 4.5 MVP 失焦风险

当前产品最容易失焦的不是修仙主题，而是把学习进度记录周边能力都提前当成 P0。

需要特别防止：

1. 把本地资源打开器当成核心产品。
2. 把学习日志做成独立日志系统。
3. 把修仙氛围做成游戏系统。
4. 把完整安全硬化误认为 P0 验证前置条件。
5. 把统计、搜索、AI 和成就墙提前加入主流程。

判断标准：如果一个功能不能直接帮助用户回答“我学到哪了、完成多少、下次做什么”，它就不能进入 P0。

## 5. 核心概念

| 普通概念 | 修仙主题表达 | MVP 含义 |
| --- | --- | --- |
| 学习项目 | 修炼方向 | 一个长期学习主题 |
| 学习资料 | 功法 / 经卷 / 典籍 | 具体学习对象 |
| 学习进度 | 参悟进度 | 当前学到哪里、完成多少 |
| 项目进度 | 修炼总进度 | 项目整体推进程度 |
| 下一步任务 | 下次闭关目标 | 下次继续做什么 |
| 继续学习 | 继续闭关 | 打开资料继续学习 |
| 快速记录 | 出关记录 | 学习后记录进度 |
| 学习记录 | 修炼日志 | 每次保存后的历史记录 |
| 最近打开时间 | 最近打开时间 | `last_opened_at`，成功打开外部资源的时间 |
| 最近出关时间 | 最近出关时间 | `last_studied_at`，保存出关记录的时间 |
| 待记录会话 | 待出关记录 | `pending_study_sessions`，已打开但尚未保存的学习会话 |

界面可以使用修仙主题词，但关键数据必须保持清楚：

- 项目真实名称
- 资料真实名称
- 当前进度描述
- 进度百分比
- 下一步任务

## 6. 核心用户流程

### 6.1 首次使用

1. 用户打开工具。
2. 首页显示空状态：“还没有修炼方向。先创建一个学习主题。”
3. 用户创建第一个修炼方向。
4. 用户添加第一个功法 / 经卷。
5. 用户填写初始参悟进度和下次闭关目标。
6. 首页出现继续闭关卡片。

### 6.2 日常继续学习

1. 用户打开工具。
2. 首页“继续闭关”显示推荐继续的资料。
3. 用户看到当前参悟位置、参悟进度、修炼总进度和下次闭关目标。
4. 用户点击“继续闭关”。
5. 工具打开本地文件、本地文件夹、网页链接，或进入仅记录流程。
6. 工具进入待记录状态；如果是仅记录资料且当前没有待记录会话，则直接打开出关记录弹窗，不创建待记录会话。

### 6.3 学习后记录

1. 用户点击“出关记录”。
2. 弹窗自动带入当前项目和资料。
3. 用户更新出关进度、参悟进度和下次闭关目标。
4. 用户保存。
5. 系统生成学习记录。
6. 系统更新资料进度、项目进度、最近出关时间。
7. 系统显示进度变化反馈。

### 6.4 下次恢复

1. 用户重新打开工具。
2. 首页继续显示上次推荐资料。
3. 用户不用回忆，即可看到：
   - 上次学到哪里
   - 完成多少
   - 项目整体进度
   - 下一步做什么

### 6.5 用户故事

#### 下次继续学习

作为一个同时学习多个资料的用户，
我希望打开工具后马上看到最应该继续的资料、上次学到的位置和下一步任务，
这样我不用回忆就能继续学习。

验收标准：

- 首页第一屏显示推荐资料。
- 推荐资料显示进度描述、进度百分比、项目总进度和下一步任务。
- 关闭重开后仍显示同一份可继续资料。

#### 学习后快速记录

作为一个刚结束学习的用户，
我希望用很少字段记录这次学到哪里、完成多少、下次做什么，
这样我不会因为记录麻烦而放弃维护进度。

验收标准：

- 出关记录可以在 30 秒内完成。
- 参悟进度是唯一必填进度字段。
- 保存后首页、项目详情、资料详情的进度同步更新。

#### 添加新资料

作为一个刚开始学习新资料的用户，
我希望能快速添加资料并填写初始进度和下一步任务，
这样它能立即进入后续学习闭环。

验收标准：

- 新资料必须属于一个项目。
- 新资料默认进度为 0。
- 添加后可以从首页或项目详情继续学习。

#### 主题不干扰理解

作为一个不熟悉修仙设定的用户，
我希望即使界面使用修仙文案，也能清楚理解真实动作和真实进度，
这样主题不会增加使用成本。

验收标准：

- 关键按钮必须同时表达真实动作。
- 进度始终显示真实百分比。
- 下一步任务始终显示用户填写的真实内容。

## 7. 页面需求

页面信息优先级：

1. 下次继续学什么。
2. 当前学到哪里。
3. 完成百分比是多少。
4. 下一步具体做什么。
5. 保存后获得什么反馈。

修仙主题、最近记录和成就反馈不能抢占以上信息的视觉优先级。

### 7.1 首页：修炼总览

首页第一屏必须看到学习进度，不做大面积游戏化封面。

必须模块：

1. 顶部栏
2. 继续闭关卡片
3. 待记录提示条
4. 修炼方向列表
5. 最近参悟记录

顶部栏显示：

- 页面标题：修炼总览
- 当前日期
- 本地保存状态
- 新增修炼方向按钮
- 新增功法 / 资料按钮

继续闭关卡片字段：

- 修炼方向：项目名称
- 当前功法：资料名称
- 资料类型
- 当前参悟位置：进度描述
- 参悟进度：资料进度百分比
- 参悟进度条
- 修炼总进度：项目进度百分比
- 下次闭关目标：下一步任务
- 最近出关：最近出关时间
- 主按钮：继续闭关：打开资料继续学习
- 次按钮：出关记录：保存本次学习进度

示例：

```text
继续闭关
修炼方向：Rust 学习
当前功法：The Rust Book
当前参悟位置：第 4 章 Borrowing 小节
参悟进度：35%
修炼总进度：53%
下次闭关目标：继续阅读 References and Borrowing
```

#### 首页推荐规则

继续闭关卡片推荐一个资料。推荐必须稳定、可复现。

规则：

推荐排序与项目状态派生的优先级故意不同。首页推荐强调“延续手头进行中的任务”，所以 `learning` 高于 `review`；项目状态强调“有需复习就标红提醒”，所以项目派生中 `review` 高于 `learning`，见 §8.1。

1. 推荐候选只包含 `learning`、`review`、`not_started`。
2. `learning` 优先级最高；有多个学习中资料时，选择最近学习的资料。
3. 没有 `learning` 时，推荐 `review`；没有 `review` 时，推荐 `not_started`。
4. `paused` 和 `completed` 不进入首页推荐；暂停资料只能从项目详情或资料详情继续操作。
5. `last_studied_at` 为空时必须排在有出关时间的资料之后，等价于 `NULLS LAST`。
6. 候选资料排序固定为：状态优先级 ASC、`last_studied_at IS NULL` ASC、`last_studied_at DESC`、`updated_at DESC`、`created_at DESC`、`id ASC`。
7. 如果没有候选资料，首页显示“暂无需要继续的功法”。

#### 待记录提示条

触发条件：用户点击“继续闭关”成功打开资料，但尚未保存出关记录。

显示：

- 你有一次闭关尚未出关记录
- 资料名称：默认显示待记录会话中的 `resource_title_snapshot`
- 开始时间
- 打开前进度
- 记录本次学习按钮
- 放弃本次记录按钮

如果当前资料标题与 `resource_title_snapshot` 不一致，提示条显示：

```text
打开时：{resource_title_snapshot}
当前标题：{current_resource_title}
```

这样用户能看出这次待记录属于打开时的哪份资料，同时知道资料已被重命名。

待记录提示条文案：

```text
你有一次闭关尚未出关记录
{资料名称} · 开始于 {开始时间} · 打开前进度 {progress_before_percent}%
```

#### 首页状态

| 状态 | 触发条件 | 页面表现 | 主操作 |
| --- | --- | --- | --- |
| 首次空状态 | 无项目 | 显示空状态，不显示继续闭关卡片 | 创建修炼方向 |
| 有项目无资料 | 至少 1 个项目，但无资料 | 继续闭关卡片显示引导态，项目列表正常显示 | 添加功法 / 资料 |
| 有资料无推荐 | 资料全部完成，或无可继续资料 | 继续闭关显示“暂无需要继续的功法” | 添加资料 / 查看最近记录 |
| 正常推荐 | 存在可推荐资料 | 显示继续闭关卡片完整信息 | 继续闭关 |
| 待记录 | 存在 `pending_study_sessions` | 首页顶部显示待记录提示条，继续闭关卡片仍可见 | 记录本次学习 |
| 加载中 | 应用启动读取本地数据 | 保留页面结构，关键区域显示轻量骨架 | 无 |
| 读取失败 | 数据库读取失败 | 显示错误说明和重试入口 | 重试 |

空状态文案：

- 无项目：还没有修炼方向。先创建一个学习主题。
- 有项目无资料：这个修炼方向还没有功法 / 资料。添加一个资料后就可以记录进度。
- 无下一步任务：还没有设置下次闭关目标。
- 全部完成：暂无待闭关功法。当前资料都已参悟完成。

### 7.2 项目详情页：修炼方向详情

必须展示：

- 修炼方向名称
- 描述
- 状态
- 修炼总进度
- 功法数量
- 最近出关时间
- 功法列表
- 最近修炼日志

必须操作：

- 新增功法 / 资料
- 编辑修炼方向
- 删除修炼方向
- 从资料列表继续闭关
- 从资料列表出关记录

功法列表字段：

- 功法名称
- 类型
- 状态
- 当前参悟位置
- 参悟进度
- 下次闭关目标
- 最近出关时间

列表排序与最近记录数量：

1. 首页项目列表按 `last_studied_at IS NULL` ASC、`last_studied_at DESC`、`updated_at DESC`、`created_at DESC`、`id ASC` 排序。
2. 项目详情的资料列表按推荐状态优先级排序：`learning`、`review`、`not_started`、`paused`、`completed`，同状态内再按 `last_studied_at IS NULL` ASC、`last_studied_at DESC`、`updated_at DESC`、`created_at DESC`、`id ASC`。
3. 首页最近修炼日志最多显示 5 条。
4. 项目详情最近修炼日志最多显示 5 条。
5. 资料详情最近修炼日志最多显示 10 条。

列表容量策略：

1. 首页项目列表首屏最多渲染 50 个项目；超过 50 个使用分页或虚拟滚动。
2. 项目详情资料列表首屏最多渲染 100 个资料；超过 100 个使用分页或虚拟滚动。
3. Repository 查询必须支持 `limit` / `offset` 或 cursor 参数，前端不得一次性渲染 5000 条资料。
4. 首页推荐、项目摘要和最近记录查询必须独立于长列表分页，不能因为列表分页而缺失关键推荐信息。

### 7.3 资料详情页：功法详情

资料详情页必须把进度放在最醒目的位置。

必须展示：

- 功法名称
- 所属修炼方向
- 类型
- 打开方式
- 路径或链接
- 状态
- 当前参悟位置
- 参悟进度
- 参悟进度条
- 下次闭关目标
- 最近打开时间
- 最近出关时间
- 最近修炼日志，最多 10 条

必须操作：

- 继续闭关
- 出关记录
- 编辑资料
- 删除资料

状态文案：

| 状态 | 展示重点 | 操作 |
| --- | --- | --- |
| 未开始 | 当前参悟位置显示“尚未开始参悟” | 继续闭关、出关记录、编辑 |
| 参悟中 | 显示进度描述、百分比、下次目标、最近记录 | 继续闭关、出关记录 |
| 待破难点 | 状态标签显示“待破难点”，下一步任务优先展示 | 继续闭关、出关记录 |
| 暂停 | 显示暂停状态，不进入首页优先推荐 | 出关记录、编辑 |
| 参悟完成 | 显示完成反馈，不进入首页推荐 | 出关记录、编辑、删除 |

默认占位：

- 当前参悟位置：尚未记录学到哪里
- 下次闭关目标：还没有设置下次闭关目标
- 最近出关：暂无出关记录
- 最近打开：暂无打开记录

资料进度达到 100% 时展示：

```text
参悟完成
这份资料已完成 100%，可以整理笔记，或进入下一份资料。
```

### 7.4 出关记录弹窗

出关记录用于低成本保存一次学习结果，不能变成复杂日志编辑器。

默认带入：

- 修炼方向
- 当前功法
- 当前参悟位置
- 当前参悟进度
- 当前下次闭关目标
- 开始时间，如果来自待记录状态

表单字段：

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| 出关进度 | 否 | 学习后学到哪里 |
| 参悟进度 | 是 | 0 到 100 |
| 下次闭关目标 | 否 | 下次继续做什么 |
| 功法状态 | P1 | 未开始、学习中、暂停、待破难点、参悟完成；P0 可由进度自动推导 |
| 本次参悟 | P1 | 简短备注，不能成为必填日志 |
| 闭关时长 | P1 | 可自动估算，也可为空 |

默认交互：

- 打开弹窗时，光标默认落在“出关进度”。
- `参悟进度` 默认填入当前资料进度。
- `下次闭关目标` 默认填入当前资料的下一步任务，允许清空。
- 如果来自待记录状态，自动显示开始时间和估算闭关时长。
- 保存中禁用按钮，按钮文案变为“保存中...”。
- 保存失败时弹窗不关闭，保留所有输入。

闭关时长估算：

1. 只有来自待记录会话时才自动估算。
2. 估算公式为 `round((now_utc - opened_at) / 60 秒)`，最小值为 0 分钟。
3. 估算值大于 12 小时时，默认不填入，只提示“可能包含挂机时间，可手动填写”。
4. `record_only` 和手动补记没有打开起点，闭关时长默认为空。
5. 手动填写的闭关时长必须在 0 到 1440 分钟之间。

资料更新时间快照：

1. 待记录会话使用打开资源时保存的 `resource_updated_at_before`。
2. `record_only` 和手动补记没有打开起点，打开弹窗时必须即时读取资料当前 `updated_at`，作为 `resource_updated_at_before`。
3. 三种来源保存时都执行同一套资料变更冲突检测，见 §9.12。
4. 如果弹窗打开后资料被其他操作修改，保存时提示资料已变化，允许确认覆盖或取消。

校验：

- 参悟进度必须是 0 到 100 的整数。
- 资料级记录必须关联资料。
- 即使进度、下一步和状态都没有变化，也允许保存一条学习记录；这表示“今天学了，但暂无进展”。
- 保存失败时保留用户输入。
- 新进度小于旧进度时必须二次确认，但允许保存，因为真实学习记录优先。
- 如果用户已修改内容后点击关闭、ESC、遮罩或返回，必须提示“尚未保存本次出关记录，确认离开？”。
- 选择“参悟完成”时，界面即时说明“进度将设为 100%”。
- 选择“未开始”时，界面即时说明“进度将设为 0%”。

MVP 不提供“同步更新资料”开关。保存出关记录时，必须同步更新资料当前进度、下一步任务、状态和最近出关时间。

出关记录来源不同，取消和关闭规则不同：

| 来源 | 打开方式 | 保存后 | 取消 / 关闭且无修改 | 关闭且有未保存修改 |
| --- | --- | --- | --- | --- |
| 待记录会话 | 从待记录提示条进入 | 创建日志，更新资料，清除待记录 | 不清除待记录，用户可稍后再记 | 二次确认，确认后仍保留待记录 |
| 仅记录资料 | `record_only` 继续闭关进入 | 创建日志，更新资料，不更新 `last_opened_at` | 不创建日志，不更新任何出关时间，不产生待记录 | 二次确认，确认后不保存任何内容 |
| 手动补记 | 从详情页或首页点击出关记录 | 创建日志，更新资料，不影响既有待记录 | 不创建日志，不更新资料 | 二次确认，确认后不保存任何内容 |

### 7.5 全局交互状态

| 状态 | 触发场景 | 页面反馈 |
| --- | --- | --- |
| 空状态 | 无项目、无资料、无记录 | 给出一句说明和一个主操作 |
| 加载中 | 应用启动、页面切换、保存中 | 显示轻量加载状态，不阻塞已加载内容 |
| 保存成功 | 创建、编辑、出关记录保存成功 | 显示短提示，并刷新相关进度 |
| 保存失败 | 数据库写入失败 | 保留用户输入，提示稍后重试 |
| 打开失败 | 路径不存在、URL 无效、权限不足 | 提示原因，并提供修改路径或仅记录进度 |
| 待记录 | 成功打开资料但未保存出关记录 | 显示待记录提示条 |
| 删除确认 | 删除项目或资料前 | 显示影响范围，要求二次确认 |

错误与恢复文案：

| 场景 | 文案 | 恢复操作 |
| --- | --- | --- |
| 保存失败 | 保存失败，刚才填写的内容已保留。请稍后重试。 | 重试 / 取消 |
| 数据读取失败 | 未能读取本地修炼记录。请重试，或检查本地数据文件。 | 重试 |
| 删除项目确认 | 删除后会移除该修炼方向、其下功法和相关记录。此操作不可撤销。 | 确认删除 / 取消 |
| 删除资料确认 | 删除后资料会移除，历史记录会保留资料名称快照。 | 确认删除 / 取消 |
| URL 协议不支持 | 只支持 http:// 或 https:// 链接。 | 修改链接 |
| 权限不足 | 没有权限打开该路径。可以修改路径，或仅记录进度。 | 修改路径 / 仅记录进度 |

### 7.6 交互状态机

MVP 必须用统一状态机管理“继续闭关”和“出关记录”，避免页面之间状态漂移。

| 状态 | 含义 | 可进入方式 | 可退出方式 |
| --- | --- | --- | --- |
| `none` | 当前没有打开或待记录会话 | 应用启动、保存完成、放弃完成 | 点击继续闭关 |
| `opening` | 正在请求后端打开资源 | 点击继续闭关 | 打开成功、打开失败、被阻断 |
| `pending` | 资源已打开，等待出关记录 | 文件、文件夹、URL 打开成功 | 保存出关记录、放弃本次记录 |
| `recording` | 出关记录弹窗打开 | 待记录、仅记录、手动补记 | 保存、取消、关闭 |
| `saving` | 正在保存出关记录 | 点击保存 | 保存成功、保存失败 |
| `saved` | 出关记录保存成功 | 保存事务提交成功 | 自动回到 `none` |
| `abandoned` | 用户放弃待记录 | 点击放弃本次记录 | 自动回到 `none` |
| `failed` | 打开或保存失败 | 后端返回错误 | 修改路径、仅记录、重试、取消 |

状态机规则：

1. `opening` 期间禁用同一资料的继续闭关按钮，避免重复创建待记录。
2. 打开失败必须进入 `failed`，不得创建 `pending_study_sessions`。
3. 保存失败必须停留在 `recording`，保留用户输入。
4. `record_only` 在没有待记录会话时直接从 `none` 进入 `recording`，不得进入 `opening` 或 `pending`。
5. 已有待记录会话时，任何 `continue_resource` 调用都必须先进入冲突处理，包括 `record_only`。
6. 手动补记不受待记录会话阻断；保存手动补记后不得清除已有待记录会话。
7. `saved` 和 `abandoned` 都必须让首页重新拉取最新数据。

### 7.7 待记录冲突处理

当已经存在 `pending_study_sessions` 时，用户再次点击任意资料的“继续闭关”，系统不得静默覆盖。

该规则适用于所有 `continue_resource({ resource_id })`，包括 `record_only`。`record_only` 虽然不创建待记录会话，但不能绕过已有待记录会话的处理提醒。

必须弹出冲突提示：

```text
已有一次闭关尚未出关记录
请先处理上一次学习记录，再开始新的闭关。
```

操作：

| 操作 | 行为 |
| --- | --- |
| 记录上一次 | 打开已有待记录会话的出关记录弹窗 |
| 放弃上一次 | 清除已有待记录，不创建日志，不更新出关时间，然后允许用户重新点击新资料 |
| 取消本次操作 | 保留已有待记录，不打开新资料 |

放弃提示文案：

```text
放弃后不会保存本次学习进度，也不会更新最近出关时间。
```

手动补记并发规则：

1. 用户从首页或详情页直接点击“出关记录”属于手动补记。
2. 手动补记允许在已有待记录会话存在时打开和保存。
3. 手动补记保存后只更新所选资料及其项目，不清除、不覆盖已有待记录会话。
4. 手动补记弹窗必须明确显示当前记录的资料名，避免用户误以为正在处理待记录会话。

### 7.8 首次使用引导链路

首次使用必须形成连续链路，不能让用户创建项目后不知道下一步。

规则：

1. 无项目时，“新增功法 / 资料”按钮禁用或提示“请先创建修炼方向”。
2. 创建项目成功后，提示用户继续添加第一个资料。
3. 添加资料时默认进度为 0，默认状态为未开始。
4. 添加资料取消后，首页回到“有项目无资料”状态。
5. 添加资料成功后，回到首页并显示继续闭关卡片。

### 7.9 长文本与可访问性

MVP 的核心信息必须在不同屏幕尺寸下保持可读。

要求：

1. MVP 面向桌面应用，不以手机 viewport 作为验收基准；最小窗口尺寸为 1024x640。
2. 资料名、路径、下一步任务过长时允许换行或中间截断，不得覆盖按钮和进度条。
3. 路径展示默认折叠中间段，避免暴露完整用户名目录。
4. 关键按钮必须支持键盘聚焦。
5. 弹窗关闭按钮、保存按钮、取消按钮必须有可见焦点。
6. 进度条必须同时显示文本百分比，不能只靠颜色表达。
7. 正文和关键数据在 100% 与 125% 系统缩放下不重叠、不截断核心含义。
8. 深色或修仙视觉主题必须满足 WCAG AA 文本对比度；关键操作按钮对比度不得低于 4.5:1。
9. 关键按钮、表单输入、进度条必须有可访问名称；M5 至少用系统屏幕阅读器做一次首页和出关弹窗冒烟验证。
10. 成就反馈使用非阻塞 Toast，不得用大弹窗打断连续记录。

键盘与弹窗规则：

1. 出关记录弹窗打开后焦点落在“出关进度”；关闭后焦点回到触发按钮。
2. 弹窗内必须启用 focus trap，Tab / Shift+Tab 只在弹窗可聚焦元素中循环。
3. Enter 在文本输入框内不提交表单；在保存按钮聚焦时提交。
4. ESC 等价于关闭弹窗；有未保存输入时必须出现离开确认。
5. Tab 顺序按视觉顺序：标题、进度、下一步、状态、备注、时长、取消、保存。

Toast 规则：

1. 成功 Toast 默认停留 3 秒。
2. 错误 Toast 默认停留 5 秒，并提供关闭按钮。
3. 多条 Toast 按出现顺序排队，最多同时展示 3 条。
4. Toast 不抢焦点，不阻断键盘操作。

### 7.10 低保真线框与视觉 token

MVP 不要求高保真视觉稿，但必须先用低保真线框固定信息层级。

首页线框：


## 8. 功能需求

### 8.1 项目管理

MVP 支持：

- 创建项目
- 编辑项目名称
- 编辑项目描述
- 删除项目
- 查看项目列表
- 查看项目详情

项目字段：

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| 名称 | 是 | 不能为空或纯空格 |
| 描述 | 否 | 学习目标说明 |
| 状态 | 自动 | 默认未启修，由资料状态派生，不在项目编辑表单中手动修改 |
| 最近出关时间 | 自动 | 保存出关记录后更新 |

项目进度：

- 不允许用户手动编辑。
- 由资料进度自动计算。
- 无资料时显示 0%。

项目名称规则：

1. MVP 允许项目重名，不设置数据库唯一约束。
2. 重名项目在列表中通过描述、状态、总进度和最近出关时间区分。
3. 项目名称仍必须 trim 后非空，并遵守 §9.5 的长度上限。

项目状态派生规则：

项目状态派生强调“有需复习就标红提醒”，因此 `review` 高于 `learning`；这与首页推荐的“延续手头进行中的任务”不同，见 §7.1。以下规则按编号顺序求值，命中即返回。

1. 项目没有资料，或所有资料都是 `not_started` 时，项目状态为 `not_started`。
2. 所有资料都是 `completed` 时，项目状态为 `completed`。
3. 任一资料为 `review` 时，项目状态为 `review`。
4. 任一资料为 `learning` 时，项目状态为 `learning`。
5. 没有 `learning` / `review`，但至少一个资料为 `paused` 且未全部完成时，项目状态为 `paused`。
6. 保存出关记录、添加资料、删除资料、修改资料非终态状态后，都必须重新派生项目状态。

示例：

| 资料状态组合 | 项目状态 | 原因 |
| --- | --- | --- |
| `learning` + `review` | `review` | 规则 3 先命中，项目层需要突出待破难点 |
| `learning` + `paused` | `learning` | 规则 4 命中，仍有进行中资料 |
| `paused` + `completed` | `paused` | 未全部完成且没有学习中/待破难点 |

### 8.2 资料管理

MVP 支持：

- 添加资料
- 编辑资料
- 删除资料
- 查看资料详情

资料字段：

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| 所属项目 | 是 | 必须属于一个项目 |
| 标题 | 是 | 不能为空或纯空格 |
| 类型 | 是 | 默认其他 |
| 打开方式 | 是 | 文件、文件夹、网页、仅记录 |
| 路径或链接 | 条件必填 | 仅记录时可为空，其余打开方式必须填写 |
| 状态 | 是 | 默认未开始；编辑资料时只能改非终态标记 |
| 进度描述 | 否 | 创建资料时可填写初始值；创建后只能通过出关记录变更 |
| 进度百分比 | 是 | 创建资料时可填写初始值；创建后只能通过出关记录变更 |
| 下一步任务 | 否 | 创建资料时可填写初始值；创建后只能通过出关记录变更 |

资料编辑边界：

1. “编辑资料”表单只包含标题、类型、打开方式、路径或链接、非终态状态标记。
2. 所属项目创建后不可编辑；MVP 不支持把资料移动到另一个项目。
3. MVP 允许同一项目下资料标题重名，不设置数据库唯一约束。
4. 创建资料时允许填写初始进度和下一步任务，因为此时还没有学习记录。
5. 创建后，`progress_text`、`progress_percent`、`next_action` 只能由 `save_study_log` 更新，并必须生成 `study_logs`。
6. `update_resource` 不接收项目 ID、进度和下一步字段；如果传入这些字段，后端必须拒绝。
7. 编辑资料不会更新 `last_studied_at`，不会创建学习记录。
8. 创建资料时如果初始进度大于 0，服务层按状态与进度规则归一化初始状态。
9. 编辑资料修改非终态状态时，当前进度必须已经在 1 到 99；否则后端拒绝，并提示通过出关记录变更进度。

打开方式变更规则：

1. 从 `file`、`folder`、`url` 切换到 `record_only` 时，`path_or_url` 必须归一化为 `null`。
2. 从 `record_only` 切换到 `file`、`folder` 或 `url` 时，必须重新填写合法的路径或 URL。
3. 在 `file`、`folder`、`url` 之间切换时，必须按目标打开方式重新校验 `path_or_url`；旧值可以预填在表单中，但保存必须满足目标类型规则。
4. 打开方式变更只更新资料元信息，不更新 `last_opened_at`、`last_studied_at`，也不创建待记录会话。
5. 如果资料存在待记录会话，仍允许编辑打开方式；保存出关记录时按 §9.12 的资料变化冲突处理。

资料类型：

- 文档
- 视频
- 网页
- 课程
- 代码仓库
- 练习
- 书籍
- 其他

打开方式：

- 文件
- 文件夹
- 网页
- 仅记录

### 8.3 学习进度

学习进度是 MVP 核心。

每个资料必须能记录：

- 当前进度描述
- 当前进度百分比
- 当前状态
- 下一步任务
- 最近出关时间

进度规则：

1. 新资料默认进度为 0。
2. 进度百分比必须在 0 到 100。
3. 状态与进度归一化以 §9.8 为权威定义。
4. 项目进度计算以 §9.7 为权威定义。

### 8.4 继续闭关

继续闭关用于打开学习资料，或进入仅记录流程。

规则：

1. 文件：检查文件存在后，用系统默认应用打开。
2. 文件夹：检查文件夹存在后，用系统文件管理器打开。
3. 网页：只允许 `http://` 和 `https://`。
4. 仅记录：不调用系统打开能力；没有待记录会话时直接进入出关记录，有待记录会话时先进入冲突处理。
5. 打开成功后更新 `last_opened_at`。
6. 打开成功后不更新 `last_studied_at`。
7. 出关记录保存后才更新 `last_studied_at`。
8. 打开失败时不进入待记录状态。

#### record_only 专项规则

`record_only` 是为了记录没有固定文件或链接的学习对象，例如纸质书、线下课程、口头学习任务。

必须满足：

1. 不调用系统打开能力。
2. 点击“继续闭关”后，如果当前没有待记录会话，直接打开出关记录弹窗。
3. 如果当前已有待记录会话，必须先返回待记录冲突处理，不得绕过 pending 直接记录。
4. 不创建 `pending_study_sessions`。
5. 用户取消弹窗后，不创建学习记录。
6. 用户取消弹窗后，不更新 `last_opened_at`。
7. 用户取消弹窗后，不更新 `last_studied_at`。
8. 用户取消弹窗后，不留下待记录状态。
9. 保存成功后，创建学习记录并更新 `last_studied_at`。
10. 保存成功后，仍然不更新 `last_opened_at`。
11. 保存成功后，不创建、不清理 `pending_study_sessions`。

### 8.5 出关记录

保存出关记录后必须：

1. 创建一条学习记录。
2. 更新资料进度。
3. 更新资料下一步任务。
4. 更新资料状态。
5. 更新资料最近出关时间。
6. 更新项目最近出关时间。
7. 重新计算项目进度；MVP 默认实时查询计算，不保存项目进度缓存。
8. 如果来源是待记录会话，清除待记录状态；手动补记不得清除既有待记录；`record_only` 保存只允许在无待记录会话时发生。
9. 显示进度反馈。

放弃本次记录后：

- 不创建学习记录。
- 不更新资料进度。
- 不更新下一步任务。
- 不更新最近出关时间。
- 如果放弃的是待记录会话，清除待记录状态；如果取消的是 `record_only` 或手动补记，不触碰既有待记录。

### 8.6 成就感反馈

MVP 只做即时反馈。

触发：

| 触发条件 | 反馈 |
| --- | --- |
| 保存出关记录 | 出关记录已保存 |
| 资料进度提升 | 显示提升百分比 |
| 资料进度下降 | 显示已记录回退，不显示负向“提升” |
| 进度和下一步无变化 | 显示“已记录本次学习，暂无进度变化” |
| 项目进度提升 | 显示项目进度变化 |
| 资料达到 100% | 显示参悟完成 |

文案示例：

```text
出关记录已保存
参悟进度提升 5%，当前进度 40%。
下次闭关目标：做 ownership 相关练习
```

```text
功法参悟完成
The Rust Book 已完成 100%。可以整理笔记，或继续下一份资料。
```

```text
已记录本次学习
本次暂无进度变化，最近出关时间已更新。
```

```text
进度回退已记录
当前进度调整为 30%。保留真实学习记录优先。
```

反馈不应阻塞用户继续操作。

## 9. 数据模型

MVP 推荐使用 SQLite。

MVP 项目进度默认通过查询实时计算，不保存项目进度缓存字段。这样可以减少事务漏同步风险。用户不能手动编辑项目进度。

规则单一事实源：

1. 状态与进度归一化以 §9.8 为权威定义，其他章节只描述用户可见行为。
2. 项目进度计算以 §9.7 为权威定义。
3. 推荐排序以 §7.1 的产品规则和 §9.13 的 SQL 排序口径共同定义；实现必须同时满足。
4. `record_only` 与待记录会话冲突以 §7.7 和 §8.4 为权威定义。
5. 如果后续章节和以上权威定义冲突，以本节指定的权威定义为准，并修正文档重复表述。

### 9.0 时间与时区规则

1. 数据库中所有时间字段统一存 UTC RFC3339，带 `Z` 或明确 offset。
2. UI 展示时间时转换为操作系统本地时区。
3. 顶部栏“当前日期”按本地时区计算。
4. MVP 不按“今天”过滤推荐资料；首页卡片命名为“继续闭关”，只表示当前推荐入口。
5. 如果未来增加真正的“今日学习”统计，日期边界必须按本地时区自然日计算，再转换为 UTC 查询范围。
6. 列表和卡片中使用相对时间，例如“2 小时前”“昨天 21:30”；详情和日志中同时显示本地绝对时间，例如 `2026-05-30 21:30`。
7. 后端只接受可解析的 RFC3339 时间；无时区、非法日期、闰秒等不能解析的输入一律拒绝。
8. 如果系统时间回拨导致 `now_utc < opened_at`，估算时长按 0 分钟处理，并在本地日志记录 `clock_skew_detected`。

### 9.5 数据约束

必须约束：

- 项目名称 trim 后不能为空。
- 资料标题 trim 后不能为空。
- 资料进度必须是 0 到 100 的整数。
- 学习时长为空，或在 0 到 1440 分钟之间。
- 状态、资料类型、打开方式必须是枚举值。
- SQLite 必须开启外键。
- 删除项目时删除项目下资料和学习记录。
- 删除资料时保留学习记录，并将 `resource_id` 置空。
- 出关记录保存必须在事务内完成。
- `study_logs.resource_title_snapshot` 必须非空。
- `study_logs.progress_before_percent` 和 `study_logs.progress_after_percent` 必须是 0 到 100 的整数。
- `pending_study_sessions.progress_before_percent` 必须是 0 到 100 的整数。
- `record_only` 的 `path_or_url` 必须为空或保存为空字符串归一化后的 `null`。
- 非 `record_only` 的 `path_or_url` trim 后必须非空。
- 新建学习记录时 `resource_updated_at_before` 必须来自打开弹窗时的资料 `updated_at`；仅兼容旧版本历史数据时允许为空。
- 项目名称和资料标题不设置唯一约束；同名数据允许保存。
- `resources.project_id` 创建后不可通过更新资料命令修改。

### 9.7 项目进度计算

项目进度不允许用户手动编辑。

计算公式：

```text
项目进度 = round(项目下所有资料 progress_percent 的平均值)
```

规则：

1. 项目没有资料时，项目进度为 0%。
2. 资料进度先按 §9.8 完成状态与进度归一化，再参与平均值计算。
3. 删除资料后，项目进度立即重新计算。
4. 项目进度用于展示，不作为用户输入项。
5. MVP 不保存项目进度缓存字段；如果未来引入缓存，必须在同一事务中同步更新。

示例：

| 资料 | 进度 |
| --- | --- |
| A | 20 |
| B | 40 |
| C | 100 |

项目进度 = `round((20 + 40 + 100) / 3) = 53%`

### 9.8 状态与进度归一化

服务层必须统一执行状态与进度归一化，前端不能各自推断。

规则：

1. `completed` 与进度 100 绑定；保存为 `completed` 时，进度强制写入 100。
2. `not_started` 与进度 0 绑定；保存为 `not_started` 时，进度强制写入 0。
3. `learning`、`review`、`paused` 的有效进度范围是 1 到 99。
4. 用户输入进度 100 且未显式选择状态时，状态归一化为 `completed`。
5. 用户输入进度 0 且未显式选择状态时，状态归一化为 `not_started`。
6. 用户同时输入 `completed` 和进度 50 时，`completed` 优先，最终保存进度 100，界面必须提前提示“进度将设为 100%”。
7. 用户同时输入 `not_started` 和进度 50 时，`not_started` 优先，最终保存进度 0，界面必须提前提示“进度将设为 0%”。
8. 从 `completed` 改回 `learning`、`review` 或 `paused`，必须同时提交 1 到 99 的进度；否则最终仍为 `completed` + 100。
9. 从 `not_started` 改为 `learning`、`review` 或 `paused`，必须同时提交 1 到 99 的进度；否则最终仍为 `not_started` + 0。
10. `update_resource` 只能修改资料元信息和非终态状态标记，不得触发进度归一化；当当前进度不是 1 到 99 时，非终态状态变更必须被拒绝。
11. 终态变更必须通过 `save_study_log` 生成学习记录。
12. P0 不提供手动选择 `review` / `paused` 的入口；P0 保存时 1 到 99 默认归一化为 `learning`。P1 起才允许在出关记录中选择 `review` / `paused`。

### 9.9 事务规则

以下操作必须使用事务，保证数据一致：

1. 保存出关记录。
2. 删除项目。
3. 删除资料。
4. 保存出关记录时更新资料状态并联动进度。
5. 成功继续闭关后写入打开时间与待记录会话。

保存出关记录的事务必须同时完成：

1. 创建 `study_logs`。
2. 更新 `resources.progress_text`。
3. 更新 `resources.progress_percent`。
4. 更新 `resources.next_action`。
5. 更新 `resources.status`。
6. 更新 `resources.last_studied_at`。
7. 更新 `projects.last_studied_at`。
8. 更新 `resources.updated_at`。
9. 更新 `projects.updated_at`。
10. 删除或清空对应 `pending_study_sessions`；手动补记保存不得触碰既有待记录表。`record_only` 正常保存只允许发生在无 pending 的前提下，有 pending 时必须先进入冲突处理。

任一步失败时，整个操作回滚，界面保留用户输入。

继续闭关成功后的事务必须同时完成：

1. 更新 `resources.last_opened_at`。
2. 创建唯一 `pending_study_sessions`。
3. 写入打开时资料名称、进度、状态、下一步和更新时间快照。

如果任一步失败，必须回滚，不得出现“最近打开时间已更新但没有待记录会话”的状态。

### 9.10 SQLite 强约束

数据库要求：

1. 每个 SQLite 连接必须执行并验证 `PRAGMA foreign_keys = ON`。
2. 启动自检 `PRAGMA foreign_key_check` 必须为空。
3. 项目进度不保存为用户可编辑字段，优先实时查询计算。
4. `projects.name` 必须使用 `CHECK(length(trim(name)) > 0)`。
5. `resources.title` 必须使用 `CHECK(length(trim(title)) > 0)`。
6. `resources.progress_percent` 必须使用 `CHECK(progress_percent BETWEEN 0 AND 100)`。
7. `study_logs.duration_minutes` 必须允许为空，填写时必须 `BETWEEN 0 AND 1440`。
8. `status`、`type`、`open_kind` 必须使用枚举 `CHECK`。
9. `open_kind != record_only` 时，`path_or_url` trim 后必须非空。
10. `open_kind = record_only` 时，`path_or_url` 必须为空或归一化为空。
11. `study_logs.resource_title_snapshot` 必须 `NOT NULL`。
12. `study_logs.progress_before_percent` 必须使用 `CHECK(progress_before_percent BETWEEN 0 AND 100)`。
13. `study_logs.progress_after_percent` 必须使用 `CHECK(progress_after_percent BETWEEN 0 AND 100)`。
14. `pending_study_sessions.progress_before_percent` 必须使用 `CHECK(progress_before_percent BETWEEN 0 AND 100)`。
15. `pending_study_sessions.resource_title_snapshot` 必须 `NOT NULL`。
16. `completed` 状态必须对应进度 100，`not_started` 状态必须对应进度 0；可由服务层强制归一化并由测试覆盖。
17. `pending_study_sessions` 必须有数据库级单例约束。
18. 主要文本字段必须落实 9.5 的长度上限；超长输入由服务层拒绝，数据库层至少覆盖必填名称和长备注。

待记录单例约束推荐使用固定单例键：

```sql
singleton_key integer not null default 1 check(singleton_key = 1),
unique(singleton_key)
```

外键行为：

| 关系 | 行为 |
| --- | --- |
| `resources.project_id -> projects.id` | 项目删除时级联删除资料 |
| `study_logs.project_id -> projects.id` | 项目删除时级联删除日志 |
| `study_logs.resource_id -> resources.id` | 资料删除时置空 |
| `pending_study_sessions.project_id -> projects.id` | 项目删除时级联删除待记录 |
| `pending_study_sessions.resource_id -> resources.id` | 资料删除时级联删除待记录 |

历史日志要求：

- 创建 `study_logs` 时必须写入 `resource_title_snapshot`。
- 创建 `study_logs` 时必须写入 before/after 进度和状态快照。
- 资料删除后，`study_logs.resource_id` 可为空，但历史资料名称不可丢。

### 9.11 迁移规则

MVP 必须有数据库迁移机制。

要求：

1. 使用 `schema_migrations(version, applied_at, checksum)` 记录迁移版本。
2. 迁移按版本顺序执行。
3. 单个迁移必须在事务中执行。
4. 迁移失败不得留下半升级 schema。
5. 迁移脚本必须可重复检测，不能重复应用。
6. M0 阶段必须覆盖空库初始化和旧版本升级测试。
7. 已应用版本的 checksum 与当前迁移文件不一致时，启动必须失败并提示迁移异常。
8. 迁移前必须开启并验证外键。
9. SQLite 表重建类迁移必须先校验旧数据是否满足新约束。
10. 迁移失败后不得继续启动到半升级数据库。
11. checksum 不一致时，只读诊断页必须提示可执行动作：使用最近备份恢复、保留当前 DB 并退出、打开数据目录。
12. 从旧备份恢复后，应用按备份中的 `schema_migrations` 版本重新判断是否需要迁移。
13. 如果同一迁移版本连续失败，应用不得反复自动重试；必须停在只读诊断页等待用户选择恢复或退出。

### 9.12 待记录会话规则

MVP 同一时间只允许存在一个待记录会话。

规则：

1. 成功打开资料后创建待记录会话。
2. 打开失败不得创建待记录会话。
3. `record_only` 不得创建待记录会话。
4. 已有待记录会话时，再次继续闭关不能静默覆盖；`record_only` 也必须先进入冲突处理。
5. 用户必须先选择：记录本次学习、放弃本次记录、取消新操作。
6. 保存出关记录后清除待记录会话。
7. 放弃本次记录后清除待记录会话。
8. 应用关闭重开后，待记录会话必须恢复。
9. 待记录会话必须保存打开时资料标题、状态、进度、下一步和 `updated_at` 快照。
10. 如果待记录期间资料被编辑，保存出关记录时必须提示用户当前资料已变化，允许确认覆盖或取消。
11. 手动补记允许与一个待记录会话并存；保存手动补记不得清除待记录会话。

删除含待记录会话的资料：

1. 删除资料前如果该资料存在待记录会话，确认弹窗必须额外提示“这会放弃尚未保存的本次学习记录”。
2. 用户确认删除后，事务内先删除资料，依赖外键级联删除对应 `pending_study_sessions`。
3. 删除完成后首页必须重新读取待记录状态，不能继续显示旧提示条。
4. §9.12 中“资料已被删除”的保存时提示，主要覆盖前端已经打开出关弹窗、保存前后端重新读取资料时发现资料不存在的路径。

资料变化冲突口径：

1. 打开出关记录弹窗时，必须带入 `resource_updated_at_before`。
2. 保存前重新读取资料；只要当前 `resources.updated_at != resource_updated_at_before`，即判定资料已变化。
3. 新建学习记录时该字段不得为空；只有兼容旧版本历史数据展示时允许为空。
4. 如果旧数据该字段为空，不执行资料变化冲突判断，只展示历史记录。
5. “取消”表示不保存，保留待记录会话和用户已输入内容。
6. “确认覆盖”表示使用弹窗里用户输入的进度、状态和下一步覆盖当前资料；不是丢弃用户输入。
7. 确认覆盖创建的 `study_logs`，before 快照仍使用待记录会话中的打开前快照，after 快照使用用户确认后的最终归一化值。
8. 如果资料已被删除，不能覆盖，必须提示资料不存在并允许放弃待记录。
9. `record_only` 和手动补记的 `resource_updated_at_before` 取打开弹窗时即时读取的资料 `updated_at`，与待记录来源使用同一冲突校验。

### 9.13 推荐索引

MVP 数据量不大，但推荐提前建立以下索引，保证首页、详情页和最近记录查询稳定：

```sql
CREATE INDEX idx_resources_project_id ON resources(project_id);
CREATE INDEX idx_resources_status_last_studied_at ON resources(status, last_studied_at DESC, updated_at DESC, created_at DESC, id ASC);
CREATE INDEX idx_resources_updated_at ON resources(updated_at);
CREATE INDEX idx_study_logs_project_studied_at ON study_logs(project_id, studied_at DESC, created_at DESC);
CREATE INDEX idx_study_logs_resource_id ON study_logs(resource_id, studied_at DESC, created_at DESC);
CREATE INDEX idx_pending_study_sessions_project_id ON pending_study_sessions(project_id);
CREATE INDEX idx_pending_study_sessions_resource_id ON pending_study_sessions(resource_id);
```

推荐查询不能依赖 SQLite 对 NULL 的默认排序，必须显式写出等价的 `NULLS LAST`：

```sql
ORDER BY
  CASE status
    WHEN 'learning' THEN 1
    WHEN 'review' THEN 2
    WHEN 'not_started' THEN 3
    ELSE 9
  END ASC,
  CASE WHEN last_studied_at IS NULL THEN 1 ELSE 0 END ASC,
  last_studied_at DESC,
  updated_at DESC,
  created_at DESC,
  id ASC
```

## 10. 技术边界

推荐技术：

- 桌面壳：Electron
- 前端：React
- 数据库：SQLite

MVP 技术要求：

- 本地离线可用。
- 不依赖网络完成核心功能。
- 文件和文件夹打开必须使用系统 API。
- 不拼接 shell 命令打开本地路径。
- URL 只允许 `http://` 和 `https://`。
- 可执行文件、脚本、快捷方式、网络路径需要提示风险。
- 关闭重开后恢复数据和待记录状态。

### 10.0 本地运行与存储边界

1. MVP 使用单实例运行；已有实例运行时，再次启动应聚焦已有窗口，不同时打开两个可写实例。
2. 主数据库位于系统应用数据目录，文件名固定为 `cultivation_system.sqlite3`。
3. 备份文件与主数据库同目录，放入 `backups/` 子目录。
4. 数据库连接由后端统一管理，前端不能直接访问数据库文件。
5. 最小窗口尺寸为 1024x640；小于该尺寸时应用可以限制缩放或显示最小尺寸提示。
6. MVP 容量基线：100 个项目、5000 个资料、50000 条学习记录内，首页数据加载目标 P80 <= 2 秒。
7. 应用自动更新机制不进入 MVP；对外发布前再定义更新渠道和回滚策略。

运行环境：

1. MVP 优先支持 Windows 10 22H2 及以上、Windows 11。
2. Electron 自带 Chromium 运行时，Windows 不检测 WebView2 Runtime；打包时必须确保 Electron 运行时随包分发，GPU 初始化失败时回退软件渲染，不进入空白窗口。
3. macOS 和 Linux 暂不作为 MVP 阻断平台；若后续支持，必须补充对应最小版本、签名和打开资源规则差异。

窗口状态：

1. 记忆上次窗口大小和位置，但不得恢复到小于 1024x640。
2. 屏幕变化导致上次位置不可见时，窗口居中打开。
3. 用户缩小到最小尺寸以下时，应用阻止继续缩小或显示“窗口过小”提示，不让核心操作重叠。

### 10.1 桌面壳 / Electron 权限边界

桌面壳必须遵守最小权限原则。MVP 采用 Electron 时，文件系统、系统打开、数据库访问都只能在主进程执行，渲染进程只能经 preload 暴露的语义 IPC 调用。

要求：

1. 前端不得调用任意 shell 命令。
2. 前端不得传任意 path/url 让后端直接打开。
3. 前端只能调用语义命令，例如 `continue_resource({ resource_id })`。
4. 文件、文件夹、URL 的校验和打开必须在 Electron 主进程完成。
5. 主进程打开资源前必须从数据库读取 `resource_id` 对应的 `open_kind` 和 `path_or_url`。
6. Electron 必须启用 `contextIsolation`，禁用 `nodeIntegration`，优先启用 `sandbox`。
7. preload 只暴露必要语义命令，不开放通用 shell/open/read 任意文件能力。

### 10.2 打开资源安全规则

文件和链接打开规则：

1. 文件路径必须规范化。
2. 文件打开前必须确认路径存在且是文件。
3. 文件夹打开前必须确认路径存在且是文件夹。
4. URL 必须使用 URL parser 校验。
5. URL 只允许 `http` 和 `https`。
6. 拒绝 `file:`、`javascript:`、`data:`、`mailto:` 和所有自定义协议。
7. MVP 默认阻断可执行文件、脚本文件、快捷方式和 UNC / 网络路径打开。
8. 被阻断时，提供“修改路径”或“仅记录进度”。
9. 禁止通过字符串拼接 shell 命令打开资源。

危险扩展示例：

- `.exe`
- `.bat`
- `.cmd`
- `.ps1`
- `.vbs`
- `.js`
- `.lnk`

补充风险扩展：

- `.com`
- `.scr`
- `.msi`
- `.hta`
- `.wsf`
- `.jse`
- `.vbe`
- `.jar`
- `.reg`
- `.url`
- `.scf`

宏文件如 `.docm`、`.xlsm`、`.pptm` 不硬阻断，但必须强提示。

### 10.3 技术架构边界

MVP 必须把业务规则放在后端服务层，前端只做展示和交互。

| 层 | 职责 |
| --- | --- |
| React 页面 | 展示数据、收集输入、显示加载/错误/空状态 |
| 前端 API Client | 调用 preload 暴露的语义 IPC，转换前端状态 |
| Electron IPC Handlers | 暴露稳定命令，不包含复杂业务逻辑 |
| Services | 处理业务事务、状态归一化、推荐规则、出关记录保存 |
| Repositories | 封装 SQL 查询和写入 |
| Domain | 枚举、状态/进度规则、推荐规则、校验规则 |
| DB | SQLite 连接、迁移、PRAGMA、事务 |
| Resource Opener | 文件、文件夹、URL 校验和系统打开 |
| Errors | 统一错误码、用户文案和恢复动作 |

原则：

1. SQLite 是唯一可信数据源。
2. 前端校验只用于体验优化，不能替代后端校验。
3. 推荐规则、状态与进度联动、路径安全、事务一致性必须在后端实现。
4. 页面保存后必须以后端返回的规范化实体刷新，而不是只做前端乐观更新。

### 10.4 后端命令契约

命令必须使用稳定 DTO。命令名、输入、输出和错误码在 M0 阶段冻结。

| 命令 | 输入 | 输出 | 说明 |
| --- | --- | --- | --- |
| `get_home_overview()` | 无 | 继续闭关推荐、待记录、项目摘要、最近日志 | 首页唯一数据入口 |
| `list_projects(input)` | 可选分页参数 | 项目摘要列表 | 用于项目列表，默认最多 50 条 |
| `get_project_detail(project_id, input)` | 项目 ID、可选资料分页参数 | 项目详情、资料列表、最近日志 | 项目详情页，资料默认最多 100 条 |
| `create_project(input)` | 名称、描述 | 创建后的项目 | trim 校验 |
| `update_project(project_id, input)` | 名称、描述 | 更新后的项目 | 项目状态自动派生，不允许手动传入 |
| `delete_project(project_id)` | 项目 ID | 删除结果 | 事务内级联删除 |
| `create_resource(input)` | 所属项目、标题、类型、打开方式、路径、初始进度、下一步 | 创建后的资料 | 资料必须归属项目；允许同名 |
| `update_resource(resource_id, input)` | 标题、类型、打开方式、路径、非终态状态标记 | 更新后的资料 | 不接收所属项目、进度、下一步、最近出关时间 |
| `delete_resource(resource_id)` | 资料 ID | 删除结果 | 日志保留快照，`resource_id` 置空 |
| `get_resource_detail(resource_id)` | 资料 ID | 资料详情、最近日志 | 资料详情页 |
| `continue_resource(input)` | 资料 ID、可选风险确认 token | 继续闭关结果 | 打开资源或进入仅记录流程 |
| `save_study_log(input)` | 资料 ID、进度、下一步、状态、来源、`resource_updated_at_before`、可选时长/备注 | 保存结果、更新后资料、项目进度 | 事务保存并生成日志 |
| `get_pending_session()` | 无 | 当前待记录会话或空 | 应用启动恢复 |
| `abandon_pending_session(session_id)` | 会话 ID | 放弃结果 | 不创建日志，不更新时间 |
| `get_enums()` | 无 | 枚举和展示文案 | 前端渲染下拉项 |

`continue_resource(input)` 返回状态必须可区分：

| 返回类型 | 含义 | 前端行为 |
| --- | --- | --- |
| `opened` | 文件、文件夹、URL 成功打开并创建待记录 | 显示待记录提示条 |
| `record_only` | 仅记录资料，不打开外部资源 | 无 pending 时直接打开出关记录弹窗 |
| `pending_conflict` | 已有待记录会话 | 弹出三选一冲突处理 |
| `blocked` | 安全规则阻断或强提示待确认 | 硬阻断显示阻断原因；强提示展示“打开一次” |
| `open_failed` | 打开失败 | 显示失败原因，不进入待记录 |

返回优先级：

1. 如果已存在待记录会话，`continue_resource(input)` 必须优先返回 `pending_conflict`，包括目标资料是 `record_only` 的情况。
2. 只有不存在待记录会话时，`record_only` 资料才返回 `record_only`。
3. 文件、文件夹、URL 只有在安全校验和系统打开成功后才返回 `opened`。

风险确认：

1. 硬阻断没有确认入口，只能修改路径或改为仅记录。
2. 强提示必须由后端返回一次性 `risk_confirm_token`；用户点击“打开一次”后，前端用同一 `resource_id` 和该 token 再次调用 `continue_resource(input)`。
3. token 必须绑定资源、路径/URL 摘要、风险类型和短时有效期；路径或 URL 改变后失效。
4. “打开一次”不记忆偏好，不写入白名单。

### 10.5 统一错误模型

后端不得把数据库原始错误、系统错误或路径细节直接暴露给前端。

错误结构：

```text
AppError {
  code: string,
  user_message: string,
  recoverable: boolean,
  details?: object
}
```

必须覆盖的错误码：

| 错误码 | 场景 | 推荐恢复动作 |
| --- | --- | --- |
| `VALIDATION_FAILED` | 表单或 DTO 校验失败 | 修正字段 |
| `NOT_FOUND` | 项目、资料、待记录不存在 | 返回首页或刷新 |
| `PENDING_SESSION_CONFLICT` | 已存在待记录会话 | 记录上一次 / 放弃上一次 / 取消 |
| `UNSUPPORTED_URL_SCHEME` | URL 协议不支持 | 修改链接 |
| `INVALID_URL` | URL 解析失败或无 host | 修改链接 |
| `PATH_NOT_FOUND` | 文件或文件夹不存在 | 修改路径 / 仅记录 |
| `PATH_BLOCKED` | 风险路径被阻断 | 修改路径 / 仅记录 |
| `PERMISSION_DENIED` | 权限不足 | 修改路径 / 仅记录 |
| `DB_CONSTRAINT_FAILED` | 数据库约束失败 | 保留输入并提示重试 |
| `MIGRATION_FAILED` | 迁移失败 | 停止启动并提示检查数据 |
| `TRANSACTION_FAILED` | 事务失败 | 回滚并保留输入 |

### 10.6 隐私与本地数据

本工具虽然是本地工具，但数据库会保存资料标题、路径、URL、学习进度、下一步任务和时间戳。这些信息可能暴露学习兴趣、工作内容、客户名称或用户名目录。

隐私要求：

1. MVP 不上传任何学习数据。
2. MVP 不做遥测。
3. MVP 不自动上传崩溃日志。
4. MVP 不读取学习文件内容。
5. MVP 不抓取网页内容、favicon 或链接预览。
6. MVP 不自动请求用户填写的 URL。
7. 错误日志默认不记录完整本地路径和完整 URL。
8. 页面展示路径时默认折叠中间段，只显示文件名、父目录或域名。
9. 删除资料时必须说明：历史学习记录会保留资料名称快照。
10. 未来如增加导出、备份或 debug 包，必须提示其中可能包含敏感学习信息。

### 10.7 路径和 URL 细化规则

路径规则：

1. 保存和打开前必须规范化路径。
2. 禁止相对路径和包含未解析 `..` 的路径进入打开流程。
3. 禁止设备路径，例如 `\\?\`、`\\.\`。
4. 禁止 UNC 和网络路径。
5. 禁止 Windows Alternate Data Streams，例如 `file.txt:stream`。
6. 禁止快捷方式 `.lnk` 和 Internet Shortcut `.url`。
7. 符号链接、junction、reparse point 必须解析最终目标；若最终目标是网络位置或风险类型，必须阻断。
8. 映射网络盘按网络路径处理。
9. 打开前必须在尽可能靠近系统打开调用的位置重新校验一次最终目标。
10. 校验后到系统打开之间如果目标被本机其他进程替换，MVP 接受该低概率 TOCTOU 风险；不为此引入后台监控或文件锁。

URL 规则：

1. 必须能被 URL parser 正确解析。
2. 只允许 `http` 和 `https`。
3. 必须有 host。
4. 禁止 userinfo，例如 `https://user:pass@example.com`。
5. 禁止控制字符、换行和空白混淆。
6. `localhost`、`127.0.0.1`、内网 IP、短链接不硬阻断，但必须强提示。
7. 页面展示 URL 时优先展示解析后的 hostname。
8. 保存资料时允许保存 `localhost`、内网 IP、短链接等强提示 URL；强提示只发生在打开阶段。

### 10.8 外部打开风险分级

| 风险级别 | 类型 | 行为 |
| --- | --- | --- |
| 硬阻断 | 可执行文件、脚本、快捷方式、安装包、注册表、设备路径、网络路径、自定义协议、`file:`、`javascript:`、`data:` | 只提供“修改路径 / 仅记录进度” |
| 强提示 | 宏文档、压缩包、未知扩展、localhost、内网 URL、短链接 | 默认推荐“仅记录进度”，允许用户确认打开一次 |
| 普通打开 | 常规本地文档、常规文件夹、普通 http/https URL | 调用系统默认应用或浏览器打开 |

硬阻断文案：

```text
已阻断此资源打开
原因：{risk_reason}
为保护本地文件安全，MVP 不打开可执行文件、脚本、快捷方式或网络路径。

[仅记录进度] [修改路径]
```

强提示文案：

```text
即将打开外部资源
资料：{title}
目标：{normalized_path_or_hostname}

风险：{reason}
本工具只会调用系统打开，不会读取文件内容；打开后的安全性取决于系统默认应用或网站。

[仅记录进度] [修改路径] [打开一次]
```

“打开一次”只对本次操作有效，不记忆到资料或全局设置；下次再次打开同一风险资源仍需提示。

强提示弹窗中点击“打开一次”后，必须禁用该按钮直到后端返回结果，避免连点创建重复打开请求。

### 10.9 备份与恢复边界

MVP 不做用户可见的完整导入导出，但必须为迁移和数据损坏准备最低限度恢复策略。

要求：

1. 每次数据库迁移前，自动创建本地备份副本。
2. 备份文件必须与主数据库位于同一应用数据目录下。
3. 默认最多保留最近 3 个迁移备份。
4. 启动成功且数据库完整性检查通过后，如果距离上一次常规备份超过 7 天，自动创建一次轻量备份。
5. 常规备份默认最多保留最近 4 个。
6. 迁移成功后不立即删除最新备份。
7. 迁移失败时不得继续启动到半升级数据库。
8. 数据库损坏时，应用进入只读诊断状态，提示用户存在本地备份。
9. 备份文件与主数据库一样包含敏感学习信息，不能自动上传。
10. 如果未来提供“清空数据”功能，必须同时说明是否删除备份。

只读诊断页必须显示：

1. 主数据库路径。
2. 最近可用备份列表、创建时间、对应迁移版本。
3. 操作入口：从备份恢复、打开数据目录、退出应用。
4. 敏感数据提示：备份与日志可能包含资料标题、路径、URL 和学习进度。

恢复流程：

1. 用户选择备份后，应用先关闭当前数据库连接。
2. 将当前损坏 DB 重命名为 `cultivation_system.corrupt.{timestamp}.sqlite3`。
3. 复制所选备份为新的 `cultivation_system.sqlite3`。
4. 重新启动数据库初始化流程；如备份版本较旧，按 §9.11 再次迁移。
5. 如果恢复后的同一迁移再次失败，停在只读诊断页，不循环重试。

## 11. MVP 验收标准

验收分为三层：

| 层级 | 含义 | 用途 |
| --- | --- | --- |
| P0 核心闭环验收 | 验证学习进度记录是否成立 | 每次 P0 迭代必须通过 |
| P1 发布阻断验收 | 验证本地桌面发布质量 | MVP 对外使用前必须通过 |
| P2 体验观察指标 | 验证主题、效率和满意度 | 不阻断 P0 开发 |

### 11.1 P0 核心闭环验收

P0 通过标准：

| 顺序 | 验收项 | 通过标准 |
| --- | --- | --- |
| 1 | 创建项目 | 空库启动后能创建一个项目，项目名称 trim 后非空 |
| 2 | 添加资料 | 能给项目添加一个资料，默认进度 0 |
| 3 | 记录学到哪里 | 能保存资料当前进度描述 |
| 4 | 记录百分比 | 能保存 0 到 100 的整数百分比 |
| 5 | 记录下一步 | 能保存下次继续做什么 |
| 6 | 首页推荐 | 首页显示一个可继续资料，并展示资料名、进度描述、百分比、下一步 |
| 7 | 出关记录 | 能通过 3 个核心字段保存一次学习记录 |
| 8 | 自动快照 | 保存出关记录后生成一条 `study_logs` |
| 9 | 进度同步 | 首页、项目详情、资料详情显示一致 |
| 10 | 关闭重开恢复 | 重启后仍能看到上次资料、进度和下一步 |

### 11.2 发布阻断级验收

以下全部通过，MVP 才达到发布级质量。P0 开发不要求一次性全部完成，但每个切片完成时必须通过对应验收。

| 顺序 | 验收项 | 通过标准 |
| --- | --- | --- |
| 1 | 首次启动空状态 | 首页显示创建第一个项目入口 |
| 2 | 创建项目 | 项目出现在首页，默认未启修，进度 0% |
| 3 | 项目名称边界 | 空字符串、纯空格不能保存；失败后保留输入；合法名称 trim 后保存；允许同名项目 |
| 4 | 添加资料 | 资料归属项目，默认进度 0% |
| 5 | 资料标题边界 | 标题空或纯空格不能保存；资料必须关联有效项目；允许同名资料；不能产生孤儿资料 |
| 6 | 进度校验 | 只接受 0 到 100 的整数；空值、纯空格、-1、101、35.5、非数字均不能保存 |
| 7 | 进度一致展示 | 首页、项目详情、资料详情显示一致 |
| 8 | 状态与进度联动 | completed 强制 100%，not_started 强制 0；100 自动 completed，0 自动 not_started；从终态改回非终态必须提交 1 到 99 的进度 |
| 9 | 项目进度计算 | 20、40、100 三个资料时项目进度为 53%；无资料时为 0%；删除资料后立即重算 |
| 10 | 下一步任务 | 首页和详情页同步展示，清空后显示默认提示 |
| 11 | 资料元信息编辑 | 资料创建后不能更换所属项目；切换打开方式时按目标类型重校验路径，切到 record_only 时清空路径 |
| 12 | 首页推荐 | 只推荐 learning、review、not_started；paused/completed 永不推荐；last_studied_at 为空按 NULLS LAST；相同时间按固定顺序排序 |
| 13 | 继续闭关：文件 | 打开有效本地文件；只更新 last_opened_at；不更新 last_studied_at；进入待记录状态 |
| 14 | 继续闭关：文件夹 | 打开有效本地文件夹；只更新 last_opened_at；不更新 last_studied_at；进入待记录状态 |
| 15 | 继续闭关：网页 | 打开有效 http/https 链接；只更新 last_opened_at；不更新 last_studied_at；进入待记录状态 |
| 16 | 仅记录资料 | 无 pending 时不调用系统打开，直接进入出关记录；有 pending 时先进入冲突处理；取消后不创建日志、不更新出关时间、不产生待记录残留 |
| 17 | 待记录唯一性 | 已有待记录会话时，任何继续闭关含 record_only 都必须提示先记录、放弃或取消，不能静默覆盖 |
| 18 | 待记录恢复 | 成功打开资料后关闭应用再打开，首页仍显示待记录提示条 |
| 19 | 待记录清除 | 保存出关记录或放弃记录后，提示条消失；重启后不会恢复旧待记录 |
| 20 | 出关记录保存 | 日志、资料进度、资料状态、项目最近出关时间、项目进度、待记录清除必须同时成功；手动补记不得清除既有 pending |
| 21 | 出关记录回滚 | 模拟写入失败时不创建日志，不更新资料/项目，不清除待记录，用户输入仍保留 |
| 22 | 放弃记录 | 不生成日志，不更新进度、下一步任务和最近出关时间 |
| 23 | 学习时长边界 | 闭关时长为空或 0 可保存；1 到 1440 可保存；负数、1441、非数字不能保存 |
| 24 | 无变化学习记录 | 出关记录字段均无变化时仍可保存，生成一条 `study_logs`，反馈文案为“已记录本次学习” |
| 25 | 删除资料 | 删除前二次确认；有 pending 时额外提示会放弃未保存记录；删除后资料不存在，项目进度重算，历史日志保留并置空 resource_id，资料名称快照仍可读 |
| 26 | 删除项目 | 删除前二次确认；确认后项目、资料、相关日志、待记录会话全部删除，无孤儿数据 |
| 27 | 持久化 | 关闭重开后项目、资料、日志、推荐资料、待记录状态、last_opened_at、last_studied_at 均恢复正确 |
| 28 | 打开失败分类 | 路径不存在、URL 无效、权限不足分别提示原因；失败时不更新 last_opened_at，不进入待记录 |
| 29 | URL 安全 | 只允许 http/https；file、javascript、data、mailto、自定义协议不能保存或打开 |
| 30 | 风险路径提示 | 可执行文件、脚本、快捷方式、网络路径打开前必须提示或阻断；“打开一次”不记忆且点击后禁用防连点；用户取消时不进入待记录 |
| 31 | 本地离线可用 | 断网状态下创建项目、添加资料、记录进度、查看历史、关闭重开恢复均可用 |
| 32 | 修仙主题可读 | 关键按钮同时表达真实动作；首页第一屏可见真实资料名、百分比、下一步任务 |
| 33 | 成就反馈 | 进度提升、进度回退、无变化保存和 100% 完成时显示对应反馈，不阻塞用户继续操作 |
| 34 | 数据库约束 | 空白名称、非法枚举、非法进度、负时长、孤儿 project_id/resource_id 写入失败 |
| 35 | 迁移可靠性 | 空库初始化成功；旧版本升级成功；故意失败的迁移完整回滚 |

### 11.3 成功指标

MVP 成功标准：

1. 首次使用时，用户能在 1 分钟内创建一个项目和一个资料。
2. 学习结束后，用户能在 30 秒内完成一次出关记录。
3. 重新打开工具后，用户能在 5 秒内判断“我下次该继续学什么”。
4. 每个资料都能稳定展示三类核心进度信息：当前学到哪里、完成百分比、下一步任务。
5. 项目进度只能由资料进度自动计算，不能被手动编辑。
6. 关闭重开后，首页推荐资料、资料进度、下一步任务、待记录状态都能恢复。
7. 若准备对外发布，80% 以上试用用户能在不解释修仙设定的情况下理解核心按钮和进度含义；自用 MVP 不以此阻断开发。
8. MVP 阻断级验收未全部通过前，不开发搜索、统计、AI、徽章墙、导入导出、独立日志管理页。

自用阶段替代验收：

1. 单人自测计时：按固定脚本创建项目、添加资料、保存出关记录各跑 5 次。
2. 出关记录从打开弹窗到保存成功 Toast 的 5 次中位数 <= 30 秒。
3. 冷启动后从数据加载完成到判断下一步学习对象 <= 5 秒。
4. 可用性 N=5 和 80% 理解率只作为对外发布前观察项。

### 11.7 本地可观测性

MVP 应保留本地结构化日志，便于排查用户机器上的持久化和打开问题。

日志事件：

- `app_start`
- `db_init`
- `migration_start`
- `migration_success`
- `migration_failure`
- `clock_skew_detected`
- `command_start`
- `command_end`
- `continue_resource_result`
- `save_study_log_commit`
- `save_study_log_rollback`
- `pending_created`
- `pending_cleared`

日志隐私规则：

1. 默认不记录完整路径。
2. 默认不记录完整 URL。
3. 路径只记录类型、扩展名和风险分类。
4. URL 只记录协议和 hostname。
5. debug 导出必须提示可能包含敏感学习信息。

日志落地规则：

1. 日志文件位于应用数据目录的 `logs/` 子目录。
2. 当前日志文件名为 `app.log`，按大小滚动。
3. 单个日志文件最大 2 MB，最多保留 5 个滚动文件。
4. 默认保留最近 30 天日志；超过保留期可在启动时清理。
5. 日志与备份一样视为敏感本地数据，不能自动上传。

## 12. MVP 开发切片

MVP 建议按以下顺序开发，确保每一阶段都有可验证结果。

### 12.1 M0：本地数据骨架

目标：先让数据可信。

范围：

- 初始化本地应用。
- 建立 SQLite 数据库。
- 建立 `projects`、`resources`、`study_logs`、`pending_study_sessions`。
- 开启 SQLite 外键。
- 加入基础枚举和进度范围约束。

完成标准：

- 应用启动时能创建数据库。
- 数据库关闭重开后数据仍存在。
- 非法进度、空项目名、空资料标题不能写入。

### 12.2 M1：项目和资料基础管理

目标：能创建修炼方向和功法资料。

范围：

- 首页空状态。
- 创建项目。
- 编辑项目。
- 删除项目。
- 添加资料。
- 编辑资料。
- 删除资料。
- 项目详情页。
- 资料详情页。

完成标准：

- 用户能创建一个项目和一个资料。
- 资料必须归属项目。
- 项目详情能显示资料列表。
- 资料详情能显示进度字段。

### 12.3 M2：学习进度展示与推荐

目标：让学习进度成为可展示、可计算、可推荐的数据；创建后的进度变更仍留到 M3 的出关记录实现。

范围：

- 资料进度描述。
- 资料进度百分比。
- 下一步任务。
- 项目进度自动计算。
- 首页继续闭关卡片。

完成标准：

- 用创建资料的初始进度或测试种子数据验证首页、项目详情、资料详情同步展示。
- 项目进度随资料进度数据变化。
- 首页能回答“下次继续学什么”。
- 编辑资料表单不提供进度和下一步输入。

### 12.4 M3：继续闭关与出关记录

目标：打通学习前和学习后的完整闭环。

范围：

- 继续闭关打开文件、文件夹、URL。
- 仅记录资料流程。
- 待记录状态。
- 出关记录弹窗。
- 保存学习记录。
- 最近参悟记录。
- 放弃本次记录。
- `last_opened_at` 和 `last_studied_at` 区分。

完成标准：

- 打开资料只更新最近打开时间。
- 保存出关记录才更新最近出关时间。
- 出关记录保存后更新资料进度、项目进度和学习记录。

### 12.5 M4：修仙主题与成就反馈

目标：让页面有修仙系统的成就感，但不牺牲效率。

范围：

- 修炼总览页面文案。
- 继续闭关卡片文案。
- 功法详情文案。
- 出关记录文案。
- 进度提升、进度回退、无变化保存反馈。
- 100% 参悟完成反馈。
- 主题文案统一。

完成标准：

- 修仙文案不遮挡真实进度。
- 进度提升、进度回退、无变化保存都有即时反馈。
- 资料 100% 时有完成反馈。
- 用户无需理解修仙设定也能完成记录。

### 12.6 M5：MVP 验收与打磨

目标：完成自用替代验收并打包冒烟；对外发布前再执行发布阻断级验收。

范围：

- 自用核心路径人工回归。
- 中文和空格路径回归。
- 持久化回归。
- 首页推荐规则回归。
- 打包冒烟测试。

完成标准：

- 自用替代验收全部通过。
- 无网络环境下核心功能可用。
- 关闭重开后能恢复首页继续学习状态。
- 若准备对外发布，再执行 `11.2 发布阻断级验收` 全量回归。

## 14. 总结

MVP 的本质不是做一个完整学习管理系统，而是先把一个最小闭环做通：

> 记录学习资料 -> 记录学习进度 -> 记录下一步 -> 首页展示 -> 继续闭关 -> 出关记录 -> 下次恢复。

只要这个闭环顺滑，修仙主题就会变成“学习有反馈”的助推器；如果这个闭环不顺，任何成就、主题和统计都会变成噪音。
