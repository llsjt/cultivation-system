# 项目文档版本管理规范

本目录用于管理版本化需求分析和实施方案。核心原则是：每个版本目录只保留一份需求分析和一份实施方案，旧版本归档只读，新需求进入新版本。

## 目录结构

```text
docs/
  版本索引.md
  versions/
    v1.4/
      需求分析.v1.4.md
      实施方案.v1.4.md
    v1.5/
      需求分析.v1.5.md
      实施方案.v1.5.md
    v1.6/
      需求分析.v1.6.md
      实施方案.v1.6.md（待补充）
```

## 版本规则

1. 每个版本目录只放同一版本的一份需求分析和一份实施方案。
2. 需求分析和实施方案必须使用同一个版本号，例如 `v1.5`。
3. 不再为同一版本额外保留 `需求文档`、`实施分析` 或 `版本说明` 副本。
4. 旧版本一旦进入“已归档”或“已完成”，不再回写修改。
5. 新增需求、调整范围、改变验收标准，都进入下一个版本。

## 文档职责

| 文档 | 职责 | 不应包含 |
| --- | --- | --- |
| 需求分析 | 当前版本的问题、范围判断、需求拆分建议和验收口径 | 具体代码实现步骤 |
| 实施方案 | 当前版本的任务拆分、技术方案、涉及模块、验证方法、发布与回滚 | 未被需求分析确认的新范围 |

## 推荐工作节奏

1. 先写需求分析，冻结“做什么”和“不做什么”。
2. 再写实施方案，拆解“怎么做”和“怎么验收”。
3. 开发时只按已确认的实施方案推进。
4. 发现新需求时，先记入开放问题；如果会改变范围，进入下一版本。
