# 实施方案 v1.7

版本：v1.7
状态：草稿
整理日期：2026-06-06
对应需求分析：`需求分析.v1.7.md`

## 1. 版本目标

v1.7 的目标是完成“架构与分层治理”：

1. 主进程补齐 repository 层，让 SQL 和 row 映射从 service 中迁出。
2. IPC/DTO 契约补齐输入、事件和关键输出校验。
3. `App.tsx` 收敛为应用壳、tab、全局弹窗和全局反馈宿主。
4. 前端 feature 中的数据请求、状态、展示 UI 和 viewModel/helper 分开。
5. 当前学习驾驶舱的跨 feature 编排进入中性 workbench 层。
6. 资源管理面板和全局资料库拆成可测试的小切片。
7. 弹窗壳、日期格式化和样式模式沉淀为通用能力。

## 2. 不做事项

- 不新增商城、抽卡、战力、排行榜、账号、云同步、在线分享。
- 不改变 v1.6 已有用户路径和文案语义。
- 不改变 pending 单会话规则。
- 不做破坏性数据库迁移。
- 不为 v1.7 强行引入 React Query、zustand、react-hook-form 或 react-router-dom。
- 不做全量 CSS 重写。
- 不把业务组件塞入 `components/`。
- 不让 repository 承担业务规则。
- 不让展示组件直接访问 `window.api`。

### PLAN-v1.7-006 基于 Git 历史节点恢复 Drizzle snapshot

目标：在任何 schema 或 repository 深度调整前，修复 Drizzle snapshot 元数据缺失问题，避免后续 diff 漂移。

改动范围：

- `drizzle/meta/*`
- `docs/artifacts/v1.7/acceptance-summary.md`

执行步骤：

1. 执行 `rtk git status --short`。若存在与本项无关的未提交改动，先记录并暂停本项；不得在脏工作区内生成或复制 snapshot。
2. 检查 `drizzle/meta/_journal.json` 与 snapshot 文件数量。
3. 若 `_journal.json` 已登记 0001/0002 但缺少对应 snapshot，将本项标记为阻塞后续数据库结构优化的 Critical 前置项。
4. 不直接运行可能重复检测既有变更的 `db:generate` 作为修复手段。
5. 不使用 `drizzle-kit pull`、introspect 或从物理数据库逆向恢复 snapshot，避免丢失原始 Drizzle schema 中的 check 约束、默认值和表级约束表达。
6. 用 `rtk git log --oneline -- drizzle/0001*.sql drizzle/0002*.sql src/main/db/schema.ts` 定位 0001/0002 迁移生成时对应的提交。
7. 如果能定位逐迁移历史节点，为每个历史节点创建 detached 临时 worktree：

```powershell
rtk git worktree add --detach ..\CultivationSystem-drizzle-0001 <commit-for-0001>
rtk git worktree add --detach ..\CultivationSystem-drizzle-0002 <commit-for-0002>
```

8. 在临时 worktree 中运行 `rtk corepack pnpm install --frozen-lockfile`，仅当依赖缺失导致 Drizzle 生成流程无法运行时执行。
9. 在对应历史节点运行 Drizzle 生成流程，只用于得到当时 schema 对应的 snapshot 文件；不得把新生成的迁移 SQL 合并回当前工作区。
10. 如果 0001/0002 没有逐迁移历史节点，则使用兜底路径：
    - 创建一个 detached 临时 worktree 指向同时包含 0001/0002 的提交：`rtk git worktree add --detach ..\CultivationSystem-drizzle-recovery <commit-with-0001-and-0002>`。
    - 在临时 worktree 中复制 `src/main/db/schema.ts` 为 `schema.v1.7-meta-recovery.ts`，只在该副本上重建中间 schema。
    - `0001` 中间 schema 只包含 `0001_pending_close_time.sql` 的三列变更，不包含 `0002_realm_breakthrough.sql` 的 realm、cultivation、evidence 或 breakthrough 表变更。
    - `0002` schema 等于该提交中的完整 schema。
    - 分别用临时 schema 替换临时 worktree 的 `src/main/db/schema.ts` 运行 Drizzle 生成流程，生成后立即恢复临时 worktree 文件。
    - 兜底路径的所有临时 schema diff 必须写入验收摘要。
11. 将缺失的 `0001_snapshot.json`、`0002_snapshot.json` 从临时节点复制到当前 `drizzle/meta/`，并核对 `_journal.json` 的 tag/breakpoints 与 snapshot 文件名一致。
12. 复制 snapshot 后执行 `rtk git diff -- drizzle/meta`，确认只新增缺失 snapshot，不改 `_journal.json` 和迁移 SQL。
13. 删除临时 worktree：

```powershell
rtk git worktree remove ..\CultivationSystem-drizzle-0001
rtk git worktree remove ..\CultivationSystem-drizzle-0002
rtk git worktree remove ..\CultivationSystem-drizzle-recovery
```

只删除本次实际创建的 worktree 路径；删除失败必须先检查 `rtk git worktree list`，不得手动删除仍被 Git 记录的 worktree 目录。

14. 修复后再运行可控的 generate/diff 检查，确认不会重复生成 0001/0002 已有结构。
15. 如果无法生成可信 snapshot，删除本次新增 snapshot、保留验收摘要中的失败记录，并停止 M1；不得进入 M2、不得新增迁移。
16. 将历史提交、命令、临时 worktree 路径、兜底 schema diff、最终 `git diff -- drizzle/meta` 和 snapshot 文件清单写入验收摘要。

验收：

- 无非预期 schema diff。
- 迁移 SQL 不被手工改写。
- snapshot 缺失未解决前，不允许进入任何 schema 结构优化或新迁移生成。
- 验收记录能追溯每个补回 snapshot 对应的 Git 历史提交。
- 无逐迁移历史节点时，验收记录能追溯临时 schema 重建过程；失败时新增 snapshot 已回滚。

### PLAN-v1.7-007 建立 repository 目录和基础类型

目标：创建 repository 层骨架，不改变业务行为。

新增文件：

- `src/main/repositories/projectRepository.ts`
- `src/main/repositories/resourceRepository.ts`
- `src/main/repositories/studyLogRepository.ts`
- `src/main/repositories/pendingSessionRepository.ts`
- `src/main/repositories/breakthroughAttemptRepository.ts`
- `src/main/services/cultivationMappers.ts`
- `src/main/repositories/types.ts`

`types.ts` 固定约束：

```ts
import type { Database, Statement } from 'better-sqlite3';

export type RepositoryDatabase = Database;

export type PreparedStatement = Statement;
```

repository 构造约束：

```ts
export class ProjectRepository {
  private readonly listWithProgressStatement: PreparedStatement;

  constructor(private readonly db: RepositoryDatabase) {
    this.listWithProgressStatement = db.prepare('...');
  }
}
```

固定规则：

- repository 实例化入参只能是 `Database`。
- 所有 `prepare` 动作在 repository 构造函数中完成，动态 SQL 例外必须在代码注释中说明原因。
- repository 方法必须同步返回，不使用 `async`、`Promise`、timer、opener、IPC、logger 或 React/DOM API。
- repository 不调用 `db.transaction()`，也不自行创建 unit of work。
- repository 不依赖 `shared/errors`，不抛 `AppError`，不决定 `NOT_FOUND`、冲突、突破是否成功等业务语义。
- repository 返回 row / projection / count / null；IPC DTO 组装保留在 service 或 service 调用的纯 mapper 中。

执行步骤：

1. 将 `ProjectRow`、`ResourceRow`、`StudyLogRow`、`PendingRow`、`BreakthroughAttemptRow` 移入 repository 层类型。
2. row 类型保持 snake_case 字段，对应 SQL 返回。
3. row -> DTO 映射抽到 `src/main/services/cultivationMappers.ts`；不得放入 repository。
4. service 内部由同一个 `db` 初始化 repositories，避免外部注入不同数据库实例。
5. repository 暴露函数签名先固定，再迁移 service 调用。

首批 repository 函数签名：


## 13. 风险与缓解

| 风险 | 表现 | 缓解 |
| --- | --- | --- |
| repository 抽离改变事务语义 | 保存日志后资源或 pending 状态不一致 | service 保留事务边界，重点测 saveStudyLog/continue/delete |
| repository 内出现隐式事务或异步 | better-sqlite3 事务闭包不再覆盖完整业务写入 | repository 构造只接收 Database，prepare 在构造函数，方法同步无状态，`rg "transaction\\(|async "` 检查 |
| App hook 抽离丢失状态 | toast、confirm、modal、overlay 不显示 | 先抽 feedback，再抽数据，再抽 action，逐步测试 |
| workbench 层继续 prop drilling 或高频重渲染 | `CurrentStudyWorkbench` 接收 20+ props，或 toast/busy 变化刷新整棵驾驶舱 | 使用 `WorkbenchDataContext`、`WorkbenchActionsContext`、`AppFeedbackContext` 三轨上下文，context 不暴露任意 setter |
| 展示组件仍偷偷 IPC | 分层治理形同虚设 | `window.api` allowlist 验收，只允许指定 hook/action 文件 |
| output schema 捕获历史字段差异 | 正常路径被误报 validation | 开发/测试拦截，生产只 warning 并返回原始数据；先覆盖高价值命令 |
| 迁移元数据修复误改 SQL | 数据迁移风险扩大 | 不手改迁移 SQL，不从物理数据库 pull；只从对应 Git 历史节点生成并合并缺失 snapshot，未恢复前不生成新迁移 |
| 资源详情旧请求覆盖新资源 | 快速切换资源后详情显示错位 | `useResourceDetail` 固定使用 active 取消标记 |
| 全局资料库 IPC 拥堵 | App 启动或保存后遍历所有项目请求详情 | 主进程提供 `get_global_resources` JOIN 聚合，前端 hook 只在 library tab 或主动刷新时发起一次 IPC |
| 新建资源后选中态丢失 | 添加成功后列表不高亮新资料 | `submitResource` 必须返回新资源，`refresh` 后按返回的 id 执行 `selectResource` |
| acceptance 脚本不可执行 | 文档声称验收但 Electron 脚本无法运行 | M6 必须运行或记录具体阻塞和手工替代范围 |
| 样式拆分造成视觉回归 | 驾驶舱布局错位 | 复用 v1.6 UI 验收脚本和截图 |
