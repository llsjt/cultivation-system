# 需求分析 v1.7

版本：v1.7
状态：草稿
整理日期：2026-06-06
前置版本：`../v1.6/需求分析.v1.6.md`
对应实施方案：`实施方案.v1.7.md`

## 分析结论

v1.7 的核心目标是把 v1.6 已形成的“修行驾驶舱”稳定下来，完成一次以架构分层和前端 UI 边界为主的治理版本。

v1.6 已经把产品重心从零散资料管理推进到主工作台、突破诊断、资源定位和有效学习时间反馈。接下来不应继续追加新的玩法或大范围视觉系统，而应处理当前代码结构中的长期维护风险：

1. 主进程存在 `service -> repository -> db` 约定与实现不一致。
2. IPC/DTO 边界已有输入校验基础，但 `select_local_file`、事件 payload、输出 schema 和部分输入 `.strict()` 仍不完整。
3. `App.tsx` 承担应用壳、数据加载、表单草稿、业务动作、弹窗、背景装饰和 tab 编排，已经成为根级巨型容器。
4. 前端部分 feature 组件直接调用 `window.api`，展示 UI、请求副作用、筛选状态和业务动作混在一起。
5. `projects`、`resources`、`studyLogs` feature 之间存在组合层穿透，缺少中性的工作台编排层。
6. 通用 UI 壳、日期格式化、资源展示文案和样式规则仍有重复与漂移。

因此，v1.7 的产品重点是“不改变用户学习闭环，只让现有体验更稳”。工程重点是“补齐分层、收敛边界、拆小组件、保留行为”。所有重构都必须以现有功能不回归为验收前提，尤其是继续学习、pending 出关、日志保存、突破尝试、资源筛选和弹窗确认。

## v1.6 交接假设

v1.7 以 v1.6 的驾驶舱、资源权重、突破诊断和有效学习时间能力已经进入代码为前提，但不默认 v1.6 的全部验收资料已经归档完成。

v1.7 承接时必须遵守：

- v1.6 的用户可见行为作为回归基线，不在 v1.7 中改变产品语义。
- 若发现 v1.6 的功能缺陷，优先登记为回归修复；只有影响分层治理的缺陷才纳入 v1.7 P0。
- v1.7 不新增数据库业务字段，除非 repository 抽离暴露出必须修正的 schema/迁移元数据问题。
- v1.7 不引入云同步、账号系统、在线分享、商城、战力、抽卡、排行榜或更重游戏化系统。
- v1.7 不为了“完成分层”一次性重写整个前端或服务层；所有改动按可验证切片推进。

## 范围原则

- 保持本地优先 Electron 桌面应用边界。
- 保持 renderer 只通过 preload 暴露的 `window.api` 访问主进程。
- `shared/` 继续只放纯逻辑、DTO、枚举、错误和可双端复用计算。
- 主进程按 `service -> repository -> db` 收敛，service 负责业务流程，repository 负责 SQL 和 row 映射。
- 前端按 `App shell -> workspace/workbench 编排 -> feature container hook -> presentational UI -> viewModel/helper` 收敛。
- `components/` 只放无业务语义的通用 UI；业务文案、业务表单和业务卡片仍留在 `features/`。
- 优先小步迁移现有代码，不引入新的全局状态库、表单库或请求库作为 v1.7 前置条件。
- 所有用户可见文案继续使用中文，修仙主题服务真实操作含义。

## P0 需求分析

### RA17-01 主进程 repository 层缺失

问题：项目约定调用链为 `service -> repository -> db`，但当前 `src/main/services/cultivationService.ts` 直接持有 `Database`，SQL 查询、写入、事务、row 映射和业务编排集中在一个 800 行以上的 service 中。

影响：新增业务时容易把查询细节继续堆进 service；测试只能围绕大服务做集成式覆盖；SQL、row 类型、DTO 映射和业务规则难以分别定位。

需求判断：v1.7 必须补齐最小 repository 层。repository 不承载业务决策，只负责持久化读写、row 映射和查询复用。service 保留业务流程、事务边界、AppError、领域规则和 opener 调度。

事务边界必须明确：repository 只接收 `Database` 实例，构造函数中完成 statement prepare，方法保持纯同步、无状态、无异步副作用、不开启事务。service 是唯一事务控制层，继续通过 `db.transaction()` 包裹一个业务动作内的多 repository 写入。

最小切片：

- `ProjectRepository`
- `ResourceRepository`
- `StudyLogRepository`
- `PendingSessionRepository`
- `BreakthroughAttemptRepository`

repository 的返回边界固定为 row / projection / count / null。`ProjectSummary`、`ResourceSummary`、`StudyLogView`、`PendingSessionView`
等 IPC DTO 由 service 调用 `src/main/services/cultivationMappers.ts` 中的纯 mapper 组装，repository 不抛 `AppError`，也不依赖 `shared/errors`。
`assertChanged`、`NOT_FOUND`、冲突判断、突破是否成功等业务语义全部保留在 service。

验收口径：

- service 不再直接散布核心 CRUD SQL。
- service 不再直接散布 `breakthrough_attempts` 写入 SQL 或突破后 `projects.realm_*` 更新 SQL。
- 事务只能由 service 或明确的 service 级 unit of work 边界控制。
- repository 构造函数只接收 `Database`，不接收 service、logger、opener 或 React/renderer 相关对象。
- repository 方法不 `async`，不调用 opener，不调用 `db.transaction()`。
- repository 不返回 IPC DTO，不依赖 `shared/errors`，不决定 AppError 语义。
- 现有 service 测试不因抽层减少覆盖。
- repository 只在 `main/` 内使用，不进入 renderer 或 shared。

### RA17-02 IPC/DTO 边界不完整

问题：大部分 IPC 命令已经通过 zod input schema 校验并统一返回 `IpcResult<T>`，但仍有边界缺口：`select_local_file` 直接 cast 输入；pending 事件 payload 在 preload 中直接断言；部分 input schema 没有 `.strict()`；输出 schema 已定义但 router 没有使用。

影响：IPC 契约无法完全由 DTO 保证。新增命令或事件时容易出现 main、preload、renderer 三处类型漂移。

需求判断：v1.7 需要建立统一的 IPC contract 规则。所有命令输入必须有 schema，事件 payload 也必须有 schema；输出 schema 先覆盖高价值命令，不强制一次性校验全部输出。

输出校验必须按运行环境降级：`process.env.NODE_ENV === 'test' || !app.isPackaged` 时拦截并返回
`IPC_CONTRACT_FAILED`；生产环境如果 output schema 校验失败，只记录 warning 级别日志并返回原始 service 数据，
避免校验代码自身错误切断用户主链路。`IPC_CONTRACT_FAILED` 是 v1.7 新增错误码，中文文案固定为
`内部数据契约异常，请重启应用后再试。`，`recoverable` 固定为 `false`。

`register` 必须改为基于 zod schema 推导 handler 入参类型的泛型注册函数。v1.7 完成后，
`src/main/ipc/router.ts` 中不得继续出现 `input as` 或 `as never` 这类 handler 入参强转。

验收口径：

- `select_local_file` 有输入/输出 DTO 类型和 zod schema。
- `select_local_file` 通过通用 `register` 注册，不再在 `router.ts` 中独立裸写 `ipcMain.handle`。
- pending session closed event 有 payload schema，并在 preload 边界解析。
- `pageInputSchema`、`GetProjectDetailInputSchema` 等对象输入拒绝多余字段。
- `register` 支持可选 output schema，至少覆盖 `get_home_overview`、`get_project_detail`、`save_study_log`、`get_project_cultivation`。
- router handler 入参类型由 schema 推导，`input as` / `as never` 强转被移除。
- 生产环境 output 校验失败不会阻断正常返回。

### RA17-03 `App.tsx` 需要退回应用壳角色

问题：`App.tsx` 当前同时管理 overview、enums、project detail、cultivation、选中项目、新建资料表单、编辑弹窗、出关草稿、confirm、toast、pending 自动弹窗、继续学习、删除、突破、背景 SVG 和 tab 布局。

影响：任何 UI 调整都容易碰到核心业务动作；新增一个弹窗或按钮会扩大 `App.tsx` 的状态和 props；测试也更难聚焦。

需求判断：v1.7 应把 `App.tsx` 收敛为应用 shell 和全局挂载点。`App.tsx` 不再拥有业务数据、草稿、弹窗草稿或
overlay 状态；这些状态全部进入 `WorkbenchProvider` 内部。`App.tsx` 只负责挂载 provider 和最外层 shell。
全局 toast/confirm/modal/overlay 由 provider 内部的 host 组件渲染，host 读取 provider 管理的状态，避免“动作在 Provider、
弹窗状态在 App”的断点。

为了避免把 `App.tsx` 拆成一堆互相传参的 hook 后形成新的 prop drilling，v1.7 必须在 `src/renderer/features/workbench` 建立中性 `WorkbenchProvider`。Provider 内部组合 bootstrap、selected project data、actions 和 feedback，但对外拆成三轨 Context：`WorkbenchDataContext` 承载低频业务数据，`WorkbenchActionsContext` 承载稳定动作 API，`AppFeedbackContext` 承载 busy、toast、confirm 等高频反馈状态。子组件只订阅自己需要的一轨，避免 toast 或 busy 更新导致整棵驾驶舱树重渲染。

验收口径：

- `App.tsx` 不再直接包含大段背景 SVG 工厂和所有业务动作实现。
- `WorkbenchProvider` 组合 bootstrap、选中项目数据、toast/confirm、核心 action 编排，并分别暴露 data/actions/feedback 三轨 Context。
- 新建项目草稿、新建资源草稿、日志草稿、编辑项目、编辑资源、资源详情弹窗、pending 冲突、突破 overlay 的状态归属在实施方案中逐项写明。
- `CurrentStudyWorkbench` 不需要接收 20 个以上业务 props。
- `App.tsx` 仍是应用最外层 shell 的宿主，但不持有全局弹窗、toast 或 overlay 的业务状态。
- 拆分后继续学习、pending、保存日志、删除确认、突破 overlay 行为不变。

### RA17-04 前端数据请求必须从展示 UI 中分离

问题：除 `App.tsx` 外，`ResourceManagementPanel` 和 `GlobalLibrary` 也直接调用 `window.api`，同时管理 loading、error、筛选、列表、详情和 UI 渲染。

影响：展示组件难以复用和测试；错误处理、unwrap、retry、loading 空态分散；同一资源详情可能被多个 UI 路径重复拉取。

需求判断：v1.7 应建立 feature container hook。展示组件只接收 viewModel、状态和 callbacks，不直接访问 IPC。所有资源详情类异步加载必须保留竞态取消机制，避免频繁切换资源时旧请求覆盖新资源详情。

验收口径：

- `ResourceManagementPanel` 的 `get_resource_detail` 请求迁入 `useResourceDetail`。
- `GlobalLibrary` 的全局资源聚合迁入 `useGlobalLibraryResources`。
- `useResourceDetail` 必须使用 active 标记处理竞态取消。
- 展示子组件不出现 `window.api`。
- `IpcResult.ok` 裸读只保留确有业务分支的冲突处理，其余统一 `unwrap/unwrapResult`。

### RA17-05 前端 feature 编排层需要独立

问题：`features/projects/CurrentStudyCockpit.tsx` 直接组合 `studyLogs/PendingStrip` 和 `resources/ResourceManagementPanel`，并用 `ComponentProps<typeof ResourceManagementPanel>` 透传大量 props。项目 feature 被迫知道资源管理和学习日志的内部结构。

影响：feature 边界不清。后续要改资源面板 props 时，项目 cockpit 也会被迫变更；跨 feature 的 UI 编排没有稳定位置。

需求判断：v1.7 应新增中性编排层 `features/workbench`。项目 feature 只保留项目展示、诊断 viewModel 和项目相关面板；资源 feature 只保留资源管理；学习日志 feature 只保留 pending 和日志表单。workbench 编排层通过 `WorkbenchDataContext`、`WorkbenchActionsContext` 和 `AppFeedbackContext` 获取必要切片，避免把跨 feature 状态继续层层透传，也避免高频反馈状态拖慢数据展示树。

验收口径：

- `projects` feature 不再直接 import `ResourceManagementPanel`。
- 跨 feature 编排位于 `features/workbench`。
- `ComponentProps<typeof ResourceManagementPanel>` 这类整包透传被显式 props 契约替代。
- workbench 子组件通过 context 读取工作台状态时，只暴露业务必要切片，不暴露任意 setter。
- 当前驾驶舱布局和行为不回归。

### RA17-10 Drizzle snapshot 元数据缺失需要前置修复

问题：迁移 SQL 已有 `0001`、`0002`，`drizzle/meta/_journal.json` 也已登记 0000、0001、0002 三个迁移条目，但 `drizzle/meta/` 中只看到 `0000_snapshot.json`。这会导致 Drizzle Kit 在未来 schema diff 生成时失去对比基准，可能把既有迁移误判为新增表或新增字段。

需求判断：这是 v1.7 的 M1 前置门禁，优先级升为 P0。若确认为缺失，必须放弃从物理数据库逆向 `pull/introspect`
恢复，改用“基于 Git 历史节点的快照生成法”：定位 `0001`、`0002` 迁移对应的历史提交，在临时 worktree 或 checkout
节点上用当时的 schema 生成对应 snapshot，再只合并缺失的 meta snapshot 文件。若 Git 历史不存在逐迁移 schema 节点，
必须在临时 worktree 中按 `0001`、`0002` 的 SQL 变更分别重建中间 schema 状态并生成 snapshot；该兜底路径仍然不得改写
当前工作区的迁移 SQL，不得从物理数据库逆向恢复。这样才能最大限度保留原始 schema 中的 check 约束、默认值和 Drizzle
元数据形态。不手改已生成迁移 SQL，也不直接用可能重复检测既有变更的 `db:generate` 硬冲当前工作区。

验收口径：

- `drizzle/meta/_journal.json` 与 snapshot 文件状态一致，且缺失 snapshot 来自对应 Git 历史节点生成结果。
- 若没有逐迁移历史节点，验收记录必须写明临时 worktree、重建的中间 schema 变更清单、生成命令和 diff 校验结果。
- 后续 `pnpm db:generate` 前后不会生成非预期 schema diff。
- 迁移 SQL 不被手工改写。
- 迁移元数据状态写入验收记录。

## P1 需求分析

### RA17-06 资源管理面板需要拆成容器与展示切片

问题：`ResourceManagementPanel` 同时承担资源主列表、搜索筛选、新建表单、详情面板、最近日志、操作按钮和资源详情加载。

需求判断：v1.7 应把资源面板拆为稳定的小组件：

- `ResourcePanelContainer`：管理本地状态、请求和动作适配。
- `ResourceMasterList`：只展示列表、筛选和选中态。
- `ResourceCreateForm`：只展示新建资料表单。
- `ResourceDetailPane`：只展示详情、最近日志和动作入口。
- `ResourceEmptyState`：处理无项目、无资源、筛选为空等状态。

验收口径：

- 各展示组件不直接调用 `window.api`。
- 列表、表单、详情可分别测试。
- 资源筛选和选中资源切换不回归。

### RA17-07 全局资料库需要主进程聚合 API 和表格 UI 拆分

问题：`GlobalLibrary` 同时负责按项目拉取资源、聚合、错误处理、筛选、表格和空态。当前如果在前端遍历 `overview.projects` 并逐个请求项目详情，项目数增加后会形成 O(N) IPC 请求风暴；当 `overview.last_saved_at` 频繁变化时，还可能造成资料库卡顿、请求拥堵甚至窗口主链路响应变慢。

需求判断：v1.7 应抽出：

- `get_global_resources` 主进程聚合 IPC：由 service/repository 通过 SQL JOIN 一次性返回全部资源及所属项目摘要，前端不再按项目拉取详情聚合。
- `useGlobalLibraryResources`
- `GlobalLibraryFilters`
- `GlobalLibraryTable`
- `GlobalLibraryState`

`get_global_resources` 首版不做分页和服务端筛选，输入固定为空对象或 `undefined`，输出固定为：

- `items: Array<{ resource: ResourceSummary; project: Pick<ProjectSummary, 'id' | 'name' | 'status' | 'progress_percent'> }>`
- `total: number`

排序固定为先沿用首页项目顺序：`COALESCE(project.last_studied_at, project.updated_at) DESC, project.created_at DESC, project.id ASC`；
同一项目内沿用项目详情资源顺序：状态优先级 `learning`、`review`、`not_started`、`paused`、其他，其后
`COALESCE(resource.last_studied_at, resource.updated_at) DESC, resource.created_at DESC, resource.id ASC`。
搜索、项目筛选、类型筛选、状态筛选继续在前端基于全量 `items` 处理；无项目时返回 `{ items: [], total: 0 }`。

验收口径：

- 全局资料库加载路径从 O(N) IPC 降为 O(1) IPC。
- `useGlobalLibraryResources` 只调用 `get_global_resources`，不遍历项目详情接口。
- 主进程聚合 API 返回统一 `IpcResult`、DTO schema 和 output schema，字段使用 IPC DTO 的 `snake_case` 契约。
- 聚合 API 不暴露分页假象；前端筛选总是针对全量资料库结果。
- 全局资料库保留现有项目筛选、类型筛选、状态筛选、搜索和重试能力。
- 加载失败时继续显示可恢复操作。
- 聚合请求只在用户进入 `library` tab 或主动刷新资料库时触发，不能在 App bootstrap 后台请求。
- `overview.last_saved_at` 改变时不立即后台拉全量资料；如果当前不在资料库 tab，只标记为 stale。

### RA17-08 通用 UI 壳和格式化工具需要沉淀

问题：弹窗结构、日期格式化、按钮/搜索/表格内联样式、资源状态文案在多个文件中重复或局部漂移。

需求判断：v1.7 应沉淀低风险通用工具：

- `components/ModalFrame`
- `components/ModalActions`
- `renderer/lib/dateFormat.ts`
- `renderer/lib/uiClasses.ts` 中可复用的搜索/筛选/表单 class 常量

验收口径：

- 业务表单字段仍留在 feature。
- 通用组件不包含修炼业务文案。
- 日期显示在侧栏、日志、统计、资源详情中使用统一 helper。

### RA17-09 文档技术栈与实际依赖需要校准

问题：项目规范中提到 React Query、zustand、react-router-dom、react-hook-form，但当前 `package.json` 没有这些依赖，源码也未使用。实际实现是 `useState/useEffect + window.api + local helper`。

影响：后续实施者可能误以为必须引入或使用这些库，导致 v1.7 分层治理偏离“最小足够设计”。

需求判断：v1.7 不以引入这些库作为目标。这是 v1.7 针对当前依赖现实做出的明确版本级例外：本版本继续使用
本地 hook、zod schema、`unwrap/unwrapResult` 和显式刷新机制收敛边界，不新增请求库、状态库、表单库或路由库。
若后续确需 React Query、zustand、react-hook-form 或 react-router-dom，进入 v1.8 之后的单独版本评估。

验收口径：

- v1.7 实施方案明确“不为分层治理新增请求/状态/表单库”。
- v1.7 实施方案明确手写 hook 的查询、刷新、stale、表单校验边界。

## P2 需求分析

### RA17-11 输出 DTO 校验逐步覆盖

问题：输出 schema 已定义，但全面校验每个 IPC 输出可能带来额外工作量和兼容风险。

需求判断：v1.7 先覆盖高价值命令，其他命令进入后续版本或按改动随手覆盖。输出校验必须用于发现契约漂移，而不能成为生产主链路的单点故障。

验收口径：

- 开发/测试环境中，关键输出校验失败时转换为统一 AppError payload。
- 生产环境中，关键输出校验失败只记录 warning 日志并返回原始数据。
- 不因输出校验改变正常用户路径。

### RA17-12 shared 测试不应依赖 Node 原生模块

问题：`shared` 测试中存在 `better-sqlite3` 依赖，可能让纯逻辑测试受原生 ABI 影响。

需求判断：v1.7 可将 SQLite 对照测试迁入 main/db 测试，shared 只保留纯函数和 property test。

验收口径：

- `shared/` 测试可以不依赖 Electron 或 better-sqlite3 ABI。
- 数据库相关校验仍由 main/db 测试覆盖。

### RA17-13 schema 与 enum 重复需要防漂移

问题：数据库 schema 中 SQL 枚举字符串与 shared enum 数组重复维护。

需求判断：v1.7 可通过一致性测试先防漂移，不强制改 schema 生成方式。

验收口径：

- 新增枚举一致性测试。
- 后续新增枚举时能被测试提醒。

## 固定决策

1. v1.7 是架构治理版本，用户可见变化应尽量少。
2. service 仍是业务入口，repository 不做业务判断。
3. repository 只做同步 SQL 和 row 映射，事务唯一控制层是 service。
4. opener 窗口生命周期逻辑保留在 service/opener 层，不进入 repository。
5. `App.tsx` 是最外层 shell 宿主，不持有业务数据、草稿、弹窗草稿或 overlay 状态。
6. workbench 使用 `WorkbenchDataContext`、`WorkbenchActionsContext`、`AppFeedbackContext` 三轨上下文，避免大规模 prop drilling 和高频反馈重渲染。
7. 跨 feature 的驾驶舱组合进入 `features/workbench`。
8. 展示组件不直接访问 `window.api`。
9. 有意需要读取 `IpcResult.ok` 的冲突分支必须通过命名 helper 或注释说明。
10. Drizzle snapshot 元数据修复是 M1 前置门禁，优先采用 Git 历史节点生成 snapshot；无逐迁移历史节点时，只能在临时 worktree 中重建中间 schema 生成 snapshot，不从物理数据库逆向恢复。
11. 全局资料库使用主进程 `get_global_resources` 聚合 API，前端不做按项目 O(N) IPC 聚合；首版返回全量结果，不做分页。
12. 所有 P0 重构都必须配套 focused 测试或等价回归验证。
13. v1.7 不新建 `main/domain` 层；领域纯函数继续放 `shared/`，service 暂作业务编排。domain 层若要补齐，进入后续版本。
