# 需求分析 v1.5

版本：v1.5
状态：草稿
整理日期：2026-05-31
前置版本：`../v1.4/需求分析.v1.4.md`
对应实施方案：`实施方案.v1.5.md`

## 分析结论

v1.5 的核心目标不是继续扩展功能，而是把现有自用 MVP 收敛成稳定、清楚、每天敢用的学习闭环。当前主要矛盾有三类：

1. 文档目标和实现已经分叉：旧需求仍写“不做境界/等级”，但代码和数据库已经实装境界、突破和道基计算。
2. 首页与出关流程还没有足够快地服务核心动作：打开应用后应立刻知道继续学什么，记录时应 30 秒内完成。
3. 前端、样式和 IPC 已经出现可维护性风险：全局控制器、样式堆叠、弱类型边界和 N+1 聚合会继续放大后续改动成本。

本文件只做需求分析和范围判断，不直接承诺全部进入 v1.5 实施。进入开发前，应把 P0 条目同步到 `实施方案.v1.5.md` 的任务、验收场景和不做事项。

## 范围原则

- P0 必须先解决“学习闭环能不能清楚运行”：首页恢复、出关记录、有效选中状态、待记录语义、关键数据查询。
- 境界系统如果保留，只能定义为“项目级轻量成就反馈”，不能升级成独立游戏系统。
- 架构重构只允许服务 P0 问题，不做全量重写、不换技术栈、不引入无关框架。
- 系统流体验要服务“继续学习、记录进度、下次接上”，视觉表现不能反过来遮挡信息。
- 行号证据来自当前审查快照，后续以文件和符号位置为准。

## P0 需求分析

### RA15-01 文档目标和代码实现冲突

问题：旧版需求文档快照中仍强调 MVP 不做境界、等级、经验值，但 `src/shared/realm.ts` 已定义境界、道基分、突破条件，`drizzle/0002_realm_breakthrough.sql` 已把 `realm_rank`、`realm_layer`、`last_breakthrough_at` 和 `breakthrough_attempts` 落库。

影响：如果继续按“不做境界”验收，当前代码会被视为越界；如果按现有代码验收，需求文档又缺少范围、文案和验收口径。二者冲突会导致后续实现不断摇摆。

需求判断：v1.5 应正式选择“保留但降维”的方案，把境界定义为项目级轻量成就反馈。它只表达一个学习方向的阶段感，不承担角色等级、战力、排行榜、成长系统等游戏功能。

范围边界：

- 进入 v1.5：统一境界定义、显示文案、突破触发条件、验收口径。
- 不进入 v1.5：角色养成、独立等级页、战力、排行榜、抽卡、成就墙扩展。

验收口径：

- 本版本文档不再同时出现“完全不做境界”和“境界已实现”的冲突表述。
- 项目级境界字段、突破记录、前端展示和反馈文案使用同一套定义。
- 普通用户能理解境界只是学习进度反馈，而不是新的任务系统。

### RA15-02 首页首屏没有完全回答核心问题

问题：`src/renderer/App.tsx` 的首页推荐卡当前主要展示资料标题、所属项目、进度条和下一步目标，缺少资料类型、上次学到哪、最近何时出关等恢复上下文。

影响：用户打开应用后仍需要扫列表或回忆历史，不能立刻判断“我该继续学什么、为什么是它、上次停在哪里、这次该做什么”。

需求判断：首页首屏必须从“推荐一个资料”升级为“恢复学习现场”。推荐卡是系统流主镜头，也是日常使用的入口，不应只做标题卡片。

必备信息：

- 资料名与资料类型。
- 所属项目与项目总进度。
- 当前进度百分比和进度描述。
- 下一步目标。
- 最近出关时间或“尚未出关”状态。
- 存在 pending 时优先提示待出关记录。

验收口径：

- 在 1024x640 桌面窗口首屏内，无需滚动即可判断下一步学习对象。
- 用户能在 5 秒内回答：继续学什么、上次学到哪、完成多少、下次做什么。
- 资料类型、最近出关、下一步目标没有数据时有清楚的空状态文案。

### RA15-03 项目刷新和选中状态有竞争

问题：`src/renderer/App.tsx` 的 `refresh` 依赖旧 `selectedProjectId` 闭包。创建项目后先 `setSelectedProjectId(project.id)` 再 `refresh()`，删除项目后先置空再刷新，都可能因为状态更新异步而使用旧 ID。

影响：创建后可能没有选中新项目，删除后可能继续请求已删除项目，表现为详情错位、旧数据闪现或错误 Toast。

需求判断：刷新函数必须显式接收目标项目 ID，并在拿到最新项目列表后校验目标是否仍存在。

建议行为：

- `refresh(nextSelectedProjectId?)` 接收目标 ID。
- 目标存在时选中目标。
- 目标不存在时选择列表第一个有效项目。
- 列表为空时清空选中和项目详情。
- `loadProject` 只接收已经校验过的有效 ID。

验收口径：

- 创建项目后，新项目立即成为当前选中项，详情加载该项目。
- 删除当前项目后，不再请求已删除项目，自动回退到有效项目或空状态。
- 手动刷新不会改变用户当前有效选中项。

### RA15-04 `App.tsx` 已成为全局控制器

问题：`src/renderer/App.tsx` 当前约 901 行，同时管理数据加载、项目选择、资源动作、出关表单、Toast、确认框、pending 会话、突破浮层和主布局。

影响：局部需求会牵动全局状态，后续修复首页、pending、出关弹窗或项目刷新时容易产生交叉回归。

需求判断：v1.5 不做大重写，但必须做支撑 P0 的局部拆分，让关键状态有清楚归属。

建议拆分：

- `useOverviewQuery`：首页概览、枚举加载、刷新和错误处理。
- `useSelectedProject`：选中项目、目标校验、详情和境界加载。
- `useResourceActions`：创建、编辑、删除、继续学习、打开资料。
- `usePendingSession`：pending 获取、自动弹出、放弃、关闭来源。
- `useConfirmDialog`：确认框请求和结果回传。

验收口径：

- `App.tsx` 不再承载主要业务 mutation 细节。
- 创建、删除、刷新、pending 关闭等核心路径可以分别测试。
- 拆分后用户可见行为保持不变，除已明确修复的问题外不引入交互变化。

### RA15-05 视觉系统靠堆叠覆盖运行

问题：`src/renderer/styles.css` 当前约 3758 行，布局、按钮、主题变量、境界色板、特效层和响应式规则混在一起，存在重复定义和后写覆盖。

影响：视觉修正不可预测，新增主题或响应式规则容易影响无关组件。系统流表现也容易滑向“堆特效”而不是强化信息层级。

需求判断：v1.5 只做局部分层，不做视觉全量重写。目标是让主题变量、布局、组件和效果分离，降低后续修改成本。

建议分层：

- `tokens.css`：颜色、字号、间距、阴影、主题变量。
- `layout.css`：应用骨架、侧栏、内容区、响应式网格。
- `components.css`：按钮、卡片、表单、弹窗、Toast、资源行。
- `themes.css`：境界主题变量，只改 token，不重写组件结构。
- `effects.css`：粒子、突破、低频系统流动效。

验收口径：

- 同一类按钮、卡片、表单不再靠多处覆盖实现最终样式。
- 切换境界主题主要改变变量，不改变组件布局。
- 1024x640、1280x720、390x844 下核心文字和按钮不重叠、不截断关键含义。

### RA15-06 IPC 类型边界不够硬

问题：`src/main/ipc/router.ts` 里多个 handler 使用 `as never` 或局部类型断言，`src/preload/api.ts` 的 `invoke<T>` 只给渲染层提供泛型假设，没有运行时输出校验。

影响：DTO、preload、主进程 service 漂移时，错误可能穿过 IPC 边界进入渲染层，变成难定位的 UI 状态 bug。

需求判断：v1.5 应建立最小 command map，把命令名、输入 schema、输出 schema 和 handler 类型绑定起来。重点覆盖本轮触碰的关键通道。

建议范围：

- 首批覆盖 `get_home_overview`、`get_project_detail`、`list_projects`、`create_project`、`delete_project`、`continue_resource`、`save_study_log`、`select_local_file`。
- `select_local_file` 从宽泛 properties 改成明确 `{ kind: 'file' | 'folder' }` 语义。
- IPC 返回统一通过 output schema 或类型守卫校验。

验收口径：

- 新增 IPC 命令必须同时声明 input schema、output schema 和 handler。
- 渲染层不再依赖裸 `invoke<T>` 证明返回类型正确。
- 输出不符合契约时返回结构化错误，而不是让 UI 消费未知数据。

### RA15-07 全局资源库 N+1 且会漏数据

问题：`GlobalLibrary.tsx` 对 overview 中每个项目调用 `get_project_detail` 聚合资源；`cultivationService.getProjectDetail` 默认每个项目只取 100 条资源。

影响：项目多时会产生 N+1 请求；单项目超过 100 条资源时，全局资源库会漏数据。全局视图的总量、筛选和分页也缺少后端口径。

需求判断：全局资源库需要主进程级 `list_resources` 查询，不能靠逐项目详情页拼接。

建议能力：

- 支持分页：`limit`、`offset`、`total`。
- 支持筛选：项目、资料类型、状态、关键词。
- 返回资源摘要时附带项目名和项目 ID。
- 排序规则与项目详情资源列表保持一致，避免两个入口看到不同顺序。

验收口径：

- 101 条以上资源不会在全局资源库漏掉。
- 多项目场景只发起一个资源列表查询，而不是按项目数量发起详情查询。
- 筛选和分页由主进程返回稳定 total，前端不做全量本地假分页。

### RA15-08 出关记录表单超出 P0 成本

问题：`StudyLogModal.tsx` 默认展示出关进度、百分比、状态、证据类型、下一步、时长等字段，其中部分字段还被要求必填。这与“30 秒记录一次学习”的目标冲突。

影响：记录成本高会降低用户坚持记录的概率。主题字段过多时，修仙表达会压过真实记录动作。

需求判断：出关弹窗应默认只暴露最低成本字段，高级字段折叠。真实动作优先，主题表达辅助。

默认字段：

- 学到哪：可为空或非阻断，支持简短文本。
- 百分比：可选，用户不填时不强制更新进度。
- 下次做什么：突出为闭环核心字段。

高级字段：

- 状态。
- 证据类型。
- 闭关时长。
- 其他用于境界计算但不需要每次干扰用户的字段。

验收口径：

- 默认视图中用户能在 30 秒内保存一次有效记录。
- 保存中不能关闭弹窗。
- 取消、关闭、Escape 对已编辑草稿触发未保存确认。
- 高级字段折叠后仍可访问、可键盘操作。

### RA15-09 修仙术语体系混乱

问题：界面中“资料、功法、秘卷、秘籍库、法门、灵网、洞府、闭关、出关”等词并存，有些页面真实名在前，有些页面主题词单独出现。

影响：用户要先猜词，再理解功能。尤其是新建、继续、记录、删除这类主动作，如果只用主题词，会增加误操作风险。

需求判断：v1.5 需要建立双层文案规则：真实含义在前，主题词在后。主题词用来增强氛围，不能替代动作含义。

建议术语：

| 真实含义 | 主题词 | 推荐显示 |
| --- | --- | --- |
| 项目 | 法门 / 修炼方向 | 修炼方向（法门） |
| 资料 | 功法 / 秘卷 | 资料（秘卷） |
| 网页 | 灵网 | 网页（灵网） |
| 学习 | 闭关 / 参悟 | 继续学习（闭关） |
| 学习记录 | 出关记录 | 学习记录（出关） |
| 项目级成就 | 境界 | 项目境界 |

验收口径：

- 主按钮、表单 label、危险操作确认必须包含真实动作。
- 主题词不单独承担删除、保存、打开、放弃等高风险动作。
- 同一概念跨首页、资源管理、全局资源库、弹窗使用同一术语表。

### RA15-10 可访问性和响应式基础不足

问题：`ResourceManagementPanel.tsx` 使用 `div onClick` 做可选资源卡；`App.tsx` 的顶部页签没有 `tablist` / `tab` 语义；突破浮层缺少完整 dialog、焦点管理和 Escape 处理口径。

影响：键盘用户无法稳定操作，屏幕阅读器无法理解选中状态。窄屏和低高度窗口下，资源管理、全局资源库、分析页也容易裁切核心内容。

需求判断：v1.5 至少补齐核心路径的语义和键盘访问，不要求达到对外发布级完整 WCAG 自动化覆盖。

建议范围：

- 页签使用 `role="tablist"`、`role="tab"`、`aria-selected`、键盘左右切换。
- 资源可选卡改为 button，或补齐 `role`、`tabIndex`、键盘事件和 `aria-pressed` / `aria-selected`。
- 所有弹窗和突破浮层使用 dialog 语义、初始焦点、焦点循环、Escape、关闭后焦点回归。
- 资源双栏、全局资源库表格、分析面板补移动端和低高度布局。

验收口径：

- 不用鼠标可以完成：切换页签、选择资料、继续学习、打开出关记录、关闭弹窗。
- Escape 行为与未保存确认规则一致。
- 1024x640 和 390x844 下主路径无关键内容重叠。

## 系统流漫画视角

### RA15-11 主线不够清楚

当前界面已经有修仙皮肤、境界色板、粒子和突破反馈，但日常主线仍应是学习工具：继续学习、记录进度、下次接上。v1.5 不应继续堆背景特效，而应让每次打开和保存都有明确因果。

验收口径：用户完成一次学习记录后，能看到“我刚记录了什么、进度怎么变、下次回来做什么”。

### RA15-12 缺少打开应用的系统主镜头

打开应用的第一屏应像系统提示面板：当前推荐、待记录状态、最近出关、项目进度同时构成“下一步指令”。这不是做营销 hero，而是把恢复学习现场变成第一视觉。

验收口径：首页第一屏的信息层级清楚，推荐卡和待记录提示优先于装饰元素。

### RA15-13 缺少保存出关后的数值跃迁

保存出关记录后，当前反馈偏 Toast 和瞬时特效。系统流体验需要持续显示这次保存带来的变化，包括进度变化、下一步目标、最近出关时间、距离突破还差什么。

验收口径：保存后首页或侧栏有持久反馈，不依赖用户记住 Toast。

### RA15-14 缺少突破时的境界定格

突破应该是少数关键事件，不应与普通资料完成反馈混用。当前文案如“功法大成 / 成功参悟秘卷”容易把项目突破和资料完成混在一起。

验收口径：项目突破使用“修炼方向 / 项目境界提升”的语义；资料 100% 使用“资料学习完成”的语义；两类反馈文案和视觉层级分开。
