版本：v1.6
状态：草稿

## 1. 版本目标

v1.6 的目标是用小幅重构完成“修行驾驶舱”：

1. 当前学习页成为主工作台，首屏回答继续学什么、上次学到哪、离突破差什么。
2. pending 待出关记录优先级最高，确保学习闭环不会断在记录处。
3. 资源的修炼定位和权重更可见，用户能理解哪些资料影响境界。
4. 境界诊断开始解释资料数量、权重和有效学习时间。
5. UI 主题收敛，修仙表达服务真实学习动作。
6. 重构只拆展示组件，不搬动核心业务状态和 IPC。
7. 前端派生展示逻辑通过轻量 ViewModel 收口，避免 JSX 中复制诊断、格式化和阈值。

## 2. 不做事项

- 不做商城、抽卡、战力、排行榜、宠物、宗门经营。
- 不做角色总等级或跨项目战力。
- 不做破坏性数据库迁移。
- 不重写 `App.tsx` 的业务流，不把 modal、toast、confirm、pending 分散到多个宿主。
- 不改变 pending 单会话规则。
- 不改变继续学习、保存日志、尝试突破的 IPC 命令名、输入语义和既有字段含义；允许在现有输出 DTO 上补充向后兼容的派生字段。
- 不把五行统计升级为核心推荐算法。
- 不做全量 CSS 重写，只做必要样式新增和局部收敛。
- 不引入新的全局状态管理库、动画框架或虚拟列表依赖。

## 4.2 前端工程约束

v1.6 的前端实现按以下边界执行：

1. 数据边界：`App.tsx` 继续持有首页 overview、项目详情、cultivation、pending、toast、modal、confirm 和突破 overlay 状态。新建驾驶舱组件不得直接调用 `window.api`。
2. 容器例外：`ResourceManagementPanel` 继续持有资源筛选、选中资源和 `get_resource_detail`；`GlobalLibrary` 当前按项目拉详情的既有 IPC 模式本轮可保留，但不能扩散到新驾驶舱组件。
3. ViewModel 边界：推荐卡、法门概览、突破诊断摘要、保存后回显全部从纯函数获得展示模型；纯函数不使用 React hook、DOM、Electron、随机数或当前时间隐式读取，时间值由调用方传入。
4. Action 边界：展示组件通过 `actions` 或命名回调接收行为，不自行决定 continue、pending 冲突、保存日志或突破重试流程。
5. 性能边界：资源筛选、诊断条件、保存后摘要、背景粒子配置使用稳定 key 和 `useMemo`；hover、focus、badge 文案变化不得造成卡片尺寸跳动。
6. 可访问边界：重复的“继续学习 / 记录进度”按钮必须有可区分 accessible name；保存后驾驶舱回显使用 `aria-live="polite"` 或等效机制；弹窗关闭后焦点回到触发入口。
7. 样式边界：新增 HUD 面板样式复用现有 Tailwind/CSS token 和 lucide-react 图标，不在组件内散落复杂 inline style。
8. 推荐边界：驾驶舱推荐资料固定从当前 `selectedProject` 的 `projectDetail.resources.items` 推导；`overview.recommended` 仅在同属当前项目时作为 fallback。
9. 突破边界：突破阈值固定从 `src/shared/realm.ts` 导出的 `BREAKTHROUGH_TARGETS` 读取，renderer 不再维护第二套阈值。

## 4.3 三视角审核结论

本方案已吸收三个子 agent 的只读审核结果，执行时按以下结论落地：

1. 10 年高级全栈视角：先打通 shared DTO、realm、service 查询、renderer ViewModel，再接 UI；数据库不迁移；推荐资料按当前项目推导；有效学习时间必须从 `study_logs.duration_minutes`、证据、心得和进度变化计算。
2. 5 年高级 UI 视角：HUD 不是漫画浮窗复刻，而是带系统感的学习仪表盘；每个面板固定为标题、状态标识、主信息、辅助解释、主动作、次动作、状态反馈；截图验收覆盖正常、pending、可突破、不可突破、空态、错误态和 reduced motion。
3. 5 年高级前端视角：新驾驶舱组件不得调用 `window.api`；`ProjectStatsPanel` 不可突破按钮必须真正禁用；重复按钮必须带资源名级 accessible name；ViewModel/helper 测试随契约层一起提交。

### PLAN-v1.6-002A 建立驾驶舱 ViewModel 和展示工具层

目标：先把驾驶舱需要的派生展示数据收口，再接入推荐、概览、诊断和保存后回显组件。

新增文件：

- `src/renderer/features/projects/cultivationDiagnostics.ts`
- `src/renderer/features/resources/resourceDisplay.ts`

修改范围：

- `src/renderer/App.tsx`
- `src/renderer/features/projects/ProjectStatsPanel.tsx`
- `src/renderer/features/resources/ResourceManagementPanel.tsx`

ViewModel 原则：

```ts
type CockpitViewModel = {
  recommendation: RecommendedStudyViewModel | null;
  cultivationStrip: ProjectCultivationViewModel | null;
  breakthroughDiagnostic: BreakthroughDiagnosticViewModel | null;
  lastStudyFeedback: LastStudyFeedbackViewModel | null;
};
```

- 输入使用 `overview`、`selectedProject`、`projectDetail`、`projectCultivation`、`lastStudyFeedback` 和调用方传入的 `now`。

- 空项目、无推荐、无 cultivation、旧日志缺字段都要有显式空态。
- `cultivationDiagnostics.ts` 导出 `buildBreakthroughConditions`、`getPrimaryBottleneck`、`buildBreakthroughDiagnosticViewModel`、`formatRealmLabel`，阈值从 `src/shared/realm.ts` 导入。
- `resourceDisplay.ts` 导出 `getResourceRoleDisplay`、`getResourceWeightDisplay`、`getResourceTypeLabel`、`getResourceStatusLabel`。

执行步骤：

1. 梳理现有 `App.tsx` 中推荐、项目、cultivation、保存日志结果的字段来源。
2. 新增纯函数 `buildCockpitViewModel`，返回 `recommendation`、`cultivationStrip`、`breakthroughDiagnostic`、`lastStudyFeedback` 四个切片。
3. `buildCockpitViewModel` 固定从当前 `selectedProject` 的 `projectDetail.resources.items` 推导推荐资料；`overview.recommended` 仅在 `project_id` 等于当前项目时作为 fallback。

5. `ProjectStatsPanel` 和 `BreakthroughDiagnosticCard` 共用 `cultivationDiagnostics.ts`。
6. `ResourceManagementPanel`、资源详情和全局资料库复用 `resourceDisplay.ts` 文案。
7. 本任务不优化既有重复 IPC；扫描到的既有重复 IPC 写入 `docs/artifacts/v1.6/acceptance-summary.md` 的风险记录。

- 新增展示 helper 都是纯 TypeScript，不依赖 React、DOM、Electron 或 `window.api`。

## 14. 风险和缓解

| 风险 | 表现 | 缓解 |
| --- | --- | --- |
| App 拆分破坏业务流 | pending、toast、modal、confirm 状态丢失 | 只拆展示组件，业务状态留在 App |
| 当前项目推荐错位 | 驾驶舱诊断 A 项目，却推荐 B 项目资料 | `cockpitViewModel.ts` 只从当前项目资源推导推荐，同项目 `overview.recommended` 才能 fallback |

| 诊断摘要和完整诊断逻辑重复 | 条件阈值不一致 | 阈值从 `src/shared/realm.ts` 导出，renderer 只通过 helper 展示 |
