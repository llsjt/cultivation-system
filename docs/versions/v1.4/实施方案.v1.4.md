## 1. 实施目标

MVP 只验证一个闭环：用户能低成本记录单个学习资料的“当前学到哪里、完成多少、下次做什么”，并在下次打开工具时立刻恢复学习上下文。

核心对象是资料级学习进度。项目、首页、日志、修仙主题、成就反馈都只服务这个对象。项目进度只作为资料进度的汇总展示，不作为独立管理目标。

第一版必须让首页第一屏回答三个问题：

1. 我现在该继续学哪个资料？
2. 这个资料上次学到哪里、完成多少？
3. 下一步具体做什么？

强制原则：

1. SQLite 是唯一可信数据源。
2. 推荐规则、状态与进度归一化、路径安全、事务一致性必须在 Electron 主进程的 Service 层实现，渲染进程校验只优化体验。
3. 渲染进程保存后必须以主进程返回的规范化实体刷新（TanStack Query invalidate），不只做乐观更新。
4. 进入任何阶段前，前一阶段的"完成标准"必须全部通过。

## 3. 范围与优先级

### 3.1 P0 核心闭环

P0 只验证用户是否愿意低成本记录学习进度，并能下次恢复。

必须包含：

| 能力 | 实施要求 |
| --- | --- |
| 创建项目 | 创建一个修炼方向，作为资料归属 |
| 添加资料 | 添加学习资料，默认进度 0 |
| 记录进度 | 记录“学到哪里”和 0 到 100 百分比 |
| 记录下一步 | 记录下次继续做什么 |
| 首页推荐 | 首页推荐一个可继续资料 |
| 出关记录 | 用最少字段保存一次学习进度 |
| 自动快照 | 保存时自动生成 `study_logs` |
| 本地持久化 | 关闭重开后项目、资料、进度、下一步仍存在 |
| 基础反馈 | 保存成功、进度提升、100% 完成有轻量提示 |

P0 出关记录只强曝光 3 个字段：

1. 出关进度：这次学到哪里。
2. 参悟进度：完成百分比。
3. 下次闭关目标：下次做什么。

P0 状态只自动产生 `not_started`、`learning`、`completed`。`review` 和 `paused` 从 P1 起进入出关记录手动选择。

### 3.2 P1 发布硬化

P1 用于让 P0 闭环达到发布质量。自用 MVP 可以先完成 P0 并持续使用；对外发布前必须补齐 P1。

必须补齐：

- 待记录会话与冲突处理。
- 文件、文件夹、URL 打开。
- `record_only` 完整规则。
- 打开失败分类。
- 路径和 URL 安全规则。
- 删除确认与删除影响。
- SQLite 强约束。
- 事务回滚。
- 迁移可靠性。
- 主进程权限最小化。
- 项目详情页和资料详情页完整状态。
- 最近学习记录展示。

### 3.3 P2 暂不进入主线

P2 只能在 P0/P1 稳定后进入，包括更完整修仙视觉、进度动画、更细状态、学习时长增强、备注增强、更多历史记录视图、搜索、筛选、统计、导入导出、AI、复习提醒。

## 4. 架构方案

### 4.1 明确技术栈

本方案面向**单人自用**场景，目标是"先尽快跑通 P0 `record_only` 竖切闭环、视觉只做轻量支撑、迭代不被工程化拖累"。完整 M0–M3 仍按 §20 的 12–18 ID 粗估执行，不再用"3 天"作为完整核心闭环承诺。技术栈选型经全栈、产品、前端、后端四个视角综合后冻结如下：

版本基准：2026-05-30 已按 Node 官方 LTS 索引与 npm registry `latest` dist-tag 复核。M0 初始化必须按下表写入 exact versions，不使用 `^` / `~` 浮动范围；若实际开工日 `latest` 已变化，先重跑 §4.1.7 的复核命令，再统一刷新本节与 M0-01 依赖命令。

> 选型核心原则：
> 1. 单一语言（TypeScript）打通主进程与渲染进程，避免 Rust/TS 双语调试。
> 2. Schema 即代码（Drizzle），不再自研迁移模块。
> 3. 优先选择"配置即用"的修仙视觉生态，不自己造动效系统。
> 4. 自用阶段不投入 Playwright、性能采样、可访问性自动化、签名、自动更新等发布级工程化。

### 4.2 分层架构

| 层 | 方案 | 职责 |
| --- | --- | --- |
| 桌面壳 | Electron 主进程 | 窗口生命周期、单实例、系统打开能力、IPC 路由 |
| 渲染进程 | React + TS + Vite | 页面展示、表单交互、状态反馈、动效 |
| 数据库 | SQLite + better-sqlite3 | 本地持久化与强约束 |
| 渲染态 | Zustand + TanStack Query | 全局状态机、IPC 缓存与失效 |
| 渲染 → 主进程 | Preload + contextBridge | 暴露 14 个语义 IPC 通道，无任意 shell/fs 能力 |
| IPC Handlers | `ipcMain.handle('cmd:xxx', ...)` | 接 DTO，转 Service，无业务逻辑 |
| Services | 业务服务层 | 事务、状态归一化、推荐、出关保存、资料变更冲突 |
| Repositories | Drizzle 查询 / 写入 | SQL 封装、分页、索引命中 |
| Domain | 领域规则 | 枚举、状态/进度、推荐排序、Zod 校验、路径风险分级 |
| DB | better-sqlite3 连接 + drizzle-kit 迁移 | 启动迁移、`PRAGMA foreign_keys`、事务封装 |
| Resource Opener | Node `path` + `fs.realpathSync` + `shell.openPath` / `shell.openExternal` | 文件、文件夹、URL 校验和系统打开 |
| Errors | 统一 AppError | 错误码、user_message、recoverable、details 脱敏 |
| Logger | pino + pino-roll | 结构化日志、按大小滚动、路径/URL 脱敏 |

架构原则：

1. SQLite 是唯一可信数据源。
2. 渲染进程禁用 nodeIntegration，启用 contextIsolation；只能调用 preload 暴露的语义 IPC，不得直接访问 fs / shell。
3. 推荐规则、状态与进度归一化、路径安全、事务一致性必须在主进程实现。
4. 渲染进程保存后必须以主进程返回的规范化实体刷新（TanStack Query invalidate），不只做乐观更新。
5. 业务全部用 TypeScript 写，主进程与渲染进程共享 Zod schema，避免双语调试。
6. **同步事务边界原则**：better-sqlite3 是同步 API，`db.transaction(() => {...})` 内部禁止出现任何 `await` / Promise / 异步 I/O。需要异步的副作用（`shell.openPath`、`shell.openExternal`、`fs.realpathSync` 之外的异步探测等）必须在事务**外**先执行并 await，拿到结果后再开同步事务写库。详见 §6.6。
7. **`updated_at` 写入不变量**：`resources.updated_at` 表示会影响资料内容、元信息或出关保存覆盖判断的版本时间。创建、改元信息、改打开方式、改非终态状态、出关保存都必须在同一事务内重写 `resources.updated_at`；仅更新 `last_opened_at` 的继续闭关打开成功不 bump `resources.updated_at`，避免本次打开行为被误判为资料变化冲突。`projects.updated_at` 同理表示项目内容/派生状态变化版本，保存出关、添加/删除资料、修改项目元信息时必须 bump。
8. **错误不穿透 IPC**：AppError 不依赖 throw 跨 IPC 传递（Electron 在 `ipcRenderer.invoke` 的 reject 上只可靠保留 `message`，会丢失 `code`/`recoverable`/`details`）。主进程 IPC router 必须 catch 后以结果信封 `{ ok, data?, error? }` 返回，详见 §5 与 §19。

## 5. 主进程命令契约（IPC）

M0 阶段冻结命令名、输入、输出和错误码。所有命令通过 Electron `ipcMain.handle` 注册，preload 层用 `contextBridge.exposeInMainWorld('api', { ... })` 暴露给渲染进程。

| 命令 | 输入 | 输出 | 说明 |
| --- | --- | --- | --- |
| `get_home_overview()` | 无 | 推荐、待记录、项目摘要、最近日志 | 首页唯一数据入口 |
| `list_projects(input)` | 可选分页参数 | 项目摘要列表 | 默认最多 50 条 |
| `get_project_detail(project_id, input)` | 项目 ID、资料分页 | 项目详情、资料列表、最近日志 | 资料默认最多 100 条 |
| `create_project(input)` | 名称、描述 | 创建后的项目 | trim 校验 |
| `update_project(project_id, input)` | 名称、描述 | 更新后的项目 | 不允许手动传状态 |
| `delete_project(project_id)` | 项目 ID | 删除结果 | 事务内级联删除 |
| `create_resource(input)` | 项目、标题、类型、打开方式、路径、初始进度、下一步 | 创建后的资料 | 资料必须归属项目 |
| `update_resource(resource_id, input)` | 标题、类型、打开方式、路径、非终态状态标记 | 更新后的资料 | 不接收项目 ID、进度、下一步 |
| `delete_resource(resource_id)` | 资料 ID | 删除结果 | 日志保留快照，`resource_id` 置空 |
| `get_resource_detail(resource_id)` | 资料 ID | 资料详情、最近日志 | 资料详情页 |
| `continue_resource(input)` | 资料 ID、可选风险确认 token | 继续闭关结果 | 打开资源或进入仅记录流程 |
| `save_study_log(input)` | 资料 ID、进度、下一步、状态、来源、`resource_updated_at_before`、可选时长/备注 | 保存结果、更新后资料、项目进度 | 事务保存并生成日志 |
| `get_pending_session()` | 无 | 当前待记录会话或空 | 应用启动恢复 |
| `abandon_pending_session(session_id)` | 会话 ID | 放弃结果 | 不创建日志，不更新时间 |
| `get_enums()` | 无 | 枚举和展示文案 | 前端渲染下拉项 |

`continue_resource(input)` 返回类型：

| 返回类型 | 含义 | 前端行为 |
| --- | --- | --- |
| `opened` | 文件、文件夹、URL 成功打开并创建待记录 | 显示待记录提示条 |
| `record_only` | 仅记录资料，不打开外部资源 | 无 pending 时直接打开出关记录弹窗 |
| `pending_conflict` | 已有待记录会话 | 弹出三选一冲突处理 |
| `blocked` | 安全规则阻断或强提示待确认 | 硬阻断显示阻断原因；强提示返回一次性确认 token，前端展示“打开一次” |
| `open_failed` | 打开失败 | 显示失败原因，不进入待记录 |

返回优先级：

1. 已存在待记录会话时，必须优先返回 `pending_conflict`，包括目标资料是 `record_only`。
2. 只有不存在待记录会话时，`record_only` 才返回 `record_only`。
3. 文件、文件夹、URL 只有安全校验和系统打开成功后才返回 `opened`。

风险确认契约：

- `block_level = "hard"` 时没有确认 token，只能修改路径或改为仅记录。
- `block_level = "warn"` 时主进程返回 `risk_confirm_token`。用户点击“打开一次”后，前端必须再次调用 `continue_resource({ resource_id, risk_confirm_token })`。
- token 必须绑定 `resource_id`、规范化后的路径/URL 摘要、风险类型和短时有效期；路径/URL 或资料 `updated_at` 变化后 token 立即失效。
- “打开一次”不记忆偏好，不写入数据库白名单。

统一错误结构：

```text
AppError {
  code: string,
  user_message: string,
  recoverable: boolean,
  details?: object
}
```

IPC 错误信封（强制）：

- AppError **不通过 throw 跨 IPC 传递**。主进程 `ipcMain.handle` 在 reject 时只能可靠保留 `message`，`code` / `recoverable` / `details` 会在 IPC 序列化中丢失，导致渲染进程无法按错误码渲染固定文案。
- 因此每个命令统一返回结果信封：

```text
IpcResult<T> =
  | { ok: true,  data: T }
  | { ok: false, error: AppError }
```

- 主进程 router 必须 `try/catch` 整个 handler：捕获到 AppError 直接装入 `{ ok:false, error }`；捕获到非预期异常（含 better-sqlite3 原生错误）兜底为 `TRANSACTION_FAILED`（写路径）或 `DB_CONSTRAINT_FAILED`（约束类）并脱敏 `details`，绝不把原始错误文本透传给渲染进程。
- preload 仅做 IPC 转发，把 `ipcRenderer.invoke` 的返回原样传出；渲染进程按 `ok` 分支处理，`ok:false` 时按 `error.code` 渲染 §5 固定文案。

必须覆盖错误码（含固定用户文案与恢复动作；渲染进程按 `code` 渲染，不展示主进程原始错误）：

| 错误码 | 场景 | 固定 user_message | 恢复动作 | recoverable |
| --- | --- | --- | --- | --- |
| `VALIDATION_FAILED` | 表单或 DTO 校验失败 | 输入有误，请检查后重试。（具体字段错误放入 `details`） | 修正字段 | true |
| `NOT_FOUND` | 项目、资料、待记录不存在 | 没有找到对应内容，可能已被删除。 | 返回首页 / 刷新 | true |
| `PENDING_SESSION_CONFLICT` | 已存在待记录会话 | 已有一次闭关尚未出关记录，请先处理上一次学习记录。 | 记录上一次 / 放弃上一次 / 取消 | true |
| `UNSUPPORTED_URL_SCHEME` | URL 协议不支持 | 只支持 http:// 或 https:// 链接。 | 修改链接 | true |
| `INVALID_URL` | URL 解析失败或无 host | 链接格式无法识别，请检查后重新输入。 | 修改链接 | true |
| `PATH_NOT_FOUND` | 文件或文件夹不存在 | 找不到该路径，文件可能已被移动或删除。 | 修改路径 / 仅记录进度 | true |
| `PATH_BLOCKED` | 风险路径被阻断 | 为保护本地安全，已阻断此资源打开。 | 修改路径 / 仅记录进度 | true |
| `PERMISSION_DENIED` | 权限不足 | 没有权限打开该路径。可以修改路径，或仅记录进度。 | 修改路径 / 仅记录进度 | true |
| `DB_CONSTRAINT_FAILED` | 数据库约束失败 | 保存失败，刚才填写的内容已保留。请稍后重试。 | 保留输入并重试 | true |
| `MIGRATION_FAILED` | 迁移失败 | 数据升级未完成，应用已停止启动以保护数据。 | 停止启动，进入只读诊断 | false |
| `TRANSACTION_FAILED` | 事务失败 | 保存失败，刚才填写的内容已保留。请稍后重试。 | 回滚并保留输入 | true |

`details` 仅承载渲染进程可安全展示的结构化信息（如字段名、风险原因分类），不得包含完整本地路径、完整 URL 或数据库原始错误文本。

### 6.3 强约束

必须落实：

- 所有时间字段存 UTC RFC3339，带 `Z` 或明确 offset。
- 项目名和资料标题 trim 后不能为空。
- 进度必须是 0 到 100 整数。
- 学习时长为空，或在 0 到 1440 分钟。
- 新建 `study_logs` 时 `resource_updated_at_before` 必须非空；DDL 允许 NULL 仅为兼容旧历史数据展示，服务层不得为新记录写入 NULL。
- 状态、类型、打开方式必须是枚举值。
- SQLite 每个连接必须开启并验证 `PRAGMA foreign_keys = ON`。
- 项目名称和资料标题允许重名，不设置唯一约束。
- `record_only` 的 `path_or_url` 必须为空或归一化为 `null`。
- 非 `record_only` 的 `path_or_url` trim 后必须非空。
- `pending_study_sessions` 使用固定 `singleton_key` 做数据库级单例约束。

推荐单例约束：

```sql
singleton_key integer not null default 1 check(singleton_key = 1),
unique(singleton_key)
```

外键行为：

| 关系 | 行为 |
| --- | --- |
| `resources.project_id -> projects.id` | 项目删除时级联删除资料 |
| `study_logs.project_id -> projects.id` | 项目删除时级联删除日志 |
| `study_logs.resource_id -> resources.id` | 资料删除时置空 |
| `pending_study_sessions.project_id -> projects.id` | 项目删除时级联删除待记录 |
| `pending_study_sessions.resource_id -> resources.id` | 资料删除时级联删除待记录 |

### 6.4 状态与进度归一化

服务层统一执行，渲染进程不能自行推断。

| 规则 | 行为 |
| --- | --- |
| `completed` | 强制进度 100 |
| `not_started` | 强制进度 0 |
| `learning` / `review` / `paused` | 有效进度为 1 到 99 |
| 输入 100 且无显式状态 | 归一化为 `completed` |
| 输入 0 且无显式状态 | 归一化为 `not_started` |
| 终态变更 | 必须通过 `save_study_log` 生成学习记录 |
| P0 | 1 到 99 默认归一化为 `learning` |
| P1 | 出关记录可选择 `review` / `paused` |

资料编辑边界：

- `update_resource` 只能修改标题、类型、打开方式、路径或 URL、非终态状态标记。
- `update_resource` 不接收项目 ID、进度、下一步、最近出关时间。
- 如果传入这些字段，主进程必须拒绝。
- 创建资料时允许填写初始进度和下一步，因为此时还没有学习记录。

### 6.5 项目进度与推荐

项目进度：

```text
项目进度 = round(项目下所有资料 progress_percent 的平均值)
```

规则：

- 无资料时项目进度为 0%。
- MVP 不保存项目进度缓存字段。
- 用户不能手动编辑项目进度。
- `projects.status` 虽同样为派生值，但为排序和列表展示落库存储；创建项目时写入派生值 `not_started`，后续每次影响项目状态的写操作都必须在同一事务内重新派生并写回 `projects.status`。

项目进度计算口径单一事实源（避免双实现漂移）：

- 读路径（首页 / 详情展示）以 SQL `round(avg(progress_percent))` 为**权威实现**。
- `src/shared/calc.ts` 的纯函数 `calcProjectProgress` 仅用于单元测试基线与反馈文案的内存计算，不得作为展示数据的独立来源。
- 必须有一条交叉测试：对同一组进度数组断言"SQL 口径结果 === 纯函数结果"，防止 SQLite `round()` 与 JS `Math.round` 在 .5 进位 / 浮点累加上漂移。`round(avg(...))` 与"先归一化再平均"必须产出同值（资料 `progress_percent` 落库时已归一化，故二者等价）。

首页推荐：

- 候选只包含 `learning`、`review`、`not_started`。
- 首页推荐强调延续当前任务，所以 `learning` 高于 `review`。
- `paused` 和 `completed` 不进入首页推荐。
- `last_studied_at` 为空按等价 `NULLS LAST` 处理。

> 设计说明（优先级方向差异）：首页推荐与项目状态派生使用相反的优先级方向，这是有意为之，不是 bug。
> - 首页推荐层（本节）：`learning` 高于 `review`，目的是优先把用户拉回“正在进行的任务”，减少中断成本。
> - 项目状态派生层（§6.4）：`review` 高于 `learning`，目的是只要项目下存在“需复习”的资料，就把项目整体标记为待破难点，避免复习需求被进行中的资料淹没。
> 两层服务的目标不同（推荐=延续手头任务，派生=暴露风险），因此方向相反且各自稳定。

推荐排序 SQL 口径：

```sql
ORDER BY
  CASE status
    WHEN 'learning' THEN 1
    WHEN 'review' THEN 2
    WHEN 'not_started' THEN 3
    ELSE 9
  END ASC,
  CASE WHEN last_studied_at IS NULL THEN 1 ELSE 0 END ASC,
  last_studied_at DESC,
  updated_at DESC,
  created_at DESC,
  id ASC
```

### 6.6 事务

必须使用事务：

1. 保存出关记录。
2. 创建资料。
3. 修改资料非终态状态。
4. 删除项目。
5. 删除资料。
6. 成功继续闭关后写入打开时间与待记录会话。

`save_study_log` 事务必须同时完成：

1. 校验 `resource_updated_at_before` 非空，并完成资料变化冲突判断。
2. 创建 `study_logs`。
3. 更新 `resources.progress_text`。
4. 更新 `resources.progress_percent`。
5. 更新 `resources.next_action`。
6. 更新 `resources.status`。
7. 更新 `resources.last_studied_at`。
8. 更新 `resources.updated_at`。
9. 更新 `projects.last_studied_at` 和 `projects.updated_at`。
10. 按来源清理待记录；`record_only` 和手动补记不得触碰既有待记录。
11. 重新派生项目状态并 `UPDATE projects.status`；项目进度仍实时计算，不落库存储。

`continue_resource` 成功后必须满足：

- 至少已经创建唯一 `pending_study_sessions`，写入打开时资料名称、进度、状态、下一步和 `updated_at` 快照。
- 系统打开成功后更新 `resources.last_opened_at`。
- 仅更新 `last_opened_at` 不更新 `resources.updated_at`，因为 `updated_at` 是资料变化冲突版本。

`continue_resource` 异步 / 同步边界（强制，避免 better-sqlite3 同步事务内 await）：

1. 入口先在事务外查询是否已有 pending（同步只读），命中则直接返回 `pending_conflict`，不打开任何资源。
2. 路径 / URL 安全校验（含 `fs.realpathSync`）在事务外完成。
3. 安全校验通过后，先开一个**同步**事务创建 pending 意图快照；此时不更新 `last_opened_at`。
4. 调用 `shell.openPath` / `shell.openExternal` 是异步操作，必须在事务**外** `await`：
   - 打开失败 / 被阻断 → 尽最大努力删除刚创建的 pending，返回 `open_failed` / `blocked`，不更新 `last_opened_at`。若删除失败，宁可留下可放弃的 pending，也不能出现"资源已打开但没有待记录"。
   - 打开成功 → 再开一个**同步**事务更新 `resources.last_opened_at`。若该更新失败，pending 仍保留，用户仍能保存或放弃本次记录。
5. 该顺序的核心取舍是：允许极小概率出现"未成功打开但有待记录提示"的可恢复假阳性，不允许出现"已打开但无待记录提示"的不可恢复漏记。

`continue_resource` 应用级串行化（防双击竞态）：

- "查无 pending → 打开 → 建 pending" 三步非原子，用户快速双击可能两次都通过"无 pending"检查，导致资源被打开两次、第二次建 pending 撞 `UNIQUE(singleton_key)`。
- 主进程必须对 `continue_resource` 持有一个进程内互斥锁（in-memory mutex / 单飞）：同一时刻只处理一个继续闭关流程。MVP 统一选择**排队等待**，不新增 `busy` 返回类型，确保 §19 DTO 不发散；渲染层禁用按钮只做体验优化。
- 渲染层 `opening` 期间禁用按钮（§8）只是体验优化，主进程互斥才是权威保障。

`create_resource`、`update_resource` 修改非终态状态、`delete_resource` 事务必须在资料写入后重新派生所属项目状态并写回 `projects.status`；删除项目时项目本身被删除，不需要派生写回。

任一步失败都整体回滚，界面保留用户输入。

### 7.1 首页：修炼总览

首页第一屏必须直接服务核心问题，不做营销页、游戏大厅、大面积世界观封面。

必须模块：

| 模块 | 内容 |
| --- | --- |
| 顶部栏 | 修炼总览、当前日期、本地保存状态、新增修炼方向、新增功法 / 资料 |

> “本地保存状态”语义：better-sqlite3 同步写入即落盘，不存在“保存中/未保存”的异步窗口。该指示器仅表达“数据已写入本地”这一事实，并展示最近一次成功写库的相对时间（取全库 `max(updated_at)`，例如“数据已保存 · 2 分钟前”）。它不是表单草稿状态，也不做云同步语义；若读库失败则显示“本地数据读取异常”并提供重试。
| 待记录提示条 | 资料名快照、开始时间、打开前进度、记录本次学习、放弃本次记录 |
| 继续闭关卡片 | 项目名、资料名、资料类型、进度描述、百分比、进度条、项目进度、下一步、最近出关 |
| 修炼方向列表 | 项目名、状态、总进度、最近出关时间 |
| 最近参悟记录 | 首页最多 5 条 |

关键按钮必须双层可理解：

- `继续闭关：打开资料继续学习`
- `出关记录：保存本次学习进度`

首页状态必须覆盖：

| 状态 | 页面表现 |
| --- | --- |
| 首次空状态 | 显示“还没有修炼方向。先创建一个学习主题。” |
| 有项目无资料 | 继续闭关卡片显示引导态 |
| 有资料无推荐 | 显示“暂无需要继续的功法” |
| 正常推荐 | 显示完整继续闭关卡片 |
| 待记录 | 顶部显示待记录提示条，继续闭关卡片仍可见 |
| 加载中 | 保留页面结构，关键区域显示轻量骨架 |
| 读取失败 | 显示错误说明和重试入口 |

### 7.2 项目详情页

必须展示：

- 修炼方向名称、描述、状态、修炼总进度。
- 功法数量、最近出关时间。
- 功法列表。
- 最近修炼日志，最多 5 条。

必须操作：

- 新增功法 / 资料。
- 编辑修炼方向。
- 删除修炼方向。
- 从资料列表继续闭关。
- 从资料列表出关记录。

资料列表排序：`learning`、`review`、`not_started`、`paused`、`completed`，同状态内按最近出关、更新时间、创建时间、ID 固定排序。

### 7.3 资料详情页

资料详情页必须把进度放在最醒目的位置。

必须展示：

- 功法名称、所属修炼方向、类型、打开方式、路径或链接、状态。
- 当前参悟位置、参悟进度、进度条、下次闭关目标。
- 最近打开时间、最近出关时间。
- 最近修炼日志，最多 10 条。

默认占位：

- 当前参悟位置：尚未记录学到哪里。
- 下次闭关目标：还没有设置下次闭关目标。
- 最近出关：暂无出关记录。
- 最近打开：暂无打开记录。

资料进度达到 100% 时展示“参悟完成”反馈。

### 7.4 出关记录弹窗

定位：低成本保存一次学习结果，不能变成复杂日志编辑器。

默认带入：

- 修炼方向。
- 当前功法。
- 当前参悟位置。
- 当前参悟进度。
- 当前下次闭关目标。
- 来自待记录状态时带入开始时间。

标题与资料名展示：

- 弹窗标题必须同时表达来源和真实动作，例如“出关记录：保存本次学习进度”。
- 标题区域必须显式展示当前记录的资料名，不得只放在隐藏 tooltip、占位符或面包屑中。
- 手动补记来源必须标注“手动补记：{当前资料名}”，避免用户误以为正在处理已有待记录会话。
- 待记录会话若资料名已变化，按 §16.5 同时展示打开时快照和当前标题。

字段：

| 字段 | 阶段 | 必填 | 说明 |
| --- | --- | --- | --- |
| 出关进度 | P0 | 否 | 学习后学到哪里 |
| 参悟进度 | P0 | 是 | 0 到 100 |
| 下次闭关目标 | P0 | 否 | 下次继续做什么 |
| 功法状态 | P1 | 否 | P0 可由进度自动推导 |
| 本次参悟 | P1 | 否 | 简短备注，不能成为必填日志 |
| 闭关时长 | P1 | 否 | 可自动估算，也可为空 |

交互规则：

- 打开后焦点落在“出关进度”。
- `参悟进度` 默认填当前资料进度。
- `下次闭关目标` 默认填当前资料下一步任务，允许清空。
- 保存中禁用按钮，按钮文案为“保存中...”。
- 保存失败弹窗不关闭，保留所有输入。
- 进度小于旧进度时二次确认，但允许保存。
- 选择“参悟完成”时，界面即时提示“进度将设为 100%”。
- 选择“未开始”时，界面即时提示“进度将设为 0%”。
- 状态强制归一化（选“未开始”归 0、选“参悟完成”归 100）与“进度回退二次确认”不叠加：当回退是由状态强制归零引起时，只展示状态即时提示（“进度将设为 0%”），不再额外弹回退二次确认；只有用户**手动**把参悟进度数值改小（未改终态状态）时才触发回退二次确认。避免同一操作连续两次确认。
- 字段无变化也允许保存一条学习记录。
- 有未保存修改时关闭、ESC、遮罩或返回，必须确认离开。

不同来源规则：

| 来源 | 保存后 | 取消 / 关闭且无修改 |
| --- | --- | --- |
| 待记录会话 | 创建日志，更新资料，清除待记录 | 不清除待记录 |
| 仅记录资料 | 创建日志，更新资料，不更新 `last_opened_at` | 不创建日志，不更新时间，不产生待记录 |
| 手动补记 | 创建日志，更新资料，不影响既有待记录 | 不创建日志，不更新资料 |

### 7.5 可落地 UI 蓝图：玄墨修炼台

UI 定位为“带修仙氛围的学习进度工作台”，不是游戏大厅。第一屏信息优先级固定为：推荐资料 > 当前参悟位置 > 参悟进度 > 下次闭关目标 > 操作按钮 > 其他辅助信息。所有装饰只能服务阅读层级，不能降低填写和判断效率。

桌面布局：

```text
┌──────────────────────────────── 顶部栏 56px ────────────────────────────────┐
│ 应用名 / 当前日期 / 本地保存状态 / 新增修炼方向 / 新增功法                 │
├──── 侧边导航 184px ────┬──────────────── 主工作区 ────────────────┬── 右侧栏 304px ─┤
│ 修炼总览               │ 待记录提示条                              │ 最近出关记录     │
│ 修炼方向               │ 继续闭关卡片                              │ 快捷状态         │
│ 最近记录               │ 修炼方向列表                              │                  │
└────────────────────────┴──────────────────────────────────────────┴────────────────┘
```

响应规则：

- `>= 1280px`：使用左侧导航 + 主工作区 + 右侧栏三列布局。
- `1024px - 1279px`：侧边导航收窄为 64px 图标栏，右侧栏移到主工作区下方；继续闭关卡片仍在首屏。
- 最小窗口 `1024x640`：顶部栏、待记录提示条、继续闭关卡片的资料名、百分比、下一步和主按钮必须无需滚动可见。
- 不做移动端优先布局；若未来支持窄屏，再单独补充小屏信息架构。

首页线框：

```text
[TopBar]
[PendingSessionBanner，只有存在 pending 时显示]

[ContinueCard]
  修炼方向 / 当前功法 / 类型 / 状态
  当前参悟位置
  参悟进度百分比 + 进度条
  下次闭关目标
  [继续闭关：打开资料继续学习] [出关记录：保存本次学习进度]

[ProjectProgressList]        [RecentStudyLogList]
```

继续闭关卡片组件规格：

| 区域 | 内容 | 视觉要求 |
| --- | --- | --- |
| 标题区 | `继续闭关`、状态标签、资料类型 | 标题不超过 24px；状态标签不得比资料名更抢眼 |
| 资料区 | 修炼方向、当前功法、当前参悟位置 | 资料名最大两行；长标题换行，不横向滚动 |
| 进度区 | 大号百分比、进度条、项目总进度 | 百分比是卡片内最大数字；进度条必须有可见文本 |
| 下一步区 | 下次闭关目标 | 独立暗色区块，优先级高于最近日志 |
| 操作区 | 继续闭关、出关记录 | 主按钮用 `accent + on-accent`；按钮文本保留双层含义 |

项目详情页布局：

```text
[ProjectHeader：名称 / 状态 / 总进度 / 最近出关]
[ResourceTable：功法名称 / 状态 / 参悟进度 / 下次目标 / 最近出关 / 操作]
[RecentStudyLogList：最多 5 条]
```

项目详情页以列表效率优先，不使用大面积卡片墙。资料列表可以有轻量“玉简”质感边框，但每行高度应稳定，避免 hover 或状态标签导致布局跳动。

资料详情页布局：

```text
[ResourceHeader：功法名称 / 所属方向 / 状态 / 类型]
[ResourceProgressPanel：当前参悟位置 / 参悟进度 / 下次闭关目标]
[Actions：继续闭关 / 出关记录 / 编辑资料 / 删除资料]
[SourceInfo：打开方式 / 折叠路径或 URL / 最近打开 / 最近出关]
[RecentStudyLogList：最多 10 条]
```

资料详情页的 `ResourceProgressPanel` 是主视觉；路径、URL、最近日志都只能作为辅助信息。路径展示默认中段折叠，例如 `D:\...\The Rust Book.pdf`；URL 默认显示协议与 hostname，完整链接只在显式展开或复制时出现。

出关记录弹窗布局：

```text
出关记录：保存本次学习进度
当前功法：The Rust Book
修炼方向：Rust 学习

出关进度
[这次学到哪里]

参悟进度
[-] [35] [+]    [range slider]

下次闭关目标
[下次具体做什么]

可选字段折叠区：本次参悟 / 闭关时长

[取消] [保存出关记录]
```

弹窗实现要求：

- 弹窗宽度建议 `560px - 640px`，最大不超过 `min(720px, calc(100vw - 48px))`。
- P0 默认只展开 3 个核心字段；P1 字段进入折叠区，不能增加 P0 填写成本。
- 参悟进度同时提供数字输入和滑杆；数字输入是最终精确来源，滑杆只做辅助。
- 保存按钮固定在弹窗底部，滚动内容不得遮挡操作区。
- `Enter` 在文本输入框内不提交；保存按钮聚焦时才提交。

组件清单：

| 组件 | 责任 |
| --- | --- |
| `AppShell` | 顶部栏、侧边导航、主内容容器、全局 Toast 容器 |
| `TopBar` | 当前日期、本地保存状态、新建入口 |
| `SideNav` | 修炼总览、修炼方向、最近记录 |
| `ContinueCard` | 首页推荐资料和继续学习主操作 |
| `PendingSessionBanner` | 待出关提示和记录 / 放弃入口 |
| `ProjectProgressList` | 首页修炼方向列表 |
| `ResourceTable` | 项目详情页资料列表 |
| `ResourceProgressPanel` | 资料详情页进度主视觉 |
| `StudyLogModal` | 出关记录弹窗 |
| `DeleteConfirmDialog` | 删除影响确认 |
| `ProgressBar` | 所有进度展示统一组件 |
| `StatusBadge` | 状态标签，依赖 `get_enums()` |
| `PathPreview` | 路径和 URL 折叠展示 |
| `ToastViewport` | FIFO 非阻塞反馈 |

组件边界：

- 不做卡片套卡片。页面分区使用无框布局或浅边界；卡片只用于继续闭关、单条重复项、弹窗。
- 卡片圆角最大 8px；按钮、输入框、标签圆角与卡片一致，不使用大药丸形态。
- 图标优先使用 Lucide；修仙装饰图标仅限自画小型 SVG，不作为主要操作语义。
- 所有交互控件必须有稳定尺寸，hover 不改变占位尺寸。
- 所有列表行必须有固定最小高度，长文本用换行或折叠，不横向撑开布局。

1. 结构层用 shadcn/Radix 思路：`Sidebar`、`Dialog`、`AlertDialog`、`Progress`、`Card`、`Button`，保证桌面工具可维护和可访问。

### 7.7 前端实现落地策略

基础组件策略：

- MVP 推荐采用 shadcn/ui 的“源码式组件”思路，而不是安装大型成品 UI 框架。组件源码进入 `src/renderer/components/ui/*` 后由项目自行维护，统一改成 §10 token。
- 若初始化阶段网络不可用或不想引入 CLI，直接用 Radix primitives + Tailwind 手写等价组件；不得因为组件库缺失而改变交互规范。
- Toast 不使用 shadcn 已废弃 toast；按 §10 自实现 `ToastViewport` FIFO 队列，或未来使用 Sonner 时必须二次封装以满足“最多 3 条、FIFO、不抢焦点”的规则。

建议首批 UI 组件底座：

```bash
pnpm dlx shadcn@4.8.3 add button card dialog alert-dialog progress input textarea label slider badge separator tooltip sidebar
```

> 若 shadcn CLI 与 Tailwind 4.3 CSS-first token 配置不匹配，则改为手动复制组件源码并把类名、CSS variables 映射到 `tokens.css`；不能降级 Tailwind 主版本来迁就 UI 组件。

组件到业务的实现映射：

| 业务组件 | 底层实现 | 关键实现点 |
| --- | --- | --- |
| `AppShell` | `SidebarProvider` / CSS Grid | 侧边栏宽度使用 `--layout-sidebar`，主内容 `minmax(0, 1fr)` 防止溢出 |
| `ContinueCard` | `Card` + `ProgressBar` + `Button` | 百分比最大；下一步区块独立；主按钮 `accent + on-accent` |
| `PendingSessionBanner` | 自定义 banner + `Button` | 固定在首页主内容顶部；不使用红色；显示快照标题 |
| `ProjectProgressList` | 语义 table 或 `role=list` | 数据密度优先；行高稳定；状态标签来自 `get_enums()` |

## 8. 状态机与待记录

统一状态机管理“继续闭关”和“出关记录”。

| 状态 | 含义 | 典型进入 | 典型退出 |
| --- | --- | --- | --- |
| `none` | 没有打开或待记录会话 | 启动、保存完成、放弃完成 | 点击继续闭关 |
| `opening` | 正在请求主进程打开资源 | 点击继续闭关 | 打开成功、失败、阻断 |
| `pending` | 已打开资源，等待记录 | 打开成功 | 保存出关、放弃记录 |
| `recording` | 出关记录弹窗打开 | 待记录、仅记录、手动补记 | 保存、取消、关闭 |
| `saving` | 正在保存出关记录 | 点击保存 | 保存成功、保存失败 |
| `saved` | 保存成功 | 事务提交成功 | 自动回到 `none` |
| `abandoned` | 放弃待记录 | 点击放弃本次记录 | 自动回到 `none` |
| `failed` | 打开或保存失败 | 主进程返回错误 | 修改路径、仅记录、重试、取消 |

关键规则：

1. `opening` 期间禁用同一资料的继续闭关按钮。
2. 打开失败不得创建 `pending_study_sessions`。
3. 保存失败停留在 `recording`，保留用户输入。
4. `record_only` 无 pending 时从 `none` 直接进入 `recording`。
5. 已有 pending 时，任何 `continue_resource` 都进入冲突处理，包括 `record_only`。
6. 手动补记允许与 pending 并存，保存后不得清除既有 pending。
7. `saved` 和 `abandoned` 后首页必须重新拉取最新数据。

待记录冲突弹窗：

```text
已有一次闭关尚未出关记录
请先处理上一次学习记录，再开始新的闭关。
```

操作：

| 操作 | 行为 |
| --- | --- |
| 记录上一次 | 打开已有待记录会话的出关记录弹窗 |
| 放弃上一次 | 清除已有待记录，不创建日志，不更新出关时间，然后允许用户重新点击新资料 |
| 取消本次操作 | 保留已有待记录，不打开新资料 |

资料变化冲突：

- 打开出关记录弹窗时带入 `resource_updated_at_before`。
- 新建学习记录时 `resource_updated_at_before` 必须非空；缺失时主进程返回 `VALIDATION_FAILED` 并保留用户输入。
- 保存前重新读取资料，当前 `updated_at` 不一致即判定资料已变化。
- 用户可确认覆盖或取消。
- “取消”表示不保存，保留待记录会话和用户已输入内容。
- “确认覆盖”表示使用弹窗里用户输入的进度、状态和下一步覆盖当前资料，不是丢弃用户输入。
- 确认覆盖创建的 `study_logs`，before 快照仍使用待记录会话中的打开前快照，after 快照使用用户确认后的最终归一化值。
- 资料已被删除时不能覆盖，只能提示资料不存在并允许放弃待记录。
- `record_only` 和手动补记没有打开起点，`resource_updated_at_before` 取“打开出关记录弹窗时即时读取的资料 `updated_at`”，并同样在保存前做一致性校验。

`save_study_log.source` × pending 自洽校验真值表（主进程强校验，渲染进程传入的 `source` 不可信）：

| source | 当前 pending 状态 | 主进程行为 |
| --- | --- | --- |
| `pending` | 无 pending | 拒绝：`VALIDATION_FAILED`（`details.reason = "no_pending_session"`），不写库 |
| `pending` | 有 pending 且 `pending.resource_id == input.resource_id` | 正常保存；提交事务时 DELETE 该 pending |
| `pending` | 有 pending 但 `pending.resource_id != input.resource_id` | 拒绝：`VALIDATION_FAILED`（`details.reason = "pending_resource_mismatch"`），不写库、不动 pending |
| `record_only` | 无 pending | 正常保存；不创建、不清理 pending |
| `record_only` | 有 pending（属于任意资料） | 拒绝：`VALIDATION_FAILED`（`details.reason = "pending_exists"`），不写库、不动 pending；record_only 入口必须先走冲突处理 |
| `manual` | 无 pending | 正常保存；不产生 pending |
| `manual` | 有 pending（属于任意资料，含同一资料） | 正常保存目标资料；**不得清除 / 覆盖**既有 pending |

> 校验顺序：先做 `source × pending` 自洽校验（上表），通过后再做 `resource_updated_at_before` 非空与资料变化冲突校验，最后才进入写事务。

`abandon_pending_session(session_id)` 的 id 不匹配行为：

- pending 为单例，但仍传入 `session_id` 以防 UI 持有过期引用。
- 若 `session_id` 与当前单例 pending 的 id 不一致，或当前无 pending：返回 `NOT_FOUND`，不删除任何数据、不更新任何时间。
- 仅当 `session_id` 与当前 pending 一致时才删除。

## 9. 安全、隐私、备份与日志

### 9.1 外部打开安全

文件和链接打开规则：

- 文件路径必须规范化。
- 文件打开前确认存在且是文件。
- 文件夹打开前确认存在且是文件夹。
- URL 必须用 URL parser 校验。
- URL 只允许 `http` 和 `https`。
- 拒绝 `file:`、`javascript:`、`data:`、`mailto:` 和所有自定义协议。
- 默认阻断可执行文件、脚本文件、快捷方式、UNC / 网络路径。
- 禁止拼接 shell 命令打开资源。

风险分级：

| 风险级别 | 类型 | 行为 |
| --- | --- | --- |
| 硬阻断 | 可执行文件、脚本、快捷方式、安装包、注册表、设备路径、网络路径、自定义协议 | 只提供“修改路径 / 仅记录进度” |
| 强提示 | 宏文档、压缩包、未知扩展、localhost、内网 URL、短链接 | 默认推荐“仅记录进度”，允许确认打开一次 |
| 普通打开 | 常规本地文档、常规文件夹、普通 http/https URL | 调用系统默认应用或浏览器打开 |

### 9.2 主进程权限

- 渲染进程禁用 `nodeIntegration`，启用 `contextIsolation`，启用 `sandbox`。
- 渲染进程不得调用任意 shell 命令。
- 渲染进程不得直接访问 `fs` / `child_process` / `shell`；只能通过 preload 暴露的语义 IPC 通道。
- 渲染进程只能调用语义命令，例如 `continue_resource({ resource_id })`。
- 文件、文件夹、URL 的校验和打开必须在主进程完成（Node `path` + `fs.realpathSync` + `shell.openPath` / `shell.openExternal`）。
- preload 暴露的 IPC 通道严格限定为 §5 列出的 14 个语义命令，不暴露通用 `invoke`、不开放任意 shell/open/read 能力。
- **内容安全策略（CSP）与资产全本地化**（保证真离线 + 纵深防御）：
  - 渲染进程必须设置严格 CSP，统一通过主进程 `session.defaultSession.webRequest.onHeadersReceived` 注入响应头（不仅写 `<meta>`，避免被绕过）。基线：`default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data:; font-src 'self'; connect-src 'self'; object-src 'none'; frame-src 'none'; base-uri 'none'`。
  - `style-src` 允许 `'unsafe-inline'` 仅为兼容 Tailwind 运行时样式；脚本一律 `'self'`，禁止 `unsafe-eval` 与远程脚本。
  - MVP 不加载远程字体、远程动效或 CDN 资产；若未来 P2 引入字体、粒子、Lottie 或纹理，必须随包本地化。断网时首屏必须照常工作。
  - 主进程拦截并拒绝渲染进程发起的任何外部 `http(s)` 网络请求（`webRequest` 默认拒绝非 `self` 资源），与 §9.3「不自动请求用户填写的 URL」一致；唯一对外动作只有用户显式点击"继续闭关"打开 URL 时的 `shell.openExternal`（交给系统浏览器，不在应用内加载）。

### 9.3 隐私

- 不上传任何学习数据。
- 不做遥测。
- 不自动上传崩溃日志。
- 不读取学习文件内容。
- 不抓取网页内容、favicon 或链接预览。
- 不自动请求用户填写的 URL。
- 错误日志默认不记录完整本地路径和完整 URL。
- 页面展示路径时默认折叠中间段。

### 9.4 备份与恢复

- 每次数据库迁移前自动创建本地备份。
- 启动成功且完整性检查通过后，若距离上一次常规备份超过 7 天，自动创建轻量备份。
- 迁移失败不得继续启动到半升级数据库。
- 数据库损坏时进入只读诊断状态，提示本地备份。
- 备份和日志都视为敏感本地数据，不能自动上传。

### 9.5 本地可观测性

结构化日志事件：

- `app_start`
- `db_init`
- `migration_start`
- `migration_success`
- `migration_failure`
- `clock_skew_detected`
- `command_start`
- `command_end`
- `continue_resource_result`
- `save_study_log_commit`
- `save_study_log_rollback`
- `pending_created`
- `pending_cleared`

日志规则：

- 位于应用数据目录的 `logs/` 子目录。
- 当前日志文件名为 `app.log`。
- 单文件最大 2 MB，最多保留 5 个滚动文件。
- 默认保留最近 30 天。
- 默认不记录完整路径和完整 URL。
- 超过保留期的日志可在启动时自动清理。
- MVP 不提供用户主动清理日志或备份的入口，避免演变为独立数据管理页；用户只能通过只读诊断页执行备份恢复相关动作。

## 10. UI 主题与可访问性

修仙主题只做命名、氛围和轻量反馈，不能替代真实学习信息。

明确不做：

- 游戏化大厅。
- 角色养成页。
- 等级、经验值、战力、排行榜。
- 抽卡、成就墙等完整游戏系统。

MVP 只提供一套深色修仙主题，不提供浅色模式或主题切换。

视觉方向：采用“玄墨修炼台”。基础体验是深色生产力工具，修仙感来自命名、沉金进度反馈、轻量墨色纹理和克制动效；不得引入人物立绘、世界地图、战力面板、境界树、抽卡按钮等会把产品误导成游戏的元素。

视觉 token：

| token | 建议值 | 用途 | 对比度验收 |
| --- | --- | --- | --- |
| `color.bg` | `#0E0B14` | 主背景 | 与 `color.text` 约 16:1 |
| `color.surface` | `#182026` | 面板背景 | 与 `color.text` 约 13:1 |
| `color.surface-raised` | `#202A31` | 弹窗、悬浮层、重点卡片 | 与 `color.text` 约 10:1 |
| `color.border` | `#2D3A35` | 面板边框、列表分隔 | 仅作边界，不作文本 |
| `color.text` | `#F2F0E8` | 正文 | 对主背景约 16:1 |
| `color.muted` | `#B8C0B2` | 次级文本 | 对主背景约 10:1 |
| `color.accent` | `#D6A84F` | 主操作和进度强调 | 与深色文字约 8:1 |
| `color.on-accent` | `#1A1206` | 叠加在 `accent` 上的按钮文字 / 图标 | 对 `color.accent` 约 9:1 |
| `color.danger` | `#E57373` | 删除和阻断提示 | 对主背景约 6:1 |
| `color.mystic` | `#8B5CF6` | 轻量氛围点缀、100% 完成反馈 | 不作为正文色 |

排版与尺度：

| 项 | 规则 |
| --- | --- |
| 字体栈 | 默认使用系统中文无衬线字体；标题可用系统 serif fallback，但 MVP 不下载远程中文字体 |
| 数字 | 百分比和时间优先使用等宽字体栈，保证列表扫描稳定 |
| 页面标题 | 20-24px，不能使用 hero 级大字 |
| 卡片标题 | 16-18px |
| 正文 | 14-16px，最小不低于 13px |
| 圆角 | 卡片、弹窗、输入框、按钮统一最大 8px |
| 边框 | 1px 半透明墨青边界，避免厚重游戏边框 |
| 阴影 | 仅弹窗和重点卡片使用弱阴影；不做大面积发光 |
| 间距 | 页面主间距 16-20px；列表行内间距 10-12px；工具型密度优先 |

布局 token：

| token | 建议值 | 用途 |
| --- | --- | --- |
| `layout.topbar` | `56px` | 顶部栏高度 |
| `layout.sidebar` | `184px` | 常规侧边导航 |
| `layout.sidebar-collapsed` | `64px` | 1024-1279px 收窄导航 |

氛围层规则：

- 背景可使用极低透明度的墨色纹理或阵纹，透明度建议 `<= 0.06`，不得影响文字对比度。
- 首页可有非常轻的金粉 / 星点静态背景；MVP 不引入粒子库，P2 再评估 tsParticles。
- 进度提升可用 150-240ms 的沉金扫光；`prefers-reduced-motion` 下关闭扫光，仅保留颜色变化和 Toast。
- 100% 完成只做短 Toast + 进度条完成态，不弹全屏庆祝。
- 错误、删除、权限阻断不使用修仙玩笑文案，优先清楚说明真实原因。

对比度约束（满足需求 §7.9 关键操作按钮 ≥ 4.5:1）：

- 主操作按钮使用 `accent` 作背景时，文字必须用 `color.on-accent`（深色），不得用 `color.text`（浅色）。
  - `on-accent #1A1206` on `accent #D6A84F` 约 9:1，通过 AA。
  - 若误用 `text #F2F0E8` on `accent #D6A84F` 约 1.9:1，不通过，禁止使用。
- 正文 `text` on `bg` / `surface`、次级 `muted` on `surface` 必须在实现后用对比度工具实测并记录，未达 4.5:1（大号文本 3:1）的组合不得进入发布。
- `danger` 作为文字色时只用于 `surface` / `bg` 背景，不作为大面积按钮背景；若作按钮背景，文字同样使用 `color.on-accent` 级别的深色并实测。
- 上述十六进制对比度为估算值，最终以实现后工具实测为准（见 §23 不确定项）。

最小使用示例：

```css
.primaryButton {
  background: var(--color-accent);
  color: var(--color-on-accent);
}
```

可访问性要求：

- 最小窗口尺寸 1024x640。
- 资料名、路径、下一步任务过长时允许换行或中间截断，不得覆盖按钮和进度条。
- 路径默认折叠中间段，避免暴露完整用户名目录。
- 关键按钮支持键盘聚焦。
- 弹窗关闭、保存、取消按钮有可见焦点。
- 进度条必须同时显示文本百分比。
- 100% 与 125% 系统缩放下不重叠、不截断核心含义。
- 关键按钮、表单输入、进度条必须有可访问名称。
- 成就反馈使用非阻塞 Toast。

Toast 规则：

- 成功 Toast 默认 3 秒。
- 错误 Toast 默认 5 秒，并提供关闭按钮。
- 多条 Toast 按出现顺序先进先出排队。
- 最多同时展示 3 条；超过 3 条时新 Toast 进入等待队列，前一条关闭或超时后再展示下一条。
- Toast 不抢焦点，不阻断键盘操作。

### 13.3 任务依赖图与并行 / 串行编排

## 14. 范围闸门与风险控制

任何功能进入 MVP，必须同时满足：

1. 能帮助用户记录资料进度。
2. 能帮助用户下次恢复学习上下文。
3. 不要求用户维护额外系统。
4. 不引入新的长期数据管理页面。
5. 不让“出关记录”超过 30 秒完成。

主要风险与控制：

| 风险 | 控制方式 |
| --- | --- |
| 把资源打开器当成核心产品 | 继续闭关只服务记录前后的学习闭环 |
| 把日志做成独立日志系统 | 日志只由保存出关记录自动生成，不做独立管理页 |
| 把修仙氛围做成游戏系统 | 只做命名、氛围和轻量反馈，不做等级、战力、排行榜 |
| 渲染进程自行推断业务规则 | 状态归一化、推荐、事务、路径安全全部放主进程 Service 层 |
| 项目进度缓存漏同步 | MVP 实时查询计算，不保存缓存字段 |
| `record_only` 绕过 pending | `continue_resource` 首先检查待记录冲突 |
| 出关记录半更新 | 事务内创建日志、更新资料、更新项目、清理 pending |
| 删除资料丢历史 | `study_logs.resource_id` 置空，保留 `resource_title_snapshot` |
| 迁移失败半升级 | 迁移事务化，失败进入只读诊断，不继续启动 |
| 日志和备份泄露隐私 | 不记录完整路径和完整 URL，不自动上传 |

### 16.4 项目与资料业务规则

项目状态派生必须在服务层实现，按以下顺序命中即返回：

1. 项目没有资料，或所有资料都是 `not_started`，项目状态为 `not_started`。
2. 所有资料都是 `completed`，项目状态为 `completed`。
3. 任一资料为 `review`，项目状态为 `review`。
4. 任一资料为 `learning`，项目状态为 `learning`。
5. 没有 `learning` / `review`，但至少一个资料为 `paused` 且未全部完成，项目状态为 `paused`。
6. 创建项目时写入派生值 `not_started`；保存出关记录、添加资料、删除资料、修改资料非终态状态后，都必须重新派生项目状态，并在同一事务内写回 `projects.status`。

> 设计说明：规则 3 把 `review` 排在 `learning` 之前，是为了让“项目里只要有需复习的资料就整体标红”，与首页推荐层 `learning` 高于 `review` 的方向相反，二者目标不同（见 §6.5 设计说明），均为有意设计。
> 求值方式：规则按编号顺序自上而下匹配，命中即返回，不继续向下匹配。
> 混合示例：项目下资料状态为 `{learning, review}` 时，先命中规则 3，项目状态为 `review`；资料状态为 `{learning, paused}` 时，规则 3 不命中、规则 4 命中，项目状态为 `learning`；资料状态为 `{completed, paused}` 时，规则 2 不成立（非全部完成）、3/4 不命中、规则 5 命中，项目状态为 `paused`。

资料编辑必须遵守：

- 所属项目创建后不可编辑，MVP 不支持移动资料到另一个项目。
- 资料编辑表单只包含标题、类型、打开方式、路径或链接、非终态状态标记。
- 创建后 `progress_text`、`progress_percent`、`next_action` 只能由 `save_study_log` 更新并生成 `study_logs`。
- 编辑资料不会更新 `last_studied_at`，不会创建学习记录。
- 编辑资料**必须**更新 `resources.updated_at`（即使只改了元信息 / 打开方式 / 非终态状态）。这是资料变化冲突检测（§8 / §9.12）能对元信息编辑路径生效的前提：出关弹窗打开后若用户又编辑了资料，保存时才能凭 `updated_at` 不一致检出冲突。漏写 `updated_at` 将导致乐观锁失效，属于必测项（§21）。
- 编辑资料修改非终态状态时，当前进度必须已经在 1 到 99；否则主进程拒绝，并提示通过出关记录变更进度。

打开方式变更必须遵守：

1. 从 `file`、`folder`、`url` 切换到 `record_only` 时，`path_or_url` 归一化为 `null`。
2. 从 `record_only` 切换到 `file`、`folder` 或 `url` 时，必须重新填写合法路径或 URL。
3. 在 `file`、`folder`、`url` 之间切换时，按目标打开方式重新校验 `path_or_url`。
4. 打开方式变更只更新资料元信息，不更新 `last_opened_at`、`last_studied_at`，也不创建待记录会话。
5. 如果资料存在待记录会话，仍允许编辑打开方式；保存出关记录时按资料变化冲突处理。

### 16.5 首次使用、列表容量与页面细则

首次使用链路必须完整：

1. 无项目时首页显示“还没有修炼方向。先创建一个学习主题。”
2. 无项目时，“新增功法 / 资料”按钮禁用或提示“请先创建修炼方向”。
3. 创建项目成功后，提示用户继续添加第一个资料。
4. 添加资料默认进度为 0，默认状态为未开始。
5. 添加资料取消后，首页回到“有项目无资料”状态。
6. 添加资料成功后，回到首页并显示继续闭关卡片。

列表容量和查询必须遵守：

- 首页项目列表首屏最多渲染 50 个项目；超过 50 个使用分页或虚拟滚动。
- 项目详情资料列表首屏最多渲染 100 个资料；超过 100 个使用分页或虚拟滚动。
- Repository 查询必须支持 `limit` / `offset` 或 cursor 参数。
- 渲染进程不得一次性渲染 5000 条资料。
- 首页推荐、项目摘要、最近记录查询必须独立于长列表分页，不能因为分页而缺失关键推荐信息。

最近记录数量：

- 首页最近修炼日志最多 5 条。
- 项目详情最近修炼日志最多 5 条。
- 资料详情最近修炼日志最多 10 条。

低保真线框必须先固定信息层级，再做视觉细化：

```text
[顶部栏：修炼总览 | 当前日期 | 本地保存状态 | 新增方向 | 新增资料]

[继续闭关卡片：项目名 / 资料名 / 进度描述 / 百分比进度条 / 下一步 / 继续闭关 / 出关记录]
[项目列表：项目名 / 状态 / 总进度 / 最近出关时间]
[最近修炼日志：最多 5 条]
```

```text
[资料标题 / 所属项目 / 状态]
[主进度区：进度描述 / 百分比 / 进度条 / 下一步]
[操作：继续闭关 / 出关记录 / 编辑资料 / 删除资料]
[来源信息：打开方式 / 路径或 URL / 最近打开时间 / 最近出关时间]

```

```text
[资料名 + 来源]

[参悟进度输入]
[下次闭关目标输入]
[P1：状态 / 本次参悟 / 闭关时长]

```

首页项目列表排序必须为：`last_studied_at IS NULL` ASC、`last_studied_at DESC`、`updated_at DESC`、`created_at DESC`、`id ASC`。项目详情资料列表同状态内也必须使用相同的稳定时间与 ID 排序口径。

待记录提示条必须显示 `resource_title_snapshot`。如果当前资料标题与快照不一致，提示条显示：

```text
打开时：{resource_title_snapshot}
当前标题：{current_resource_title}
```

放弃待记录必须显示：

```text
放弃后不会保存本次学习进度，也不会更新最近出关时间。
```

### 16.6 出关记录、键盘与弹窗细则

闭关时长估算：

1. 只有来自待记录会话时才自动估算。
2. 估算公式为 `round((now_utc - opened_at) / 60 秒)`，最小值 0 分钟。
3. 估算值大于 12 小时时，默认不填入，只提示“可能包含挂机时间，可手动填写”。
4. `record_only` 和手动补记没有打开起点，闭关时长默认为空。
5. 手动填写的闭关时长必须在 0 到 1440 分钟之间。

弹窗键盘规则：

- 出关记录弹窗打开后焦点落在“出关进度”。
- 弹窗关闭后焦点回到触发按钮。
- 弹窗内启用 focus trap，Tab / Shift+Tab 只在弹窗可聚焦元素中循环。
- Enter 在文本输入框内不提交表单；保存按钮聚焦时提交。
- ESC 等价于关闭弹窗；有未保存输入时必须出现离开确认。
- Tab 顺序按视觉顺序：标题、进度、下一步、状态、备注、时长、取消、保存。

资料名展示规则：

- 待记录会话、`record_only`、手动补记三种来源的弹窗标题区都必须显示资料名。
- 手动补记必须显示“手动补记：{当前资料名}”；该文本必须在弹窗首屏可见。
- 如果待记录会话的 `resource_title_snapshot` 与当前资料名不同，弹窗沿用 §16.5 的“打开时 / 当前标题”双行展示。

关闭规则必须区分来源：

| 来源 | 保存后 | 取消 / 关闭且无修改 | 关闭且有未保存修改 |
| --- | --- | --- | --- |
| 待记录会话 | 创建日志，更新资料，清除待记录 | 不清除待记录，稍后可再记 | 二次确认，确认后仍保留待记录 |
| 仅记录资料 | 创建日志，更新资料，不更新 `last_opened_at` | 不创建日志，不更新任何出关时间，不产生待记录 | 二次确认，确认后不保存任何内容 |
| 手动补记 | 创建日志，更新资料，不影响既有待记录 | 不创建日志，不更新资料 | 二次确认，确认后不保存任何内容 |

### 16.7 全局交互状态与固定恢复文案

全局状态必须统一处理：空状态、加载中、保存成功、保存失败、打开失败、待记录、删除确认。

固定恢复文案和动作：

| 场景 | 文案 | 恢复操作 |
| --- | --- | --- |
| 保存失败 | 保存失败，刚才填写的内容已保留。请稍后重试。 | 重试 / 取消 |
| 数据读取失败 | 未能读取本地修炼记录。请重试，或检查本地数据文件。 | 重试 |
| 删除项目确认 | 删除后会移除该修炼方向、其下功法和相关记录。此操作不可撤销。 | 确认删除 / 取消 |
| 删除资料确认 | 删除后资料会移除，历史记录会保留资料名称快照。 | 确认删除 / 取消 |
| URL 协议不支持 | 只支持 http:// 或 https:// 链接。 | 修改链接 |
| 权限不足 | 没有权限打开该路径。可以修改路径，或仅记录进度。 | 修改路径 / 仅记录进度 |

删除含待记录会话的资料时，确认弹窗必须额外提示“这会放弃尚未保存的本次学习记录”。删除完成后首页必须重新读取待记录状态，不能继续显示旧提示条。

删除项目时若该项目下存在待记录会话（FK 级联会一并删除该 pending），删除项目确认弹窗也必须追加同样的提示：“这会放弃尚未保存的本次学习记录。” 删除完成后首页同样必须重新读取待记录状态。

### 16.8 路径、URL 与外部打开细化规则

路径规则：

- 保存和打开前必须规范化路径。
- 禁止相对路径和包含未解析 `..` 的路径进入打开流程。
- 禁止设备路径，例如 `\\?\`、`\\.\`。
- 禁止 UNC 和网络路径。
- 禁止 Windows Alternate Data Streams，例如 `file.txt:stream`。
- 禁止快捷方式 `.lnk` 和 Internet Shortcut `.url`。
- 符号链接、junction、reparse point 必须解析最终目标；若最终目标是网络位置或风险类型，必须阻断。
- 映射网络盘按网络路径处理。
- 打开前必须在尽可能靠近系统打开调用的位置重新校验最终目标。
- 校验后到系统打开之间如果目标被本机其他进程替换，MVP 接受低概率 TOCTOU 风险，不引入后台监控或文件锁。

URL 规则：

- 必须能被 URL parser 正确解析。
- 只允许 `http` 和 `https`。
- 必须有 host。
- 禁止 userinfo，例如 `https://user:pass@example.com`。
- 禁止控制字符、换行和空白混淆。
- `localhost`、`127.0.0.1`、内网 IP、短链接不硬阻断，但必须强提示。
- 页面展示 URL 时优先展示解析后的 hostname。
- 保存资料时允许保存 `localhost`、内网 IP、短链接等强提示 URL；强提示只发生在打开阶段。

硬阻断扩展必须包括：`.exe`、`.bat`、`.cmd`、`.ps1`、`.vbs`、`.js`、`.lnk`、`.com`、`.scr`、`.msi`、`.hta`、`.wsf`、`.jse`、`.vbe`、`.jar`、`.reg`、`.url`、`.scf`。宏文件如 `.docm`、`.xlsm`、`.pptm` 不硬阻断，但必须强提示。

强提示弹窗中的"打开一次"只对本次操作有效，不记忆到资料或全局设置；点击后必须禁用该按钮直到主进程返回结果，避免连点创建重复打开请求。

### 16.9 迁移、备份与恢复细则

迁移规则：

- 使用 drizzle-kit 自动维护的 `__drizzle_migrations` 元表记录已应用迁移；列含 drizzle 自动生成的 `id`、`hash`、`created_at`，等价替代需求 §9.11 的 `schema_migrations(version, applied_at, checksum)`。
- 迁移文件由 `pnpm drizzle-kit generate` 从 schema 自动生成，按文件名顺序（`0000_xxx.sql`、`0001_xxx.sql` …）执行。
- 每个迁移文件首尾人工补 `BEGIN;` / `COMMIT;`（当前冻结 drizzle-kit 0.31.10 的 sqlite 迁移文件不默认包裹事务），确保单个迁移事务化。
- 迁移失败时事务整体回滚，不得留下半升级 schema。
- drizzle migrator 会比对 `__drizzle_migrations.hash` 与文件 hash，已应用的迁移文件不会重复执行；若文件被改写导致 hash 不一致，启动必须失败并映射为 `MIGRATION_FAILED`。
- 启动 migrate 前先执行 `PRAGMA foreign_keys = ON`；migrate 后执行 `PRAGMA foreign_key_check` 必须为空。
- SQLite 表重建类迁移（drizzle 自动生成的 `__new_xxx` 临时表方案）必须先校验旧数据是否满足新约束。
- 同一迁移版本连续失败时，应用不得反复自动重试；必须停在只读诊断页（§9.4）。

备份规则：

- 每次数据库迁移前自动创建本地备份副本。
- 迁移备份默认最多保留最近 3 个。
- 启动成功且数据库完整性检查通过后，如果距离上一次常规备份超过 7 天，自动创建轻量备份。
- 常规备份默认最多保留最近 4 个。
- 迁移成功后不立即删除最新备份。
- 备份文件与主数据库一样包含敏感学习信息，不能自动上传。
- MVP 不提供正常运行期的手动清理备份入口；备份保留按上述数量策略自动滚动。

只读诊断页必须显示：

1. 主数据库路径。
2. 最近可用备份列表、创建时间、对应迁移版本。
3. 操作入口：从备份恢复、打开数据目录、退出应用。
4. 敏感数据提示：备份与日志可能包含资料标题、路径、URL 和学习进度。

恢复流程：

1. 用户选择备份后，应用先关闭当前数据库连接。
2. 将当前损坏 DB 重命名为 `cultivation_system.corrupt.{timestamp}.sqlite3`。
3. 复制所选备份为新的 `cultivation_system.sqlite3`。
4. 重新启动数据库初始化流程；如备份版本较旧，按迁移规则再次迁移。
5. 如果恢复后的同一迁移再次失败，停在只读诊断页，不循环重试。

### 16.10 推荐索引与查询口径

MVP 必须建立以下索引，保证首页、详情页和最近记录查询稳定：

```sql
CREATE INDEX idx_resources_project_id ON resources(project_id);
CREATE INDEX idx_resources_status_last_studied_at ON resources(status, last_studied_at DESC, updated_at DESC, created_at DESC, id ASC);
CREATE INDEX idx_resources_updated_at ON resources(updated_at);
CREATE INDEX idx_study_logs_project_studied_at ON study_logs(project_id, studied_at DESC, created_at DESC);
CREATE INDEX idx_study_logs_resource_id ON study_logs(resource_id, studied_at DESC, created_at DESC);
CREATE INDEX idx_pending_study_sessions_project_id ON pending_study_sessions(project_id);
CREATE INDEX idx_pending_study_sessions_resource_id ON pending_study_sessions(resource_id);
```

推荐查询不能依赖 SQLite 对 NULL 的默认排序，必须显式写出等价 `NULLS LAST`。Repository 测试必须断言索引存在、推荐排序稳定、最近日志排序稳定。

### 16.11 自用 M5 验收清单（不再要求逐条 36 条）

本方案的 M5 不要求把需求文档 §11.2 的 36 条逐条转为 E2E 用例和发布门禁；自用阶段执行下列**核心断言抽样**，覆盖最容易造成真实数据错误的高风险规则，但不宣称等同于 36 条发布级验收。每条建议人工跑一次：

- 空字符串、纯空格项目名和资料标题不能保存；失败后保留输入；合法名称 trim 后保存；允许同名项目和同名资料。
- 进度只接受 0 到 100 的整数；空值、纯空格、-1、101、35.5、非数字均不能保存。
- `completed` 强制 100，`not_started` 强制 0；100 自动 `completed`，0 自动 `not_started`。
- 从终态改回非终态必须提交 1 到 99 的进度。
- 项目下资料进度为 20、40、100 时项目进度必须为 53%；无资料为 0%；删除资料后立即重算。
- 首页只推荐 `learning`、`review`、`not_started`；`paused` / `completed` 永不推荐；相同时间按固定顺序排序。
- 文件、文件夹、网页继续闭关只更新 `last_opened_at`，不更新 `last_studied_at`，并进入待记录状态。
- `record_only` 无 pending 时直接进入出关记录；有 pending 时先进入冲突处理；取消后不创建日志、不更新出关时间、不产生 pending 残留。
- 已有待记录会话时，任何继续闭关都必须提示记录、放弃或取消，不能静默覆盖。
- 待记录关闭重开后恢复；保存或放弃后清除，重启后不恢复旧 pending。
- 出关保存必须在事务内同时完成日志、资料、项目最近出关时间、项目状态派生写回、pending；模拟写入失败时全部回滚并保留输入。
- 手动补记不得清除既有 pending。
- 无变化保存仍新增 1 条 `study_logs`，反馈文案为“已记录本次学习，本次暂无进度变化”。
- 删除资料前二次确认；有 pending 时额外提示会放弃未保存记录；删除后日志保留并置空 `resource_id`。
- 删除项目前二次确认；确认后项目、资料、相关日志、待记录全部删除，无孤儿数据。
- 路径不存在、URL 无效、权限不足分别提示原因；失败时不更新 `last_opened_at`，不进入 pending。
- URL 只允许 `http` / `https`；`file`、`javascript`、`data`、`mailto`、自定义协议不能保存或打开。
- 风险路径打开前必须提示或阻断；“打开一次”不记忆且点击后禁用；取消时不进入 pending。
- 断网状态下创建项目、添加资料、记录进度、查看历史、关闭重开恢复均可用。
- 渲染进程不能调用任意 shell/open/read 文件；外部打开必须经过 `resource_id` 语义命令。

> 未来对外发布时再把 36 条转为可重复运行的 E2E 用例与门禁脚本。

## 18. 数据库 DDL（建表语句）

以下 DDL 是**目标 schema 形状**，不是手写执行的 SQL；实际 schema 由 Drizzle 在 `src/main/db/schema.ts` 用 `sqliteTable` 声明，再由 `pnpm drizzle-kit generate` 生成等价 `.sql` 迁移文件。每次 schema 改动后，应执行 `sqlite3 cultivation_system.sqlite3 .schema` 与本节 DDL 比对，确保 Drizzle 生成的 CREATE TABLE / CHECK / INDEX 与本节一致（U17 决策）。

> 说明：SQLite 的 `CHECK` 无法直接表达"正则/RFC3339 解析"等复杂校验，时间格式、枚举语义的完整校验由 Service 层 + Zod 兜底，数据库层只做基础硬约束。

```sql
PRAGMA foreign_keys = ON;

CREATE TABLE projects (
  id              TEXT PRIMARY KEY,
  name            TEXT NOT NULL CHECK (length(trim(name)) > 0 AND length(trim(name)) <= 120),
  description     TEXT CHECK (description IS NULL OR length(description) <= 1000),
  status          TEXT NOT NULL CHECK (status IN ('not_started','learning','paused','review','completed')),
  created_at      TEXT NOT NULL,
  updated_at      TEXT NOT NULL,
  last_studied_at TEXT
);

CREATE TABLE resources (
  id               TEXT PRIMARY KEY,
  project_id       TEXT NOT NULL,
  title            TEXT NOT NULL CHECK (length(trim(title)) > 0 AND length(trim(title)) <= 200),
  type             TEXT NOT NULL CHECK (type IN ('document','video','web','course','repo','exercise','book','other')),
  open_kind        TEXT NOT NULL CHECK (open_kind IN ('file','folder','url','record_only')),
  path_or_url      TEXT CHECK (path_or_url IS NULL OR length(path_or_url) <= 2048),
  status           TEXT NOT NULL CHECK (status IN ('not_started','learning','paused','review','completed')),
  progress_text    TEXT CHECK (progress_text IS NULL OR length(progress_text) <= 500),
  progress_percent INTEGER NOT NULL CHECK (progress_percent BETWEEN 0 AND 100),
  next_action      TEXT CHECK (next_action IS NULL OR length(next_action) <= 500),
  created_at       TEXT NOT NULL,
  updated_at       TEXT NOT NULL,
  last_opened_at   TEXT,
  last_studied_at  TEXT,
  FOREIGN KEY (project_id) REFERENCES projects(id) ON DELETE CASCADE,
  -- record_only 必须无路径；其余打开方式 trim 后必须非空
  CHECK (
    (open_kind = 'record_only' AND (path_or_url IS NULL OR length(trim(path_or_url)) = 0))
    OR (open_kind <> 'record_only' AND path_or_url IS NOT NULL AND length(trim(path_or_url)) > 0)
  )
);

CREATE TABLE study_logs (
  id                         TEXT PRIMARY KEY,
  project_id                 TEXT NOT NULL,
  resource_id                TEXT,
  resource_title_snapshot    TEXT NOT NULL CHECK (length(resource_title_snapshot) <= 200),
  studied_at                 TEXT NOT NULL,
  duration_minutes           INTEGER CHECK (duration_minutes IS NULL OR (duration_minutes BETWEEN 0 AND 1440)),
  content                    TEXT CHECK (content IS NULL OR length(content) <= 2000),
  progress_before_text       TEXT CHECK (progress_before_text IS NULL OR length(progress_before_text) <= 500),
  progress_before_percent    INTEGER NOT NULL CHECK (progress_before_percent BETWEEN 0 AND 100),
  progress_after_text        TEXT CHECK (progress_after_text IS NULL OR length(progress_after_text) <= 500),
  progress_after_percent     INTEGER NOT NULL CHECK (progress_after_percent BETWEEN 0 AND 100),
  status_before              TEXT NOT NULL CHECK (status_before IN ('not_started','learning','paused','review','completed')),
  status_after               TEXT NOT NULL CHECK (status_after IN ('not_started','learning','paused','review','completed')),
  next_action_before         TEXT CHECK (next_action_before IS NULL OR length(next_action_before) <= 500),
  next_action                TEXT CHECK (next_action IS NULL OR length(next_action) <= 500),
  resource_updated_at_before TEXT,
  created_at                 TEXT NOT NULL,
  FOREIGN KEY (project_id) REFERENCES projects(id) ON DELETE CASCADE,
  FOREIGN KEY (resource_id) REFERENCES resources(id) ON DELETE SET NULL
);

CREATE TABLE pending_study_sessions (
  id                         TEXT PRIMARY KEY,
  project_id                 TEXT NOT NULL,
  resource_id                TEXT NOT NULL,
  resource_title_snapshot    TEXT NOT NULL CHECK (length(resource_title_snapshot) <= 200),
  opened_at                  TEXT NOT NULL,
  progress_before_text       TEXT CHECK (progress_before_text IS NULL OR length(progress_before_text) <= 500),
  progress_before_percent    INTEGER NOT NULL CHECK (progress_before_percent BETWEEN 0 AND 100),
  status_before              TEXT NOT NULL CHECK (status_before IN ('not_started','learning','paused','review','completed')),
  next_action_before         TEXT CHECK (next_action_before IS NULL OR length(next_action_before) <= 500),
  resource_updated_at_before TEXT NOT NULL,
  singleton_key              INTEGER NOT NULL DEFAULT 1 CHECK (singleton_key = 1),
  created_at                 TEXT NOT NULL,
  UNIQUE (singleton_key),
  FOREIGN KEY (project_id) REFERENCES projects(id) ON DELETE CASCADE,
  FOREIGN KEY (resource_id) REFERENCES resources(id) ON DELETE CASCADE
);

-- 迁移元表：由 drizzle-kit 自动创建并维护 `__drizzle_migrations`，本方案不再手写 schema_migrations。
```

索引见 §16.10。`projects.status` 虽为派生值，仍落库存储以便排序与展示；创建项目时写入 `not_started`，保存出关记录、添加资料、删除资料、修改资料非终态状态等相关写路径必须在同一事务内重新派生并写回 `projects.status`。`projects.progress_percent` 不落库，始终按 §6.5 实时计算。

`study_logs.resource_updated_at_before` 在 DDL 中允许 NULL 仅为兼容旧历史数据；所有新建学习记录必须由服务层断言该字段非空，并写入打开弹窗时读取到的资料 `updated_at`。

// result=blocked 且 block_level=warn 时，前端展示“打开一次”；二次调用必须带 risk_confirm_token。
