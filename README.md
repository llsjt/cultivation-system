# Cultivation System

Cultivation System 是一款本地优先的学习进度与动力反馈应用。它帮助用户管理长期学习方向、记录每次学习进展、看见持续推进的成果，并通过“境界”这类阶段反馈保持学习动力。

这个项目不是单纯的资料收藏夹，也不是复杂的游戏化系统。它关注的是一个更实际的问题：用户学了很多东西之后，能不能清楚知道自己学到哪里、下一步该做什么、最近有没有真的推进。

## 这个项目是干什么的

很多个人学习过程会卡在三件事上：

- 资料很多，但不知道哪一个最应该继续。
- 学习经常中断，下次打开时需要重新回忆上次进度。
- 学习记录分散，用户看不到自己的持续积累，动力容易掉。

Cultivation System 用一个本地桌面应用把这些信息集中起来：

- 用“修炼方向”管理一个长期学习主题。
- 用“资料”管理书籍、课程、网页、文件夹、代码仓库等学习入口。
- 用“出关记录”保存每次学习后的进度、心得、时长和下一步目标。
- 用首页推荐和待记录会话帮助用户接上上一次学习。
- 用项目境界、层数和突破反馈给用户阶段性成就感。

一句话概括：

> 让用户看见自己的学习进度，并通过持续反馈保持学习动力。

## 适合谁使用

- 正在自学课程、技术、语言、书籍或长期主题的人。
- 有很多学习资料，但经常忘记上次学到哪里的人。
- 希望轻量记录学习进度，而不是维护复杂任务系统的人。
- 更看重本地数据可控，不依赖云同步的人。
- 喜欢一点主题化反馈，希望学习过程更有持续感的人。

## 核心概念

| 概念 | 含义 | 例子 |
| --- | --- | --- |
| 修炼方向 | 一个长期学习项目或主题 | React 进阶、算法训练、英语阅读 |
| 资料 | 某个方向下的学习材料 | PDF、视频课、网页、文件夹、代码仓库 |
| 继续学习 | 打开资料并开始一次学习 | 打开课程链接、打开本地文件夹 |
| 待出关记录 | 已开始但还没有保存结果的一次学习 | 打开资料后，学习结束前的 pending 会话 |
| 出关记录 | 一次学习后的记录 | 本次学到哪、进度变化、下次做什么 |
| 项目境界 | 当前学习方向的阶段反馈 | 炼气、筑基、金丹等境界和层数 |

修仙主题只是表达层。真实产品目标仍然是学习进度管理和动力反馈。

## 主要功能

- 管理修炼方向：创建、编辑、删除学习方向。
- 管理学习资料：支持纯记录、文件、文件夹和网页链接。
- 首页继续学习：推荐最适合继续的资料，并显示当前进度和下次目标。
- 待出关恢复：打开资料后生成待记录会话，应用重开后仍可恢复。
- 保存出关记录：更新资料进度、项目进度、最近学习时间和学习日志。
- 放弃待记录：放弃不会生成学习日志，也不会改动进度和下一步目标。
- 项目境界反馈：根据资料掌握、试炼资料、复盘证据和稳定性计算境界状态。
- 全部资料视图：跨方向查看和筛选学习资料。
- 本地数据存储：使用 SQLite 保存项目、资料、学习记录和待记录会话。

## 具体怎么用

### 1. 创建修炼方向

第一次打开应用后，先创建一个修炼方向。它代表一个长期学习目标，例如：

- 学完一本书
- 完成一门课程
- 掌握一个技术方向
- 维护一个长期训练计划

创建后，这个方向会出现在左侧列表中，并成为后续资料和学习记录的归属。

### 2. 添加学习资料

进入某个修炼方向后，可以添加资料。资料支持几种入口：

- 纯记录：不绑定文件或链接，只用来记录学习进度。
- 文件：绑定一个本地文件，例如 PDF、文档、视频。
- 文件夹：绑定一个本地目录，例如课程资料目录或代码仓库。
- 网页链接：绑定一个 URL，例如课程页面、文档、文章。

添加资料时建议填写：

- 资料名：用户能看懂的材料名称。
- 当前进度：例如 0%、30%、80%。
- 进度描述：例如“读到第 3 章”“完成 Hooks 部分”。
- 下一步目标：例如“继续看 Suspense”“完成第 5 节练习”。
- 资料权重：表示这份资料对当前方向的重要程度。

### 3. 继续学习

在首页推荐卡或资料列表中点击“继续学习”。

系统会根据资料类型打开对应入口：

- 文件会用系统默认程序打开。
- 文件夹会用系统文件管理器打开。
- 网页会用浏览器打开。
- 纯记录资料会直接进入记录流程。

当资料被打开后，系统会创建一条待出关记录。它表示“这次学习已经开始，但还没有保存学习结果”。

### 4. 保存出关记录

学习结束后，回到应用保存出关记录。通常需要填写：

- 本次学到哪里。
- 当前进度百分比。
- 下次继续做什么。
- 学习时长。
- 状态或证据类型。

保存后，系统会同步更新：

- 资料进度。
- 资料状态。
- 项目整体进度。
- 最近学习时间。
- 学习日志。
- 首页推荐和项目反馈。

### 5. 处理待出关记录

如果应用检测到还有待出关记录，会在首页提示。

你可以选择：

- 保存本次学习进度：生成学习日志，并更新进度。
- 放弃本次记录：不生成日志，也不更新进度。

同一时间只保留一个待出关会话。这样可以避免多份资料同时 pending 导致记录混乱。

### 6. 查看进度和境界

项目面板会展示当前方向的整体状态，包括：

- 当前进度。
- 资料数量。
- 最近学习记录。
- 当前境界和层数。
- 是否满足突破条件。
- 还缺哪些学习积累。

境界反馈不是独立游戏系统，而是帮助用户看到“这个方向已经积累到什么阶段”的轻量反馈。

## 当前项目状态

当前项目处于自用 MVP 向稳定版本收敛的阶段。

- v1.5：重点是让首页、继续学习、出关记录和项目反馈形成稳定学习闭环。
- v1.6：重点是优化境界提升模型，让资料数量、资料权重和学习时间更合理地影响反馈。

版本文档入口：

- `docs/版本索引.md`
- `docs/versions/v1.5/需求分析.v1.5.md`
- `docs/versions/v1.5/实施方案.v1.5.md`
- `docs/versions/v1.6/需求分析.v1.6.md`

根目录旧版 `docs/需求文档.md` 和 `docs/实施方案.md` 不再作为当前入口。

## 技术栈

- 桌面应用：Electron + electron-vite
- 前端：React + TypeScript
- 数据库：SQLite + better-sqlite3 + Drizzle ORM
- 表单和校验：react-hook-form + zod
- 日志：pino
- 测试：Vitest + Testing Library
- 打包：electron-builder

## 本地运行

### 环境准备

项目使用 pnpm，仓库中声明的包管理器版本是：

```text
pnpm@9.15.9
```

安装依赖：

```powershell
rtk corepack pnpm install
```

### 启动开发版

```powershell
rtk corepack pnpm dev
```

`dev` 脚本会先执行 `rebuild:electron`，确保 `better-sqlite3` 能在 Electron ABI 下运行。

## 常用命令

类型检查：

```powershell
rtk corepack pnpm typecheck
```

代码检查：

```powershell
rtk corepack pnpm lint
```

测试：

```powershell
rtk corepack pnpm test
```

构建：

```powershell
rtk corepack pnpm run build
```

说明：`test` 脚本会先执行 `rebuild:node`，确保 `better-sqlite3` 能在 Node/Vitest 环境下运行。

## 打包和验证

生成 portable 包：

```powershell
rtk corepack pnpm run package:portable
```

打包后冒烟验证：

```powershell
rtk corepack pnpm run smoke:portable
```

产物位置：

- `release\CultivationSystem 0.0.0.exe`
- `release\win-unpacked\CultivationSystem.exe`

注意：`package:portable` 会把 `better-sqlite3` 重建为 Electron ABI。打包后如果要继续跑 Node/Vitest 测试，需要切回 Node ABI：

```powershell
rtk corepack pnpm run rebuild:node
```

如果要回到 Electron 运行环境：

```powershell
rtk corepack pnpm run rebuild:electron
```

## 数据和安全边界

- 数据默认保存在本地 SQLite 数据库中。
- 项目启动迁移前会保留迁移备份。
- 完整性检查通过后，会按 7 天间隔保留常规备份。
- 当前不做云同步、多人协作、账号系统或在线分享。

## 已知非阻断项

- 当前仍使用默认 Electron 图标，适合自用 MVP，不影响核心功能。
- `package.json` 暂缺 `description` 和 `author`，electron-builder 会给出 warning，但不影响 portable 包生成和启动。

## 项目边界

为了保持产品主线清楚，当前阶段不做：

- 角色养成、战力、排行榜、抽卡等复杂游戏系统。
- 云同步、团队协作和社交功能。
- 过重的任务管理、日历排程或打卡系统。
- 为了主题化牺牲真实操作含义的文案设计。

这个项目首先要做好一件事：让用户每次回来都能看到自己已经走了多远，并知道下一步怎么继续。
